# Замер скорости -- Qwen3.6-27B Q4_K_M (2026-04-28)

**Тип**: инфраструктурный замер (`llama-bench`), **не Aider Polyglot**
**Цель**: оценить рентабельность prompt-based бенчмарка на 27B dense до запуска Aider smoke
**Результат**: модель **в 4.7× медленнее MoE-альтернативы**, prompt-based бенчмарк нерентабелен -- решение не запускать smoke
**Логи**: bench-вывод в bash output, без сохранения в /tmp

## Контекст

Qwen3.6-27B Q4_K_M скачана 27 апреля 2026 как кандидат для smoke + --tries 2 (ожидание 70-78% pass_rate_2 на основе SWE-V 77.2% леaderboard). Первый запуск Aider smoke остановлен через 45 минут без единой закрытой задачи -- модель работала, но multi-turn цикл с retry на низкой tg-скорости разрастался экспоненциально.

Перед повторным запуском замерено через `bench.sh` для определения реальной скорости и оценки разумности продолжения.

## Среда

| Компонент | Значение |
|-----------|----------|
| Hardware | AMD Strix Halo (gfx1151), 120 GiB unified memory |
| Kernel | 6.19.8-061908-generic |

| Модель | `Qwen3.6-27B-Q4_K_M.gguf` (15.65 GiB, 26.90B параметров) |

## Команда замера

```bash
AI_BACKEND=vulkan ./scripts/inference/bench.sh ~/models/Qwen3.6-27B-Q4_K_M.gguf 2048 256
```

## Результаты

| test | t/s |
|------|-----|
| **pp2048** | **286.72 ± 2.93** |
| **tg256** | **12.41 ± 0.01** |

### Контекст: где это в шкале платформы

| Модель | tg256 (tok/s) | Архитектура | Δ vs 27B dense |
|--------|----------------|-------------|----------------|
| [Qwen3-Coder 30B-A3B](../../../models/families/qwen3-coder.md#30b-a3b) | 86.0 | MoE A3B (full attention) | **+6.9×** |
| [Qwen3.6-35B-A3B (multimodal)](../../../models/families/qwen36.md#35b-a3b) | 58.7 | MoE A3B (hybrid) | **+4.7×** |
| [Qwen3-Coder Next 80B-A3B](../../../models/families/qwen3-coder.md#next-80b-a3b) | 53.0 | MoE A3B (hybrid) | **+4.3×** |
| **Qwen3.6-27B (dense)** | **12.4** | **dense** | -- |

12.4 tok/s -- это близко к **теоретическому потолку** memory bandwidth: 256 GB/s ÷ 15.65 GiB ≈ 16.4 tok/s. Реальные ~75% эффективности (12.4 / 16.4) -- стандарт для dense моделей на Vulkan/Strix Halo.

## Архитектурная особенность

llama.cpp metadata показывает архитектуру **qwen35** (а не qwen36 hybrid Gated DeltaNet, как ожидалось из roadmap). Возможные причины:

1. **GGUF metadata legacy**: unsloth/Qwen3.6-27B-GGUF мог использовать обновлённый llama.cpp архитектуру qwen35 как backwards-compatible base
2. **27B вариант реально dense без recurrent**: hybrid Gated DeltaNet объявлено для 35B-A3B и Coder Next, но не для базового dense 27B
3. **Misidentification в metadata**: возможен баг в конвертере (unsloth)

В любом случае это означает что **cache-reuse архитектурно НЕ blocked** для 27B (нет recurrent memory). Если бы скорость была разумной, cache-friendly workloads работали бы корректно.

## Расчёт рентабельности Aider smoke

При tg = 12.4 tok/s одна aider-задача (multi-turn с --tries 2):
- Базовая итерация: ~2K сгенерированных токенов = **~2.7 минуты**
- 3-5 reasoning итераций per task = **~8-13 минут**
- Сложная задача (много retry, длинный context shift): **15-20+ минут**

Итого:
- `--quick --tries 1` (10 задач, no retry): **~1.5-2 часа**
- `--smoke --tries 2` (20 задач): **~5+ часов** (как видно из failed pilot, может растягиваться больше)
- `--full --tries 2` (195 задач): **~50+ часов**

Сравнение с MoE 35B-A3B (smoke + --tries 2): **1h 55m** на 20 задач. То есть 27B dense smoke в **2.5-3× медленнее** при потенциально лишь +3.8pp pass_rate_2 (77.2% SWE-V vs 73.4% у 35B-A3B).

## Решение

**Не запускать prompt-based бенчмарки на 27B dense**. Trade-off скорость vs качество не оправдан:

- 35B-A3B (MoE) уже даёт 70.0% pass_rate_2 (Aider Polyglot smoke 20) при 248 sec/case
- 27B dense теоретически даст 75-78% pass_rate_2 (по экстраполяции SWE-V), но при ~600+ sec/case
- +5-8pp качества за 2.5× больше времени wall clock -- неоправданно для daily agent

## Где 27B dense всё-таки имеет смысл

1. **Batch-задачи с длинным prompt + коротким output**: pp = 286.7 tok/s достойно. Например, классификация/extraction по большим документам -- модель быстро "поглощает" prompt, медленно генерирует короткий ответ → выгоднее dense чем MoE
2. **Точечная сверка качества**: запросить 27B 1-2 раза на сложной задаче, сравнить с 35B-A3B -- смысл есть
3. **После llama.cpp PR #20376** (Vulkan f16 GATED_DELTA_NET): возможно ускорение на dense моделях через f16 опт-ы. Replay через 1-3 мес после merge
