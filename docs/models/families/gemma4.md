# Gemma 4 (Google, 2 апреля 2026)

> Multimodal MoE с native function calling, 256K контекстом, thinking-режимом, bounding box prediction, adaptive image tokens (70-1120). Четыре размера: E2B, E4B, 26B-A4B (MoE), 31B (dense).

**Тип**: MoE (3.8B active / 25.2B total) -- основной вариант; также 31B dense и edge E2B/E4B
**Лицензия**: Apache 2.0 (E2B, E4B), Gemma Terms of Use (26B, 31B)
**Статус на сервере**: скачана (26B-A4B Q6_K_XL + mmproj-BF16)
**Направления**: [llm](../llm.md), [vision](../vision.md)
**Function calling**: native (лучшее сочетание FC + vision на платформе)
**Vision**: да (SigLIP-style encoder, mmproj-BF16, 1.19 GB, adaptive 70-1120 tokens/image)
**Audio**: только E2B и E4B (USM-style conformer encoder)

## Обзор

Gemma 4 26B-A4B -- multimodal MoE-модель от Google нового поколения 2026. Total 25.2B параметров, активных 3.8B, что даёт скорость как у dense 4B-модели. Поддерживает text и images, native function calling, контекст 256K, thinking-режим через `<|think|>` token.

На платформе используется как основная multimodal через `vulkan/preset/gemma4.sh`. Vision реализован двухкомпонентно: основной GGUF + отдельный `mmproj-BF16.gguf` (1.19 GB) с весами vision-encoder'а.

## Варианты

Семейство включает **четыре размера**, от edge до flagship:

| Вариант | Параметры | Active | Контекст | Vision | Audio | VRAM Q4 | Статус | Лицензия |
|---------|-----------|--------|----------|--------|-------|---------|--------|----------|
| **E2B** | 2B dense | 2B | 32K | да | **да** | ~2 GiB | не скачана | Apache 2.0 |
| **E4B** | 4B dense | 4B | 32K | да | **да** | ~3 GiB | не скачана | Apache 2.0 |
| **26B-A4B** | 25.2B MoE | 3.8B | **256K** | да | нет | ~17 GiB | **скачана** (Q6_K_XL) | Gemma TOU |
| **31B** | 31B dense | 31B | **256K** | да | нет | ~19 GiB | не скачана | Gemma TOU |

**E2B и E4B** -- edge-варианты с **аудио-входом** (USM-style conformer encoder) помимо vision и text. Designed для мобильных устройств и embedded. На Strix Halo избыточны -- 26B-A4B качественнее при том же VRAM.

**31B dense** -- flagship, все параметры активны. Даёт лучшее качество в бенчмарках чем 26B-A4B MoE, но медленнее (dense 31B vs 3.8B active). На Strix Halo: ~19 GiB Q4 помещается, но tg ~5 tok/s (memory-bound на 31B), что непрактично для интерактивного чата.

**26B-A4B MoE** -- основной рабочий вариант на платформе. Баланс: 3.8B active = скорость dense 4B, качество большой 26B.

## Архитектура и особенности

- **MoE 8/128 + 1 shared expert** -- активны 3.8B параметров на токен, скорость как у 4B-модели
- **SigLIP-style vision encoder** в отдельном `mmproj-BF16.gguf` файле
- **Контекст 256K** -- можно загрузить много кадров видео или серию скриншотов
- **Native function calling** -- из коробки, без отдельного fine-tune
- **Thinking-режим** через `<|think|>` token в начале system prompt
- **Variable aspect ratio** изображений -- понимает портрет/панораму без принудительного crop
- **Sliding window attention** -- чередование local (512-1024 токенов) и global слоёв
- **Bounding box prediction** -- нативный JSON output для координат объектов в кадре

## Интересные технические детали

### Адаптивные image tokens: 70 → 1120 на кадр

SigLIP vision encoder в Gemma 4 **не фиксирует количество токенов на изображение**. Вместо этого используется **configurable token budget**: 70, 140, 280, 560 или 1120 vision-токенов на кадр.

Это означает:
- **70 токенов** (минимум) -- грубое понимание сцены, достаточно для "что на фото" и классификации. Минимальный VRAM, максимальная скорость
- **280 токенов** (среднее) -- баланс для большинства задач (описание, OCR крупного текста)
- **1120 токенов** (максимум) -- pixel-level детализация для OCR мелкого текста, bounding box prediction, chart analysis

При загрузке 30 скриншотов в 256K контекст: `30 × 280 = 8400` vision-токенов -- менее 4% контекста. Если важна детализация: `30 × 1120 = 33600` -- ~13% контекста, всё ещё управляемо.

На практике llama-server автоматически выбирает budget на основе разрешения входного изображения. Малые картинки (<384px) → 70 токенов, средние → 280, большие (>1024px) → 560-1120.

### Чередование local и global attention

Gemma 4 **не использует чисто global attention** на всех слоях (как в Llama/Qwen). Вместо этого чередуются:

- **Local sliding-window слои** (window = 512 или 1024 токенов) -- видят только ближайший контекст. Быстрые, экономные по KV-cache
- **Global full-context слои** -- видят все 256K токенов. Медленнее, дороже

Чередование (упрощённо): local → local → global → local → local → global → ...

Это **снижает KV-cache** (local слои не кэшируют дальние токены) и ускоряет inference, но создаёт **побочный эффект**: sliding window cache не переиспользуется между turn'ами (ctx-shift не работает), поэтому в multi-turn каждый запрос фактически пересчитывает весь prefix.

### Per-Layer Embeddings (PLE) в E2B/E4B

Маленькие варианты (E2B, E4B) используют **PLE** -- дополнительный параллельный канал рядом с основным residual stream:

```
Основной поток:    x → attention → FFN → x'
PLE-поток:         e_token + e_context → маленький вектор на каждый layer
```

PLE даёт каждому слою **дополнительный conditioning signal**, который зависит:
1. От **token identity** (какой токен) -- через lookup table
2. От **контекста** (что вокруг) -- через mini-attention

Результат: E2B с PLE даёт качество, которое без PLE потребовало бы ~3-4B параметров. Это key innovation для edge-deployment.

### Аудио-encoder в E2B/E4B

Малые варианты имеют **USM-style conformer audio encoder** (как в Gemma 3n):
- Принимает аудио-вход (до 90 секунд)
- Конвертирует в sequence of audio tokens
- Токены объединяются с text и vision в общий input

Это делает E2B и E4B **tri-modal**: text + images + audio. У 26B-A4B и 31B аудио-входа нет.

Практический use case на edge: smart doorbell с Gemma 4 E4B -- видит камеру, слышит звук, отвечает текстом.

### Bounding box prediction: native JSON

Gemma 4 умеет возвращать координаты объектов прямо в JSON, без post-processing:

Промпт: "Найди все кнопки на скриншоте, верни координаты"

Ответ:
```json
[
  {"label": "Submit", "bbox": [120, 340, 280, 380]},
  {"label": "Cancel", "bbox": [300, 340, 420, 380]},
  {"label": "Settings icon", "bbox": [780, 10, 820, 50]}
]
```

Координаты `[x1, y1, x2, y2]` в пикселях исходного изображения. Это **нативная способность** модели, не требующая grammar constraints -- модель обучена возвращать bbox как часть ответа.

Полезно для: UI testing automation, accessibility audit (где clickable area < 44px), visual regression detection (сравнение bbox между screenshots).

### Shared expert: 1 общий + 8 из 128 routed

Gemma 4 26B-A4B использует MoE с **shared expert pattern**: помимо 8 из 128 routed-экспертов, всегда активен **1 shared expert**, обрабатывающий каждый токен. Это даёт baseline-знание (общеязыковое) + специализацию (предметную область) одновременно.

Сравнение с другими MoE на платформе:

| Модель | Routed | Shared | Active total |
|--------|--------|--------|--------------|
| Mixtral 8×22B | 2 из 8 | нет | 39B |
| [Qwen3-Coder 30B-A3B](qwen3-coder.md) | 8 из 128 | нет | 3B |
| [Qwen3.5-122B-A10B](qwen35.md) | top-8 из 128 | нет | 10B |
| **Gemma 4 26B-A4B** | 8 из 128 | **1 (≈0.4B)** | **3.8B** |

Shared expert работает как "общий мозг", который не теряется в специализации. Прагматический эффект: модель меньше "забывает" русский язык при переключении на код, и наоборот. Это особенно заметно при смене языка ответа в multi-turn диалоге.

### Родословная vision encoder: PaliGemma → Gemini Flash

Vision encoder в `mmproj-BF16.gguf` -- прямой потомок **PaliGemma 3** (Google). PaliGemma в свою очередь -- упрощённая дистилляция vision-стека Gemini Flash. То есть Gemma 4 фактически наследует визуальное понимание Gemini Flash, но в форме, помещающейся в 23 GiB.

Это объясняет:
- Почему Gemma 4 так сильна на screenshot-to-code -- PaliGemma была обучена на UI-данных и code-from-design tasks
- Почему bounding box prediction работает out-of-the-box -- detection был частью pretraining-задач PaliGemma
- Почему variable aspect ratio понимается без crop'а -- наследие архитектуры Gemini

### Multilingual: 140 языков из коробки

Gemma 4 обучена на корпусе с **140 языками**, включая русский, украинский, белорусский, казахский, грузинский. Это шире чем у Llama 4 (≈100 языков с приоритетом английского) и Qwen3 (29 языков с приоритетом китайского + английского).

На практике для русского:
- Function calling работает на русских описаниях tool'ов (`description: "Получает погоду в указанном городе"`)
- OCR кириллицы в скриншотах работает (но мелкий русский шрифт хуже чем у [qwen3-vl](qwen3-vl.md) -- разница ~10-15% в accuracy)
- Системные промпты можно писать по-русски без потери качества instruction-following
- Смешанный контекст (русский + английские технические термины REST/gRPC/Kubernetes) обрабатывается естественно

### QAT checkpoints от Google: Q4 без потери качества

Google официально публикует **Quantization-Aware Training (QAT)** checkpoints для Gemma 4. Модель **обучена с учётом будущей квантизации**, в отличие от стандартной post-training quantization.

Эффект на качество:
- Q4_K_M даёт качество близкое к BF16 (typical loss < 1% на бенчмарках)
- В обычных post-quantized моделях разница BF16 vs Q4 -- 3-5%
- Q5_K_M практически неотличим от F16

Прагматичный вывод для платформы: **Q6_K_XL избыточен с точки зрения качества**. Q4_K_M даёт тот же результат при 17 GiB вместо 22 GiB. Q6_K_XL рекомендуется только потому что 22 GiB всё ещё легко помещается в 120 GiB unified memory, и нет смысла экономить ради 5 GiB. Если планируется параллельный запуск с большим Coder Next или другими VLM -- стоит спуститься на Q4_K_M.

### Distillation chain: Gemini 3 Flash → Gemma 4

По заявлениям Google в Gemma 4 technical report, обучение использовало **synthetic data, сгенерированные Gemini 3 Flash**. То есть Gemma 4 -- это, фактически, дистилляция Gemini 3 Flash в открытые веса, в 100× меньшем размере.

Косвенные признаки этого в поведении модели:
- Стиль ответов часто напоминает Gemini (структура с заголовками, использование emoji в bullets, склонность к нумерованным спискам)
- Иногда модель "просвечивает" -- упоминает Google или Gemini в неожиданных местах ("As a Google-trained model...")
- Качество на ряде закрытых бенчмарков сравнимо с Gemini 3 Flash, что необычно для 26B-модели

### Sliding window: разный размер окна по вариантам

Local sliding-window слои у разных вариантов имеют разный размер окна:

| Вариант | Sliding window | Эффект |
|---------|----------------|--------|
| E2B / E4B | 512 токенов | Минимум, скорость на edge |
| **26B-A4B** | **1024 токена** | Баланс, default на платформе |
| 31B dense | 4096 токенов | Максимум, лучше long-context reasoning |

На 26B-A4B local-слои "видят" 1K соседних токенов, на 31B -- 4K. Для большинства задач разница незаметна, но для long-context reasoning (понимание длинного диалога с нелинейными отсылками назад) 31B даёт более связные ответы. На Strix Halo 31B всё равно непрактичен из-за скорости (~5 tok/s).

### Tokenizer 262K: крупнейший словарь в open-weights

Gemma 4 использует **262144-токенный** SentencePiece tokenizer -- наследие Gemma 3. Для сравнения: Llama 4 -- ~150K, Qwen3 -- 152K, DeepSeek V3 -- 130K. Это не bloat: словарь специально расширен под 140 языков и code-friendly разбиение.

Прагматический эффект на платформе:

- **Русский текст**: ~1.4 токена на слово (vs ~2.1 у Llama 4). На длинном русском контексте Gemma 4 эффективно "видит" больше слов в те же 256K токенов
- **Восточно-европейские языки** (украинский, белорусский, польский) разбиваются на цельные слова, а не на subword'ы по букве
- **Код**: служебные конструкции (`def `, `function`, `=>`, `=== `) -- single-token, ускоряет генерацию
- **Math symbols** (греческие буквы, операторы $\sum$, $\int$, $\partial$) -- single-token каждый, что критично для LaTeX-генерации

Цена за это: больше памяти под embedding table (262144 × hidden_dim ≈ 800 MB на embedding), но это окупается через tied embeddings (см. ниже).

### Pan-and-Scan: адаптивное чтение больших изображений

Native разрешение vision-encoder -- **896×896 пикселей**. Изображения большего размера обычно ресайзятся, что приводит к потере мелких деталей (пятнышки на скрине, мелкий шрифт). Gemma 4 решает это через **Pan-and-Scan (P&S)** -- алгоритм наследник PaliGemma:

1. Изображение сначала ресайзится в 896×896 для **общего контекста** (overview)
2. Затем **разбивается на тайлы** 896×896, каждый кодируется отдельно
3. Тайлы конкатенируются в общий sequence vision-токенов

Эффект: full-resolution скриншот 1920×1080 → 1 overview-тайл + 6 detail-тайлов = ~7× больше vision-токенов, но **без потери pixel-level информации**. Для UHD (3840×2160) -- ~25 тайлов.

В llama.cpp P&S включён по умолчанию для крупных изображений. Контроль через prompt: "опиши общее содержание скриншота" → меньше тайлов, "перечитай весь мелкий текст внизу страницы" → P&S активируется.

### GQA 2:1: меньше экономии KV, больше точности attention

Gemma 4 26B-A4B использует Grouped Query Attention с соотношением **2 query heads на 1 KV head** (16 Q : 8 KV). Сравнение с другими моделями:

| Модель | Q heads | KV heads | Ratio | Экономия KV |
|--------|---------|----------|-------|-------------|
| Llama 4 30B (Maverick) | 32 | 4 | 8:1 | 87.5% |
| Qwen3-Coder 30B-A3B | 32 | 4 | 8:1 | 87.5% |
| **Gemma 4 26B-A4B** | **16** | **8** | **2:1** | **50%** |

Gemma 4 -- консервативнее остальных: **больше KV cache** (выше memory pressure), но и **меньше потерь точности** attention при длинном контексте. Это объясняет почему Gemma 4 не страдает от "lost-in-the-middle" эффекта на 256K так сильно как Qwen3-Coder Next на тех же длинах.

Цена: KV cache на 256K контексте у Gemma 4 ≈ 7 GiB, у Qwen3-Coder Next с тем же контекстом ≈ 1.8 GiB. Одна из причин почему пресет ограничен `--parallel 4` -- больше слотов уже не помещается.

### Tied embeddings: input = output weights

Gemma 4 использует **tied embeddings** -- input embedding (`embed_tokens.weight`) и output projection (`lm_head.weight`) разделяют один и тот же тензор весов. Это унаследовано из Gemma 1/2/3, в отличие от Llama 3+/Qwen3, где embeddings untied.

Эффект:

- **Экономия памяти**: 262144 × 4096 = ~1 GB параметров, которые **не дублируются**
- **Регуляризация**: трудно для модели "обмануть" output distribution отдельно от input semantics
- **Известный недостаток**: ограничивает разнообразие output distribution на сложных задачах с большим вариативным словарём (например creative writing)

Для платформы это значит, что real-world размер 26B-A4B -- ближе к 25.2B (vs ~26.5B было бы при untied). Q6_K_XL занимает 22 GiB вместо ~23 GiB.

### MatFormer в E4B: матрёшечная архитектура

E4B использует **MatFormer (Matryoshka Transformer)** архитектуру -- наследие Gemma 3n. Это значит что внутри весов 4B-модели **уже встроена** работающая 2B-подмодель: можно "вынуть" подсеть E2B из весов E4B без отдельного обучения.

Это позволяет deploying на edge-устройствах с **dynamic inference cost**:

- Battery low → deploy E2B-подсеть (faster, less accurate)
- Plugged in → deploy полную E4B (slower, more accurate)
- Same weights file, runtime selection через layer skipping

На Strix Halo это не релевантно (используется только 26B-A4B), но интересно как architectural pattern.

### No default system prompt by design

Единственная из современных open-weight моделей **без официального default system prompt** от вендора. Сравнение:

| Модель | Default system prompt |
|--------|----------------------|
| Llama 4 | "You are a helpful AI assistant..." (короткий) |
| Qwen3 | "You are Qwen, created by Alibaba Cloud..." (длинный) |
| **Gemma 4** | **отсутствует** -- разработчик пишет с нуля |
| DeepSeek V3 | "You are DeepSeek-V3..." (короткий) |

Google объясняет это в model card: "We believe applications should define their own system prompts based on use case." Прагматический эффект:

- **Плюс**: больше контроля, нет "leaking" мнений Google в выход модели
- **Минус**: разработчику нужно писать prompt сам, без шаблона. Default behavior без system prompt -- nicely instruction-following, но без identity ("я просто AI-ассистент")

В пресете на платформе system prompt не задан -- модель ведёт себя нейтрально и адаптируется под пользовательский promptу. Это совпадает с замыслом Google и удобно для function calling (никакого "Я Gemma" в JSON-ответах).

## Сильные кейсы

- **Function calling по визуальному входу** -- получает скриншот UI, вызывает tool с выделенными координатами
- **Reasoning по диаграмме** через thinking-режим
- **Длинный контекст 256K** -- много кадров видео, серия скриншотов, длинная инструкция
- **Универсальный VLM "общего назначения"** -- сильна на смешанных задачах
- **Variable aspect ratio** -- понимает портрет/панораму без потери информации
- **Скорость MoE** -- как у 4B при качестве 26B

## Слабые стороны / ограничения

- **Sliding window attention** -- ранее требовал агрессивных OOM-защит (`--parallel 1 -c 65536`). На текущей платформе с 120 GiB unified memory защиты сняты: пресет работает с `-c 250000 --parallel 4 --no-mmap`. Если запустить параллельно с большой Coder Next моделью (45 GiB) -- защиты могут понадобиться обратно
- **KV-cache shifting не работает** -- multi-turn чат пересчитывает префикс. `--cache-reuse 256` в пресете установлен, но llama.cpp выдаёт `cache reuse is not supported - ignoring n_cache_reuse = 256`
- **OCR хуже** [qwen3-vl](qwen3-vl.md) и [internvl](internvl.md)
- **Не специализирована под конкретный домен** (math/science -- лучше специализированные модели)

## Оптимальная стратегия применения на платформе

### Какой квант брать

| Квант | Размер | Качество | Когда |
|-------|--------|----------|-------|
| **Q6_K_XL** ⭐ | 22 GiB | ~99% F16 | **Рекомендуемый дефолт**. Скачан, используется в пресете |
| Q4_K_M | 17 GiB | ~96% F16 | Если нужно держать параллельно с большим Coder Next (Q4 + Coder Next + FIM = 64 GiB) |
| Q8_0 | 27 GiB | ~99.5% F16 | Не нужен -- разница с Q6_K_XL в пределах шума, +5 GiB не оправданы |

Q6_K_XL -- золотая середина для платформы 120 GiB. Q4_K_M только если хочется освободить место под второй большой сервер.

### Когда брать Gemma 4, а когда альтернативы

| Задача | Gemma 4 | Лучшая альтернатива | Почему |
|--------|---------|----------------------|--------|
| Скриншот UI → код | **Gemma 4** ⭐ | -- | Специально натренирована на screenshot-to-code, native function calling даёт чистый JSX |
| Function calling по картинке | **Gemma 4** ⭐ | -- | Vision + tool use в одной модели без лишних оберток |
| Длинный контекст 256K + vision | **Gemma 4** ⭐ | -- | 256K больше всех VLM на платформе |
| Чистый OCR (не-латинские шрифты, мелкий текст) | -- | [Qwen3-VL 30B-A3B](qwen3-vl.md#30b-a3b) | OCR -- профильная сила Qwen3-VL, у Gemma слабее |
| Math reasoning по диаграмме | -- | [InternVL3.5-38B](internvl.md#3-5-38b) | InternViT-6B даёт лучше chart understanding и math |
| Vision + audio (omni) | -- | [Qwen2.5-Omni 7B](qwen25-omni.md) | Gemma 4 не работает с аудио |
| Простые "опиши фото", quick chat | Gemma 4 | [Qwen3-VL 30B-A3B](qwen3-vl.md#30b-a3b) | Обе работают, выбор по тому что уже запущено |
| Multi-image сравнение | -- | [Qwen3-VL 30B-A3B](qwen3-vl.md#30b-a3b) или [Pixtral 12B](pixtral.md) | У Gemma нет специальной оптимизации под multi-image |

**Главное правило**: брать Gemma 4 когда нужно **vision + (function calling или длинный контекст или screenshot-to-code)**. Если задача чисто OCR или reasoning над диаграммой -- альтернативы лучше.

### Параллельные конфигурации

Gemma 4 Q6_K_XL занимает 22 GiB + mmproj 1.2 GiB = ~23 GiB, оставляя 97 GiB запаса. Можно держать параллельно:

- **Стандарт**: Gemma 4 (23 GiB) + FIM 1.5B (2 GiB) = 25 GiB → 95 GiB запас
- **С Coder Next**: Gemma 4 (23 GiB) + Coder Next (45 GiB) + FIM (2 GiB) = 70 GiB → 50 GiB запас
- **С Devstral 2**: Gemma 4 (23 GiB) + Devstral 2 24B (14 GiB) + FIM (2 GiB) = 39 GiB → 81 GiB запас
- **Vision-zoo**: Gemma 4 (23 GiB) + Qwen3-VL 30B-A3B (20 GiB) + InternVL3.5-38B (27 GiB) = 70 GiB → 50 GiB запас (`--parallel 1` везде)

В последней конфигурации можно переключаться между всеми тремя vision-моделями через `/model` в opencode без перезагрузки серверов -- идеально для сравнительных задач.

## Базовые сценарии (простые)

Описательные / OCR задачи:

- **"Опиши что на скриншоте"** -- быстрый описательный ответ
- **OCR простого документа** (счёт, чек, скан паспорта) → структурированный JSON
- **"Что не так на этом фото?"** -- описание видимых проблем (битый интерфейс, неправильная вёрстка)
- **Перевод текста с фото вывески / меню** -- встроенный multilingual + OCR без отдельной модели
- **Объяснение мема** -- визуальный контекст + культурная отсылка
- **Распознавание emoji / стикеров** в скриншоте чата -- для анализа эмоционального тона переписки

Структурное извлечение:

- **Визитка → vCard JSON** -- скан визитки → структурированный контакт с полями name, email, phone, company
- **Whiteboard / флипчарт → markdown** -- фото сессии → структурированные заметки
- **Скриншот чата (Telegram, Slack, Discord) → структурированная история** -- для архивации или анализа sentiment
- **Каталогизация коллекции** (книги на полке, инструменты в гараже, домашняя коллекция) → JSON inventory
- **Распознавание объектов на фото** для inventory / каталогизации

Code / dev tasks:

- **Скриншот ошибки IDE → объяснение и фикс** в [opencode](../../ai-agents/agents/opencode.md)
- **"Сгенерируй HTML по этому скетчу"** -- простая wireframe → стартовый каркас
- **Скриншот terminal → разбор команды** ("что делает эта команда?", "почему упала?")

Domain-specific:

- **Идентификация растения / животного** по фото (фото листа → название вида + основные параметры ухода)
- **Проверка визуальной идентичности логотипа** (правильный ли это лого бренда / типографика по гайдлайну)
- **Описание блюда по фото** для меню или каталога (название, ингредиенты, примерная калорийность)

Извлечение данных из бытовых документов:

- **Билет (авиа / ж/д / автобус) → JSON** -- маршрут, дата, время, места, ПЦР/багаж в structured output
- **Посадочный талон → calendar event** через function calling: распознаёт время, сразу вызывает `add_to_calendar`
- **Фото счёта в ресторане → split bill** (детальный список позиций с ценами, разделение на participants)
- **Фото тетради ученика с задачей → пошаговое решение** -- OCR + математический reasoning через `<|think|>`
- **Фото показаний счётчика** (электричество, газ, вода) → JSON с актуальной цифрой (полезно для умного дома, где ручной ввод утомляет)
- **Фото блюда → расчёт калорий** с estimation размера порции (ошибка ±15-20%, но достаточно для трекинга)

Все эти задачи 30B-A3B Instruct тоже выполнит, но Gemma 4 быстрее на function-call'ах и стабильнее на длинных prompt'ах. Главное преимущество для простых задач -- **скорость** (3.8B active даёт ~70 tok/s) и **поддержка 140 языков** в одной модели.

## Сложные сценарии

### 1. Screenshot-to-code production-ready компонент

Скриншот реального UI компонента (например карточка товара из e-commerce). Gemma 4:
- Распознаёт компоненты Material UI / Ant Design / Tailwind
- Генерирует **полный JSX/TSX** с правильными props
- Учитывает adaptive layout (mobile/desktop варианты, если поданы оба)
- Через function calling может **сразу** вызвать `write_file` с готовым компонентом

Минус по сравнению с [Qwen3-VL](qwen3-vl.md) -- хуже определяет конкретную UI-библиотеку, склонна делать "плоский" Tailwind вариант. Плюс -- более чистый код на выходе.

### 2. Анализ wireframe и итеративная разработка

Загрузить wireframe в Figma-стиле → Gemma 4 с function calling:
- Извлекает структуру компонентов в JSON
- Для каждого компонента вызывает `create_file` с TSX-кодом
- Подключает routing, state management через дополнительные tool calls
- Возвращает план файловой структуры

Один промпт → готовый стартовый шаблон проекта. Это работает потому что Gemma 4 -- одна из немногих VLM с **native function calling** без оберток.

### 3. Reasoning по длинной видеоинструкции

Загрузить серию кадров (20-50 скриншотов) из обучающего видео + текстовый prompt с задачей. Gemma 4:
- Использует 256K контекст для удержания всех кадров одновременно (в отличие от 128K у Qwen3-VL)
- Через `<|think|>` reasoning-режим связывает кадры в последовательность действий
- Восстанавливает алгоритм или процедуру из видео в виде пошаговой инструкции
- Может сразу написать код для воспроизведения (например туториал по библиотеке → рабочий пример)

256K контекст -- ключевой плюс Gemma 4 для multi-frame задач.

### 4. Multi-document reasoning с переключением функций

Загрузить несколько разнородных документов: PDF договора, скриншот таблицы, фото подписи. Gemma 4:
- Через function calling вызывает разные tools для каждого типа: `extract_pdf_text`, `parse_table_image`, `verify_signature`
- Координирует results между tool calls
- Выдаёт сводное заключение

Это близко к agentic-режиму, но в одной модели без оркестратора.

### 5. UI accessibility audit

Скриншот веб-страницы → Gemma 4 проверяет:
- Контрастность текста (визуально оценивает)
- Размер кликабельных областей (>= 44×44 для mobile)
- Наличие визуальных подсказок для interactive элементов
- Иерархия заголовков
- Альтернативный текст для иконок (если виден alt в DOM-скриншоте)

Через function calling выдаёт structured report с severity для каждой проблемы.

### 6. Live debugging UI с серией скриншотов

3-5 скриншотов одного экрана в разных состояниях (idle / loading / error / success). Gemma 4:
- Восстанавливает state machine
- Находит **противоречия** между состояниями (например loader не убирается на success)
- Предлагает фикс на уровне кода (если виден React DevTools-скриншот)
- Через function calling сразу вызывает `apply_patch`

### 7. Reasoning по техническому скриншоту с длинной инструкцией

Сложный случай: скриншот системы (например Kubernetes Dashboard, Grafana) + 20K токенов документации + промпт "помоги разобраться". Gemma 4:
- Использует 256K контекст для удержания всей документации
- Анализирует скриншот в контексте документации
- Через thinking-режим строит цепочку рассуждений
- Возвращает structured ответ с references на конкретные места в документации

### 8. Form filling из скана документа + business logic

Сканированная анкета или счёт → Gemma 4:
- OCR извлекает поля
- Function calling вызывает `validate_field` для каждого
- Применяет business rules (дата не в будущем, сумма с НДС = сумма без НДС × 1.20)
- Заполняет форму в системе через tool calls

Альтернатива -- [Qwen3-VL](qwen3-vl.md#30b-a3b) даст лучше OCR, но без function calling придётся делать orchestration снаружи.

### 9. Bounding box для UI-тестирования

Скриншот веб-приложения → промпт: "Найди все интерактивные элементы и верни bounding boxes в JSON"

Gemma 4 возвращает нативный JSON с координатами:
```json
[
  {"label": "Login button", "bbox": [520, 400, 720, 450], "type": "button"},
  {"label": "Email input", "bbox": [520, 280, 720, 320], "type": "input"},
  {"label": "Hamburger menu", "bbox": [20, 10, 60, 50], "type": "icon"}
]
```

Координаты -- в пикселях исходного скриншота, `[x1, y1, x2, y2]`. Это **нативная способность** (не grammar constraint), модель обучена возвращать bbox как часть structured output.

Применения:
- **Cypress/Playwright test generation**: bbox → координаты для `page.click(x, y)`
- **Visual regression**: сравнить bbox между двумя скриншотами (до и после деплоя)
- **Accessibility audit**: проверить что все clickable areas >= 44x44 px
- **Auto-annotation**: разметить UI-элементы для ML-датасета

### 10. Настройка image token budget для batch-обработки

При batch-обработке большого количества изображений (1000+ скриншотов из e2e тестов) -- важна скорость. Gemma 4 позволяет **снизить token budget** до 70 токенов на кадр:

- **70 tokens**: ~50 мс на кадр на Strix Halo. Достаточно для classification ("Pass/Fail"), detection грубых ошибок, тривиального OCR
- **280 tokens** (default): ~150 мс на кадр. OCR среднего текста, описание сцены
- **1120 tokens**: ~500 мс на кадр. Pixel-level детализация, мелкий текст

Для batch в 1000 скриншотов разница: 50 сек (70 tokens) vs 8 мин (1120 tokens). Выбор token budget через prompt engineering: "Describe this screenshot very briefly" (модель выбирает меньше tokens) vs "Provide detailed OCR of all text visible" (модель выбирает больше).

### 11. Multimodal RAG: изображения + документы в одном контексте

256K контекст позволяет загрузить **одновременно** документ (20K токенов текста) + серию скриншотов (10 × 280 = 2800 vision tokens) + промпт. Всё в одном запросе, без RAG-пайплайна:

Пример: "Вот техническое задание (PDF извлечённый в текст) и 10 скриншотов текущего UI. Какие требования из ТЗ не реализованы?"

Gemma 4:
1. Парсит текст ТЗ, извлекает требования
2. Анализирует скриншоты, находит реализованное
3. Cross-reference: требования vs визуальное состояние
4. Выдаёт structured список несоответствий

Это **multimodal RAG в одном запросе** -- без embedding модели, без vector store, без чанкинга. Работает пока всё влезает в 256K.

### 12. Code review по скриншотам (before/after)

Два скриншота: `before.png` (старый UI) и `after.png` (после PR). Промпт: "Сравни оба скриншота, опиши что изменилось визуально"

Gemma 4:
- Извлекает layout differences
- Находит новые/удалённые элементы
- Замечает color changes, font changes, spacing differences
- Через function calling может сразу вызвать `create_issue` если найдена регрессия

Use case: visual code review в CI pipeline -- автоматическая проверка что PR не сломал UI.

### 13. Chart-to-data extraction для финансов и аналитики

Финансовый отчёт в виде графика (line chart, bar chart, candlestick) → Gemma 4:
- Распознаёт оси (категории, числовые шкалы, даты)
- Извлекает значения для каждой точки данных
- Через function calling вызывает `save_to_csv` со structured данными
- Учитывает legend, color-coding для multi-series данных

Точность зависит от плотности данных:
- **Bar chart** (10-20 столбцов с метками): ~95% accuracy
- **Line chart** (≤30 точек): ~85%
- **Dense line chart** (50+ точек, overlap): ~70-80%, последние теряются

Production-паттерн: Gemma 4 как первичный экстрактор + human-in-the-loop проверка через Streamlit UI для редактирования спорных значений.

### 14. Whiteboard mathematical proof → LaTeX + проверка

Фото доски с математическим выводом или доказательством → Gemma 4:
- Распознаёт математическую нотацию (греческие буквы, индексы, дроби, интегралы, матрицы)
- Восстанавливает структуру (assumption → step → step → conclusion)
- Возвращает LaTeX-исходник
- Через thinking-режим может **проверить** доказательство на корректность шагов

Альтернатива -- [Qwen3-VL](qwen3-vl.md) даёт лучше OCR математического текста (выше точность распознавания символов), но Gemma 4 сильнее в reasoning над сутью. Для combined task "распознай + проверь" Gemma 4 выигрывает за счёт встроенного thinking-режима.

### 15. Game UI / level design analysis

Скриншот видеоигры → Gemma 4:
- Идентифицирует жанр и потенциальные механики
- Описывает UI: HUD, mini-map, inventory, dialog system, quest tracker
- Анализирует UX-решения (заметность важных элементов, ergonomic flow, информационная плотность)
- Находит проблемы (clutter, низкая контрастность, missing affordances для interactive)

Use case'ы:
- **Game design review** -- автоматическая первичная оценка UI билдов
- **Конкурентная разведка** -- сравнительный анализ UI конкурентов
- **Accessibility-аудит для геймеров** -- проверка контраста, размера элементов, цветовых кодировок (color-blind mode)

### 16. Architectural / engineering drawing → структурированное описание

Технический чертёж (ER-диаграмма базы данных, network diagram, system architecture, data flow) → Gemma 4:
- Извлекает узлы и связи в JSON-граф
- Описывает архитектуру в свободной форме на русском
- Через function calling может сгенерировать **Mermaid / PlantUML / D2** код для воспроизведения чертежа в редактируемом виде
- Находит несоответствия (missing connections, orphan nodes, dangling references)

256K контекст полезен: можно загрузить **несколько связанных диаграмм одновременно** (системная + database + network) для cross-reference анализа -- "проверь что entities из ER-диаграммы соответствуют сервисам в архитектурной".

### 17. Video QA: понимание последовательности из ключевых кадров

20-50 ключевых кадров обучающего/демо видео + промпт "извлеки пошаговую инструкцию" → Gemma 4:
- 256K контекст вмещает все кадры (50 × 280 = 14K vision tokens, или 50 × 1120 = 56K с pixel-level детализацией)
- Через `<|think|>` строит хронологию событий
- Учитывает изменения между кадрами (state transitions)
- Возвращает пошаговую инструкцию или транскрипт действий

Альтернативы у которых меньше контекст ([qwen3-vl](qwen3-vl.md) -- 128K, [internvl](internvl.md) -- 32K) на длинных последовательностях не справятся либо потребуют RAG-пайплайна. Gemma 4 делает это в одном запросе.

### 18. Multimodal sentiment analysis

Скриншоты соцсетей (Twitter, Telegram, Reddit threads) → Gemma 4:
- OCR извлекает текст постов и комментариев
- Анализирует визуальные признаки (emoji, стикеры, мемы, screenshots в screenshots)
- Учитывает реакции (количество лайков, реплаев) и threading-структуру
- Возвращает структурированную оценку sentiment с конкретными цитатами

Особенность Gemma 4 здесь -- **multilingual sentiment**: правильно интерпретирует русский / украинский / английский в смешанных тредах без отдельной языковой модели.

### 19. PII compliance audit на скриншотах приложения

Серия скриншотов внутреннего приложения (admin-панели, customer support tools, internal dashboards) → Gemma 4:
- Через bbox prediction находит **места отображения PII** (email, телефон, номера карт, паспортные данные, IP-адреса, employee IDs)
- Через function calling классифицирует severity (PII high/medium/low по GDPR / 152-ФЗ)
- Возвращает structured report для compliance team: `[{screen, bbox, pii_type, severity, recommendation}]`
- Cross-reference с whitelist (например "поле email допустимо в admin-панели, но не на public-страницах")

Применение: автоматизированный privacy-аудит UI перед production-деплоем. Pattern: CI на каждый PR с UI-изменениями делает screenshots → Gemma 4 audit → блокирует merge при найденных нарушениях.

### 20. Cross-version UI regression: эволюция приложения за релизы

Серия скриншотов одного экрана из 4-5 последовательных релизов (v1.0 → v1.1 → ... → v1.4) загружается в 256K контекст. Gemma 4:
- Восстанавливает **timeline визуальных изменений** (что добавилось, что удалилось, что переместилось в каждом релизе)
- Идентифицирует **regression** -- фичи которые были в v1.2 но пропали в v1.3 без changelog-записи
- Через function calling вызывает `query_changelog` для каждого релиза и cross-references с visual changes
- Возвращает отчёт: "Visible changes vs documented changes mismatch"

Применение: документирование UX-эволюции продукта, обнаружение untracked changes (когда дизайнер сдвинул что-то без записи в changelog). 256K контекст ключевой -- альтернативные VLM с 32-128K не вмещают серию из 5 крупных скриншотов.

### 21. Educational tutor: персонализированная обратная связь по тетради

Фото тетради ученика (математика / физика / химия) → Gemma 4 через thinking-режим:
- OCR извлекает решение
- Через `<|think|>` **проходит решение шаг за шагом**, отмечая место первой ошибки
- Различает **типы ошибок**: вычислительная, концептуальная (непонимание формулы), методологическая (неправильный подход)
- Через function calling вызывает `lookup_textbook(topic)` для подтягивания релевантных правил
- Формирует feedback: "Ошибка в шаге 3 -- ты раскрыл скобки до приведения подобных. Правило: ..."

Multilingual важен: ученик может писать частично по-русски (вывод), частично формулами/латиницей. Gemma 4 понимает обе части без отдельной OCR-модели.

### 22. Diagram-as-code reverse: фото → Mermaid / PlantUML / D2

Фото whiteboard'а или черновой схемы (UML, sequence, state machine, ER) → Gemma 4:
- Распознаёт элементы (узлы, стрелки, лейблы, типы связей one-to-many / many-to-many)
- Восстанавливает структуру в graph form
- Через function calling вызывает `generate_mermaid` или `generate_plantuml` или `generate_d2` (выбор по пожеланию пользователя)
- Возвращает **исполняемый код** диаграммы

Это особенно ценно при offboarding -- сотрудник уходит, на доске остались наброски системы. Gemma 4 переводит в редактируемый формат за минуту, не теряя информации.

Альтернативы: [InternVL3.5](internvl.md) даст лучше парсинг точных значений (если на схеме есть числа), но Gemma 4 с native function calling выдаёт сразу исполняемый код в один turn.

### 23. Asymmetric A/B testing: одно и то же UI у двух пользователей

Два скриншота: User A (treatment группа) и User B (control группа) одного и того же экрана → Gemma 4:
- Описывает **визуальные различия** между вариантами
- Через bbox prediction находит **конкретные элементы** которые отличаются
- Через function calling вызывает `query_analytics_for_variant(A)` и `query_analytics_for_variant(B)` для подтягивания метрик
- Связывает визуальную разницу с поведенческой метрикой ("кнопка переехала на 50px вниз → CTR упал на 12%")

Это автоматизация UX-аналитики, которая обычно требует дизайнера + аналитика. Gemma 4 делает первичный pass за 30 секунд.

### Антипаттерны

- **Не использовать Gemma 4 для чистого OCR мелкого текста** -- [Qwen3-VL](qwen3-vl.md) точнее на 10-15%
- **Не использовать Gemma 4 для math по диаграммам** -- [InternVL3.5](internvl.md) даёт более глубокое reasoning над схемами
- **Не запускать без `--parallel 1 --no-mmap`** -- sliding window cache схлопывается, OOM
- **Не пытаться использовать `--cache-reuse` agressively** -- llama.cpp игнорирует для Gemma 4, prefix пересчитывается каждый turn (см. слабые стороны выше)
- **Не использовать F32 mmproj** -- BF16 даёт идентичное качество при половинном размере
- **Не загружать 31B dense для интерактивного чата** -- ~5 tok/s на Strix Halo, тяжёлый memory-bound. Лучше 26B-A4B (MoE, ~70 tok/s)

```bash
# Через Vulkan-пресет (mmproj подключен, контекст 250K)
./scripts/inference/vulkan/preset/gemma4.sh -d
```

Пресет автоматически:
- Подключает `--mmproj $MODELS_DIR/mmproj-BF16.gguf` -- vision-encoder для multimodal
- `-c 250000` (250K) -- расширен с исходных 64K, на платформе 120 GiB unified memory хватает
- `--parallel 4` -- 4 слота для параллельных запросов
- `--no-mmap` -- модель сразу в RAM, без mmap-overhead
- `--cache-reuse 256` -- установлен, но llama.cpp игнорирует его на Gemma 4 из-за sliding window
- `--jinja` -- Jinja2 chat-template для function calling
