# Qwen3-VL (Alibaba, 2026)

> Vision-флагман от Qwen, лучший OCR/document understanding в open-source.

**Тип**: MoE (3B-22B active)
**Лицензия**: Apache 2.0
**Статус на сервере**: скачана (30B-A3B Instruct Q4_K_M + mmproj F16)
**Направления**: [vision](../vision.md)

## Обзор

Qwen3-VL -- специализированная vision-серия от Alibaba 2026. 30B-A3B вариант -- быстрая MoE с 3B активных параметров, как у [qwen3-coder Next](qwen3-coder.md), но с vision. 235B-A22B -- флагман уровня Gemini-2.5-Pro / GPT-5 на multimodal-бенчмарках.

### 30B-A3B Instruct {#30b-a3b}

Рекомендуемая модель для платформы. MoE с 3B active -- скорость как у Qwen3-Coder.

### 235B-A22B {#235b-a22b}

На пределе VRAM (135 GB Q4 + ~3 GB mmproj + контекст). 22B активных -- значительно мощнее в reasoning чем 3B-A3B вариант. Уровень Gemini-2.5-Pro / GPT-5 на multimodal-бенчмарках.

**Ограничения на платформе**:
- Q4_K_M: 135 GB модели + 3 GB mmproj = 138 GB -- помещается в 120 GiB **только впритык** через GTT-overflow
- Реалистично: контекст ограничен ~16-32K
- Скорость: ~22 tok/s (по аналогии с Qwen3.5-122B-A10B)
- Загрузка модели: 3-5 минут

**Когда брать 235B вместо 30B-A3B**:
- Если простые задачи (OCR счёта, описание фото) -- хватит 30B-A3B (быстрее, помещается без проблем)
- Если задача требует **глубокого визуального reasoning** или это **критичный одноразовый запрос** -- 235B даёт результат уровня закрытых флагманов

#### Сложные сценарии где 235B действительно нужна

**1. Reverse engineering сложного UI / системы**

Скриншот корпоративного интерфейса (например ERP-системы или биржевого терминала) с десятками виджетов, графиков, контекстных меню и метрик. 235B может:
- Извлечь иерархию компонентов и их функциональное назначение
- Восстановить data flow между виджетами по визуальным связям
- Предложить упрощённый дизайн с обоснованием каждого решения
- Найти UX-проблемы (несогласованность, скрытые функции, accessibility issues)

30B-A3B справится поверхностно, 235B сделает экспертный аудит.

**2. Анализ научных публикаций с диаграммами**

Загрузить PDF научной статьи с графиками, формулами, таблицами, схемами эксперимента. 235B может:
- Проверить **внутреннюю согласованность** -- соответствие текста графикам, цифр в таблицах формулам
- Найти **методологические проблемы** в схеме эксперимента
- Сравнить с **другой статьёй** (загрузить две одновременно) -- найти противоречия в выводах
- Воспроизвести цифры из графика и пересчитать выводы

Это уровень peer-review.

**3. Forensic-анализ изображения**

Подозрительное фото (документ, удостоверение, сцена). 235B может:
- Найти **признаки фотошопа** (несоответствие теней, перспективы, артефакты сжатия в разных областях)
- Распознать **манипуляции с метаданными** через визуальные несоответствия (например тени не соответствуют времени по EXIF)
- Идентифицировать **несовместимости** (одежда не по сезону для геолокации, исторические анахронизмы)
- Дать аргументированное заключение с указанием конкретных областей подозрения

**4. Перевод документа с сохранением layout**

Сложный документ (договор на корейском, китайская инструкция, арабская брошюра) с:
- Форматированием (списки, таблицы, выделения)
- Подписями к иллюстрациям
- Сносками и ссылками
- Юридическими формулировками

235B может перевести сохраняя **структуру и нюансы**, а не просто текст. Распознаёт устойчивые юридические/технические выражения и переводит идиоматично.

**5. Архитектурный анализ кодбейса по скриншотам**

Серия скриншотов из IDE: дерево проекта, открытые файлы с кодом, output терминала, диаграммы зависимостей в плагине. 235B может:
- Восстановить **архитектуру системы** по visible-частям
- Найти **антипаттерны** (циркулярные зависимости, god-objects, дублирование)
- Предложить **рефакторинг** на уровне конкретных файлов
- Объяснить **почему именно так** не работает (с указанием на проблемные строки в коде)

Это работа сениор-инженера на ревью.

**6. Multi-document compliance check**

Загрузить набор документов (договор, спецификации, регуляторные требования, политики компании) в различных форматах -- PDF, скриншоты Word, фото подписей. 235B может:
- Проверить **взаимную совместимость** документов
- Найти **противоречия** между ними
- Идентифицировать **отсутствующие пункты** для соответствия regulatory requirements
- Сгенерировать структурированный отчёт о найденных issues

**7. Анализ трейдинг-стратегии по графикам**

Скриншоты графиков с разными таймфреймами одного актива (1m, 5m, 1h, 1d) + индикаторы. 235B может:
- Распознать **паттерны** (head-and-shoulders, divergence, support/resistance)
- Найти **соответствие/расхождение** между таймфреймами
- Дать **рекомендацию по входу/выходу** с обоснованием
- Указать **риски** (false breakout, low volume)

Не торговая система, но качественный manual analysis.

**8. Reverse-engineering алгоритма по визуализации**

Анимация/видео работы алгоритма (например визуализация сортировки или ML-обучения). Несколько кадров → 235B:
- Восстановить **код алгоритма**
- Объяснить **сложность по времени и памяти**
- Сравнить с известными алгоритмами
- Найти **оптимизации**

**9. Творческий разбор картины / фотографии**

Фото произведения искусства или композиции. 235B может дать **искусствоведческий анализ**:
- Композиция (правило третей, золотое сечение, диагонали)
- Цветовая палитра и её эмоциональное воздействие
- Стилистическая принадлежность и влияния
- Технические особенности (фокус, экспозиция, постобработка)
- Контекст эпохи и автора (если узнаётся)

**10. Multi-image storytelling и продолжение**

Несколько фотографий → 235B восстанавливает **возможный сюжет**, описывает связи между изображениями, генерирует продолжение истории, или наоборот восстанавливает что было до. Полезно для:
- Сценарной работы
- Исторического анализа фотоархивов
- Crime investigation (по серии фото места происшествия)

## Workflow: обратный инжиниринг UI по скриншоту

Отдельный сценарий, заслуживающий пошагового playbook. Цель -- по скриншоту (или серии) восстановить: компонентное дерево, использованный фреймворк, библиотеку компонентов, потоки данных, архитектурные решения. Применим к собственным конкурентным анализам, наследованию legacy-интерфейсов без исходников, аудиту UX, обучению.

### Когда брать какую модель

| Задача | Модель | Причина |
|--------|--------|---------|
| Один экран, базовая идентификация стека | [30B-A3B Instruct](#30b-a3b) | Помещается с запасом, быстро, достаточно для типичных React/Vue/Material UI |
| Один экран, нестандартный/сложный UI | 30B-A3B Thinking | Reasoning над визуальными деталями, но скорость падает |
| 5+ скриншотов одного экрана в разных состояниях | [235B-A22B](#235b-a22b) | 22B active вытягивают cross-frame reasoning |
| Серия экранов одного flow (login → dashboard → settings) | 235B-A22B | Восстановление data flow между экранами требует глубокого reasoning |
| Только публичный продукт без API | [30B-A3B](#30b-a3b) + опциональный второй проход через [Gemma 4 26B-A4B](gemma4.md) | Gemma натренирована на screenshot-to-code, даёт более чистый JSX |

Гибридная стратегия: первый проход через 30B-A3B (быстрый, дешёвый), при недостатке детализации -- селективный второй проход через 235B-A22B на проблемных областях. Параллельно держать оба сервера на платформе невозможно (135 + 18 GiB > 120), поэтому переключаться через `tmux send-keys` или останавливать один пресет.

### Подготовка ввода

1. **Высокое разрешение** -- минимум 1920×1080, лучше 2560×1440. Vision-encoder Qwen3-VL ресайзит в окно 28×28-патчей; на низком разрешении мелкий текст и иконки теряются.
2. **Не сжимать в JPEG** перед отправкой -- артефакты сжатия путают модель в области градиентов и тонких линий. PNG или lossless WebP.
3. **Серия состояний одного компонента** -- собрать 3-5 скриншотов: idle / hover / active / disabled / error. Это даёт модели понимание интерактивных переходов, что критично для определения framework (React Hooks vs Vue Composition API дают разные паттерны изменения DOM).
4. **DevTools-скриншоты как контекст** -- если есть доступ к браузеру, дополнительно снять Elements panel, Network tab (XHR/Fetch), Computed styles. Загружать вместе с UI -- модель сможет верифицировать гипотезы о стеке против реального DOM.
5. **Обрезка по областям интереса** -- отдельные скриншоты для header/sidebar/main content/footer, плюс один общий обзорный. Помогает модели не путать соседние области и даёт фокус для каждого вопроса.

### Многошаговая стратегия (4 фазы)

#### Фаза 1: Идентификация стека и компонентной библиотеки

Промпт-шаблон:

```
Это скриншот веб-интерфейса. Выполни инвентаризацию:
1. Какой UI-фреймворк наиболее вероятен (React / Vue / Angular / Svelte / vanilla)? Аргументируй визуальными признаками.
2. Какая компонентная библиотека (Material UI / Ant Design / Chakra / Tailwind UI / shadcn/ui / Bootstrap / собственная)? Укажи 3-5 признаков для каждой кандидата.
3. CSS-подход (Tailwind utility-first / CSS-in-JS / SCSS modules / vanilla)? Какие визуальные маркеры подсказывают?
4. Темизация: какие primary/secondary/accent цвета, типографика (шрифт, размеры), spacing scale.
5. Иконки: какой набор (Lucide / Heroicons / Font Awesome / Material Icons / Phosphor / собственный)?
6. Дай confidence score 0-1 для каждого вывода.

Формат ответа: structured JSON с полями {framework, ui_library, css_approach, theme, icons, confidence_per_field}.
```

Почему именно так:
- Просим **аргументацию визуальных признаков** -- модель не угадывает, а ищет конкретные маркеры (закруглённые углы 8px → Material, square 0px → Ant Design, gradient borders → Tailwind UI)
- Просим **confidence score** -- видно, где модель уверена, а где гадает (полезно для решения, нужен ли второй проход через 235B)
- Структурированный JSON -- легко парсить программно для следующих фаз

#### Фаза 2: Декомпозиция компонентного дерева

Промпт после получения ответа Фазы 1:

```
На основе предположения, что это <framework> + <ui_library>, разложи интерфейс на компонентное дерево.

Для каждого компонента укажи:
- Название компонента из <ui_library> (например `<Card>`, `<DataTable>`, `<Sidebar>`)
- Props, которые видны или подразумеваются (variant, size, color, disabled state)
- Дочерние компоненты (рекурсивно)
- Какие данные он отображает (имя поля и тип, как ты его видишь)

Формат: вложенное дерево JSX с комментариями про data binding.
```

Что получаем: каркас React/Vue-кода. Это уже 60-70% реверса -- можно копировать в проект и доводить руками.

#### Фаза 3: Восстановление data flow

Здесь начинается то, ради чего стоит брать **235B-A22B**: множественные источники данных, связи между виджетами.

Промпт:

```
Проанализируй data flow в этом интерфейсе.

1. Какие сущности отображаются (пользователь / заказ / товар / транзакция / etc)? Перечисли поля для каждой.
2. Какие виджеты показывают одну и ту же сущность в разных представлениях (карточка vs таблица vs график)?
3. Какие фильтры / сортировки / поисковые поля видны? К каким виджетам они применяются?
4. Какие действия (кнопки, меню) триггерят изменения в каких виджетах? Восстанови наблюдаемые dependency-цепочки.
5. Какие данные явно загружаются асинхронно (skeleton loaders, progress bars, "loading...")?
6. Есть ли признаки real-time обновлений (badges с числами, индикаторы статуса, timestamp "2 минуты назад")?

Формат: список сущностей + dependency graph между виджетами в формате `widget_a --triggers--> widget_b`.
```

#### Фаза 4: Реконструкция архитектуры состояния

Промпт:

```
На основе data flow из предыдущего шага, предложи архитектуру состояния (state management).

1. Что должно жить в global state (Redux / Zustand / Pinia / Vuex / Context)?
2. Что в server state (React Query / SWR / Apollo / TanStack Query)?
3. Что в local state компонентов?
4. Какие endpoints API логически должны существовать (REST или GraphQL)? Перечисли endpoint-ы с указанием, какой виджет их потребляет.
5. WebSocket / SSE -- есть ли признаки real-time подписок?

Формат: список stores + список endpoint-ов + связь "endpoint X → виджеты Y, Z".
```

Этот шаг практически невозможен на 30B-A3B без сильных галлюцинаций -- именно здесь 235B оправдывает свой VRAM.

### Антипаттерны и ловушки

- **Не давать всё одним промптом**. Один большой "проанализируй интерфейс" даёт поверхностный ответ. Многошаговая декомпозиция выше -- это не лишние итерации, это способ заставить модель удерживать рабочую гипотезу между фазами.
- **Не верить confidence < 0.7 без проверки**. На неоднозначных интерфейсах модель часто угадывает популярный фреймворк (React + Material UI) даже когда это shadcn или собственная библиотека. Проверять через DevTools-скриншот.
- **Не использовать [Gemma 4 26B-A4B](gemma4.md) для анализа стека**. Gemma натренирована на **генерацию** screenshot-to-code, она склонна сразу выдавать JSX без анализа фреймворка. Брать её только для финального шага -- генерации кода после идентификации стека через Qwen3-VL.
- **Не путать визуальные признаки с реальностью**: иконка с круглым углом не означает Material UI, это может быть Tailwind UI с радиусом 8px. Просить модель давать **3-5 независимых маркеров** на каждое утверждение.
- **Не игнорировать responsive-поведение**: если есть mobile/desktop версии одного экрана -- грузить обе. Breakpoint-логика часто выдаёт фреймворк (Bootstrap grid классы vs Tailwind responsive префиксы vs CSS Grid).

### Параллельные источники для верификации

После Фазы 1 ([Qwen3-VL 30B-A3B](#30b-a3b) выдала "вероятно Material UI"), полезно:
1. Скриншот загрузить в публичные MUI/Ant Design демо-страницы вручную и сравнить визуально (никакого AI -- глазами).
2. Если публичный продукт -- открыть Wappalyzer-extension в браузере и проверить.
3. Через DevTools посмотреть на CSS-классы (`.MuiButton-root` -- однозначный маркер Material UI; `.ant-btn` -- Ant Design; utility-классы вроде `flex items-center px-4 py-2` -- Tailwind).

Модель даёт **гипотезу**, не приговор. Любые critical-решения (миграция legacy на тот же стек) перепроверять руками.

### Идеальный output

После 4 фаз получаем (для типичного React + MUI dashboard):

1. **Стек**: React 18, Material UI v5, TypeScript, React Query для server state, Zustand для UI state
2. **Компонентное дерево**: вложенный JSX (~80% точности)
3. **Сущности**: User, Order, Product со списком полей
4. **Endpoints**: `GET /api/orders?status=&page=&limit=`, `POST /api/orders/:id/refund`, `GET /api/users/me`
5. **State stores**: ordersStore (Zustand), useOrdersQuery (React Query), useAuthContext (Context API)

Это **рабочая основа** для воспроизведения интерфейса с нуля или для понимания чужого продукта на уровне архитектуры за 30-60 минут работы вместо нескольких дней реверс-инжиниринга глазами.

## Сильные кейсы

- **OCR на 30+ языках** -- лучшая в open-source. Русский, китайский, японский, арабский в одном потоке
- **Document understanding** -- структурированный JSON-вывод из PDF/счетов/договоров
- **Video understanding** -- ввод нескольких кадров, action recognition
- **Structured output** -- строго валидный JSON по schema
- **Agentic GUI** -- кликабельные координаты на скриншотах
- **Чтение комиксов/манги** -- понимает порядок панелей, текст в баблах
- **UI-to-code** -- скриншот макета → HTML/React

## Слабые стороны

- Reasoning -- через Thinking-вариант (выше latency)
- 235B на пределе VRAM, мало места под контекст

## Идеальные сценарии

- Парсинг чеков/счетов → JSON для бухгалтерии
- Извлечение данных из научных публикаций
- Browser automation: скриншот → следующий клик
- Многоязычный OCR
- Замена платных Document AI сервисов
