# Генерация текста

Генерация текста в LLM -- авторегрессивный процесс. Модель
предсказывает токены по одному, каждый следующий зависит от всех
предыдущих. Процесс состоит из двух фаз: prompt processing
(параллельная обработка входа) и token generation (последовательная
генерация выхода).

## Авторегрессия

"Авторегрессия" означает: каждый новый элемент зависит от всех
предыдущих элементов. LLM генерирует текст по одному токену за шаг.

```
Шаг 1:  [Столица] [Франции] [--]  -->  [это]
Шаг 2:  [Столица] [Франции] [--] [это]  -->  [Париж]
Шаг 3:  [Столица] [Франции] [--] [это] [Париж]  -->  [.]
Шаг 4:  [Столица] [Франции] [--] [это] [Париж] [.]  -->  [EOS]
                                                             ^
                                                        конец генерации
```

На каждом шаге:
1. Все предыдущие токены (prompt + уже сгенерированные) подаются
   на вход модели
2. Модель вычисляет вероятности для следующего токена
3. Выбирается один токен (сэмплирование --
   [sampling.md](sampling.md))
4. Выбранный токен добавляется к последовательности
5. Повторяется до EOS или достижения лимита

### Условия остановки

Генерация прекращается, когда:

| Условие | Описание |
|---------|----------|
| EOS-токен | Модель сгенерировала end-of-sequence токен |
| max_tokens | Достигнут лимит длины ответа |
| stop-строка | Сгенерированный текст содержит стоп-последовательность |
| Контекст заполнен | input + output >= context window |

```bash
# в llama-server
llama-server \
    --model model.gguf \
    --ctx-size 8192 \       # контекстное окно
    --n-predict 2048         # максимум токенов в ответе
```


## Две фазы inference

Inference LLM состоит из двух принципиально разных фаз:

```
+------------------------------------------------------------------+
|                                                                  |
|  Фаза 1: PROMPT PROCESSING (prefill)                            |
|                                                                  |
|  Вход: все токены промпта [N токенов]                           |
|  Обработка: параллельная (все токены за один проход)            |
|  Результат: KV-cache для всех N токенов                        |
|  Bottleneck: compute (FLOPS)                                   |
|  Время: пропорционально N                                      |
|                                                                  |
+------------------------------------------------------------------+
                              |
                              v
+------------------------------------------------------------------+
|                                                                  |
|  Фаза 2: TOKEN GENERATION (decode)                              |
|                                                                  |
|  Вход: один новый токен + KV-cache                              |
|  Обработка: последовательная (токен за токеном)                 |
|  Результат: logits для следующего токена                        |
|  Bottleneck: memory bandwidth                                   |
|  Время: пропорционально количеству генерируемых токенов         |
|                                                                  |
+------------------------------------------------------------------+
```

Визуально на временной шкале:

```
Время -->

|<--- prefill --->|<-------- decode (токен за токеном) -------->|
|                 |                                              |
| обработка       | t1  t2  t3  t4  t5  t6  ... tM              |
| всего промпта   |                                              |
| за один проход  | каждый токен = один проход через модель      |
|                 |                                              |
| ~0.5-5 сек      | ~0.05-0.2 сек на токен                      |
| (зависит от N)  | (зависит от модели)                         |
```


## Prompt Processing (prefill)

### Что происходит

Весь промпт (системное сообщение + пользовательский запрос +
контекст) обрабатывается за один проход через все слои Transformer.

```
Промпт: [tok_1] [tok_2] [tok_3] ... [tok_N]

               +----------------------------------+
               |     Transformer (все слои)       |
[tok_1..N] --> |                                  | --> KV-cache
               |  Все N токенов обрабатываются    |     + logits
               |  параллельно на GPU              |     для tok_{N+1}
               +----------------------------------+
```

### Compute-bound

Prefill ограничен вычислительной мощностью GPU (FLOPS),
а не пропускной способностью памяти.

Причина: на каждый токен промпта выполняется полный forward pass
через все слои, но данные (веса) читаются из памяти один раз
и переиспользуются для всех N токенов.

```
Вычисления prefill:
  Для каждого слоя:
    Attention: 4 * N * d_model^2 + 2 * N^2 * d_model  FLOPS
    FFN:       3 * N * d_model * d_ff                   FLOPS

  Всего: ~ 2 * N * params  FLOPS (упрощённо)

Для Llama 3.1 8B, N=4096:
  ~2 * 4096 * 8B = ~65 TFLOPS
```

### Время prefill

Время зависит от длины промпта и вычислительной мощности GPU:

```
                      N (длина промпта)
t_prefill = --------------------------------
            GPU_TFLOPS * utilization / (2 * params)
```

Пример для Radeon 8060S:

| Промпт (токенов) | Llama 3.1 8B | Qwen2.5 32B | Llama 3.1 70B |
|-------------------|-------------|-------------|---------------|
| 512 | ~0.1 с | ~0.4 с | ~0.9 с |
| 2048 | ~0.4 с | ~1.6 с | ~3.5 с |
| 8192 | ~1.5 с | ~6 с | ~14 с |
| 32768 | ~6 с | ~24 с | ~55 с |
| 131072 | ~25 с | ~100 с | ~220 с |

[!] Значения приблизительные. Реальная скорость зависит от
реализации (Vulkan shaders, Flash Attention), утилизации GPU
и длины контекста (attention O(N^2)).


### Prompt caching

Если пользователь отправляет несколько запросов с одинаковым
системным промптом, KV-cache для системного промпта можно
переиспользовать.

```
Запрос 1: [system_prompt] + [user_query_1]
  --> prefill всего промпта

Запрос 2: [system_prompt] + [user_query_2]
  --> KV-cache для system_prompt уже есть
  --> prefill только user_query_2

Экономия: system_prompt может быть 1000+ токенов
```

В llama-server prompt caching реализован автоматически
в рамках одного слота (--parallel N для нескольких слотов).


## Token Generation (decode)

### Что происходит

После prefill модель генерирует токены по одному. На каждом
шаге через модель проходит только один новый токен.

```
Шаг decode:

  Новый токен: [tok_{N+k}]
  KV-cache: ключи и значения для [tok_1 ... tok_{N+k-1}]

  +----------------------------------+
  |     Transformer (все слои)       |
  |                                  |
  | [tok_{N+k}] + KV-cache --------> | logits для tok_{N+k+1}
  |                                  |
  | Q: от нового токена              |
  | K,V: из кэша + новый            |
  +----------------------------------+
  |
  +--> обновить KV-cache (добавить K,V нового токена)
```

### Один токен за шаг

Каждый шаг decode:

1. Embedding нового токена -> вектор [1, d_model]
2. Для каждого слоя:
   - Вычислить Q из нового токена: [1, d_model] * W_Q
   - Прочитать K, V из кэша: [seq_len, d_head]
   - Вычислить attention: Q * K^T (один запрос ко всем ключам)
   - FFN: обработать результат
3. LM Head: [1, d_model] * W_lm -> logits [vocab_size]
4. Сэмплирование -> один token ID

```
Вычисления за один decode-шаг:
  Attention: 4 * d_model^2 + 2 * seq_len * d_model  FLOPS
  FFN:       3 * d_model * d_ff                       FLOPS

  Всего: ~ 2 * params  FLOPS (упрощённо, для одного токена)
```

[!] На каждый токен выполняется ~2*params FLOPS вычислений,
но для этого нужно прочитать ВСЕ веса модели из памяти.


## Почему decode -- memory-bound

### Arithmetic Intensity

Arithmetic Intensity (AI) -- отношение вычислений к объёму
чтения данных из памяти:

```
                    FLOPS
AI = -----------------------------------
     bytes прочитанных из памяти
```

Для decode-шага одного токена:

```
FLOPS: ~2 * params (вычисления)
Bytes: ~params * bytes_per_param (чтение всех весов)

              2 * params
AI = ----------------------------- = 2 / bytes_per_param
     params * bytes_per_param

Для Q4_K_M (~0.56 bytes/param):
  AI = 2 / 0.56 = 3.6 FLOPS/byte

Для FP16 (2 bytes/param):
  AI = 2 / 2 = 1.0 FLOPS/byte
```

### Сравнение с возможностями GPU

```
Radeon 8060S:
  Compute:  ~30 TFLOPS (FP16)
  Bandwidth: 256 GB/s

  Compute/Bandwidth ratio = 30000 / 256 = ~117 FLOPS/byte
```

Чтобы GPU работал на полную мощность, нужен AI >= 117.
Decode даёт AI = 1-4. Это означает: GPU простаивает на
~97% вычислительной мощности, ожидая данные из памяти.

```
+---------------------------------------------------------+
|  GPU Utilization при decode (одиночный запрос)           |
|                                                         |
|  Compute:   [##.......................................] |
|              ~3%                                        |
|                                                         |
|  Bandwidth:  [########################################] |
|              ~100%                                      |
|                                                         |
|  Bottleneck: BANDWIDTH (memory-bound)                   |
+---------------------------------------------------------+
```

### Визуализация bottleneck

```
                                       "Крыша" (Roofline model)
Производительность                     /
(FLOPS)           ...................../
                  .                  /
                  .     compute    /
                  .     bound    /
                  .            /
                  .          /    <-- переход
                  .        /
                  .      /
                  .    /  <-- memory-bound зона
                  .  /         (decode живёт здесь)
                  ./
                  +----------------------------------
                  AI (FLOPS/byte) -->

  Decode AI = 1-4          Точка перехода = 117
  (далеко слева)           (Radeon 8060S)
```


## Формула скорости генерации

### Теоретический максимум

Для memory-bound операций скорость определяется bandwidth:

```
                   bandwidth (GB/s)
tok/s = ------------------------------------
        model_size_in_memory (GB)

Radeon 8060S: bandwidth = 256 GB/s
```

### Расчёт для разных моделей

| Модель | Квантизация | Размер (GB) | tok/s (теория) | tok/s (практика) |
|--------|------------|-------------|----------------|------------------|
| Llama 3.2 1B | Q4_K_M | 0.8 | 320 | 150-200 |
| Llama 3.2 3B | Q4_K_M | 2.0 | 128 | 80-100 |
| Llama 3.1 8B | Q4_K_M | 4.9 | 52 | 35-45 |
| Llama 3.1 8B | Q8_0 | 8.5 | 30 | 20-28 |
| Llama 3.1 8B | FP16 | 16.0 | 16 | 12-15 |
| Qwen2.5 32B | Q4_K_M | 19.8 | 13 | 8-12 |
| Llama 3.1 70B | Q4_K_M | 40.8 | 6.3 | 4-6 |
| Llama 3.1 70B | Q3_K_M | 33.2 | 7.7 | 5-7 |

### Почему практика < теории

Несколько факторов снижают реальную скорость:

```
Теоретический tok/s: bandwidth / model_size

Потери:
1. KV-cache чтение:     -5..15%  (доп. чтение из памяти)
2. Overhead вычислений:  -5..10%  (softmax, RMSNorm, embedding)
3. Kernel launch:        -3..5%   (накладные расходы Vulkan)
4. Memory alignment:     -2..5%   (невыровненные чтения)
5. Tensor parallelism:   -5..10%  (если используется)

Итого: реальная скорость = 60-80% от теоретической
```

### Зависимость от квантизации

```
tok/s (приблизительно, Llama 3.1 8B, Radeon 8060S)

  FP16:     |==========|                          12-15 tok/s
  Q8_0:     |====================|                 20-28 tok/s
  Q6_K:     |========================|             25-33 tok/s
  Q5_K_M:   |============================|         30-38 tok/s
  Q4_K_M:   |================================|     35-45 tok/s
  Q3_K_M:   |=====================================| 40-50 tok/s
  Q2_K:     |========================================| 45-55 tok/s
                                                      ^
                                                      |
                                            качество падает
```

[!] Меньше квантизация -> быстрее генерация, но хуже качество.
Q4_K_M -- стандартный компромисс. Q3 и ниже -- заметная деградация.


## KV-cache

KV-cache (Key-Value cache) -- хранилище промежуточных результатов
attention для уже обработанных токенов. Без KV-cache каждый
decode-шаг потребовал бы полного пересчёта attention для всех
предыдущих токенов.

### Зачем нужен KV-cache

Без кэша (наивный подход):
```
Шаг 1: обработать [tok_1, tok_2, tok_3]
Шаг 2: обработать [tok_1, tok_2, tok_3, tok_4]      <-- повтор
Шаг 3: обработать [tok_1, tok_2, tok_3, tok_4, tok_5] <-- повтор
```

С KV-cache:
```
Prefill: обработать [tok_1, tok_2, tok_3] --> сохранить K,V в кэш
Шаг 1:  обработать [tok_4], K,V из кэша   --> дополнить кэш
Шаг 2:  обработать [tok_5], K,V из кэша   --> дополнить кэш
```

Экономия: без кэша каждый шаг O(n^2), с кэшем -- O(n) на шаг.

### Как работает KV-cache

На каждом слое Transformer, для каждого decode-шага:

```
1. Новый токен tok_k:
   Q_new = tok_k * W_Q    [1, d_k]      -- вычисляем Q
   K_new = tok_k * W_K    [1, d_k]      -- вычисляем K
   V_new = tok_k * W_V    [1, d_v]      -- вычисляем V

2. Добавляем K_new и V_new в кэш:
   K_cache = [K_1, K_2, ..., K_{k-1}, K_new]   [k, d_k]
   V_cache = [V_1, V_2, ..., V_{k-1}, V_new]   [k, d_v]

3. Attention:
   scores = Q_new * K_cache^T     [1, k]
   weights = softmax(scores / sqrt(d_k))
   output = weights * V_cache     [1, d_v]
```

### Размер KV-cache

KV-cache растёт линейно с длиной контекста:

```
KV-cache size (bytes) =
  n_layers * n_kv_heads * head_dim * 2 * context_len * bytes_per_element
                                    ^
                            (K и V -- два тензора)
```

Подробная таблица: [context-window.md](context-window.md).

Примеры для FP16 (2 bytes/element):

| Модель | Контекст 4K | 8K | 32K | 128K |
|--------|-----------|------|------|-------|
| Llama 3.1 8B (8 KV-heads) | 0.25 GB | 0.5 GB | 2.0 GB | 8.0 GB |
| Qwen2.5 32B (8 KV-heads) | 0.5 GB | 1.0 GB | 4.0 GB | 16.0 GB |
| Llama 3.1 70B (8 KV-heads) | 0.6 GB | 1.3 GB | 5.0 GB | 20.0 GB |

[!] KV-cache занимает VRAM помимо весов модели.
Для Llama 3.1 70B Q4 (~40 GB) + KV-cache 32K (~5 GB) = ~45 GB.
На Radeon 8060S (96 GiB) остаётся ~50 GB запаса.


### Влияние KV-cache на скорость

KV-cache добавляет чтение из памяти на каждом decode-шаге:

```
Без KV-cache (невозможный сценарий -- для сравнения):
  Чтение: только веса модели

С KV-cache:
  Чтение: веса модели + KV-cache

Добавочные чтения:
  bytes_kv = 2 * n_layers * n_kv_heads * head_dim * seq_len * 2
  (2 за K и V, 2 за bytes/element для FP16)
```

Для Llama 3.1 8B, контекст 32K:
```
Чтение весов:     4.9 GB (Q4_K_M)
Чтение KV-cache: 2.0 GB
Итого:           6.9 GB

tok/s = 256 / 6.9 = ~37 (вместо 52 без KV)
Деградация: ~30%
```

Для длинных контекстов KV-cache может стать доминирующим
фактором замедления:

```
Llama 3.1 8B Q4_K_M, Radeon 8060S (256 GB/s):

Контекст | Веса  | KV     | Итого | tok/s
---------|-------|--------|-------|------
    1K   | 4.9   | 0.06   | 5.0   | ~51
    4K   | 4.9   | 0.25   | 5.2   | ~49
    8K   | 4.9   | 0.50   | 5.4   | ~47
   32K   | 4.9   | 2.00   | 6.9   | ~37
  128K   | 4.9   | 8.00   | 12.9  | ~20
```

[!] На длинных контекстах чтение KV-cache может занимать
больше bandwidth, чем чтение весов модели. Подробнее о
KV-cache и VRAM: [context-window.md](context-window.md).


### KV-cache quantization

Для экономии VRAM и bandwidth KV-cache тоже можно квантизировать:

```bash
# llama-server: квантизация KV-cache
llama-server \
    --model model.gguf \
    --ctx-size 32768 \
    --cache-type-k q8_0 \   # квантизация ключей
    --cache-type-v q8_0      # квантизация значений
```

Эффект квантизации KV-cache:

| Формат | Bytes/element | KV-cache 32K (8B) | Потеря качества |
|--------|--------------|-------------------|-----------------|
| FP16 | 2.0 | 2.0 GB | Нет |
| Q8_0 | 1.0 | 1.0 GB | Минимальная |
| Q4_0 | 0.5 | 0.5 GB | Заметная |

[!] Квантизация KV-cache до Q8_0 -- хороший компромисс.
Q4 для KV-cache снижает качество заметнее, чем Q4 для весов.


## Speculative Decoding

Speculative decoding -- техника ускорения генерации за счёт
параллельного предсказания нескольких токенов с помощью
маленькой "draft" модели.

### Идея

```
Обычная генерация (Llama 3.1 70B):
  Шаг 1: --> [tok_1]      (0.15 с)
  Шаг 2: --> [tok_2]      (0.15 с)
  Шаг 3: --> [tok_3]      (0.15 с)
  Шаг 4: --> [tok_4]      (0.15 с)
  Итого: 4 токена за 0.6 с

Speculative decoding:
  1. Draft model (1B) генерирует 4 токена: [d1, d2, d3, d4]  (0.01 с)
  2. Target model (70B) проверяет все 4 за ОДИН проход          (0.15 с)
  3. Если d1, d2 совпали, d3 не совпал:
     Принимаем d1, d2, корректируем d3
  Итого: 3 токена за 0.16 с (вместо 0.45 с)
```

### Как это работает

```
Draft model (быстрая, маленькая):
  Генерирует K кандидатов: [d_1, d_2, ..., d_K]

Target model (основная, большая):
  Проверяет кандидатов за один forward pass
  (эквивалентно prefill для K токенов)

Проверка:
  Для каждого d_i:
    - target_prob = P_target(d_i | d_1..d_{i-1})
    - draft_prob  = P_draft(d_i | d_1..d_{i-1})
    - Если target_prob / draft_prob >= rand():  принять
    - Иначе: отклонить d_i и все последующие,
      сэмплировать из скорректированного распределения
```

### Эффективность

Ускорение зависит от "acceptance rate" -- доли совпадений
между draft и target моделью:

| Acceptance rate | Ускорение | Типичные задачи |
|-----------------|-----------|-----------------|
| 90%+ | 3-4x | Код (предсказуемый), перевод |
| 70-90% | 2-3x | Техническое письмо |
| 50-70% | 1.5-2x | Creative writing |
| <50% | ~1x (без ускорения) | Очень творческие задачи |

```bash
# llama-server с speculative decoding
llama-server \
    --model Llama-3.1-70B-Q4_K_M.gguf \
    --model-draft Llama-3.2-1B-Q8_0.gguf \
    --draft-max 8 \        # максимум 8 draft-токенов за шаг
    --draft-min 2 \        # минимум 2 draft-токена
    --gpu-layers 99 \
    --gpu-layers-draft 99
```

[!] Speculative decoding не меняет результат генерации --
это математически доказуемая эквивалентность. Ускоряется
только процесс, не меняется качество.


## Continuous Batching

При обслуживании нескольких пользователей llama-server
использует continuous batching -- динамическое объединение
запросов.

### Проблема naive batching

```
Naive batching (статический):
  Запрос A: [prefill] [d d d d d d d d d EOS]
  Запрос B: [prefill] [d d d d d d d d d d d d d d d d EOS]
  Запрос C: [prefill] [d d d d EOS]
                       ^                     ^
                       |                     |
                 Запрос C закончился,   Запрос A закончился,
                 но слот занят          но ждём B

  --> GPU простаивает, ожидая завершения самого длинного запроса
```

### Continuous batching

```
Continuous batching:
  Запрос A: [prefill] [d d d d d d d d d EOS]
  Запрос B: [prefill] [d d d d d d d d d d d d d d d d EOS]
  Запрос C: [prefill] [d d d d EOS]
  Запрос D:                    [prefill] [d d d d d d d EOS]
                               ^
                               |
                          C закончился, D начинается немедленно
```

Преимущества:
- GPU всегда загружен
- Новые запросы начинаются не дожидаясь завершения старых
- Лучше утилизация VRAM (KV-cache освобождается по мере
  завершения запросов)

```bash
# llama-server с несколькими слотами
llama-server \
    --model model.gguf \
    --parallel 4 \          # 4 параллельных запроса
    --ctx-size 8192 \       # контекст на каждый слот
    --cont-batching          # continuous batching
```

[!] `--parallel N` выделяет `N * ctx_size` VRAM для KV-cache.
Для 4 слотов по 8K: 4 * 8192 = 32768 токенов KV-cache.


## Метрики производительности

### Основные метрики

| Метрика | Обозначение | Описание |
|---------|-------------|----------|
| Tokens per second (decode) | tok/s, tg | Скорость генерации |
| Tokens per second (prefill) | pp tok/s | Скорость обработки промпта |
| Time to First Token | TTFT | Задержка до первого токена ответа |
| Total Generation Time | TGT | Полное время генерации ответа |
| Throughput | req/s | Запросов в секунду (для сервера) |

### Формулы

```
TTFT = t_prefill = f(prompt_length, model_size, GPU_compute)

tg (tok/s) = bandwidth / (model_size + kv_cache_size)

TGT = TTFT + output_tokens / tg

Throughput = parallel_slots * tg / avg_output_tokens
```

### Пример полного расчёта

Задача: генерация ответа на запрос из 2048 токенов,
ответ ~500 токенов, Llama 3.1 8B Q4_K_M, Radeon 8060S.

```
Параметры:
  Prompt: 2048 токенов
  Output: ~500 токенов
  Model size: 4.9 GB
  Bandwidth: 256 GB/s

TTFT (prefill):
  ~0.4 с (обработка 2048 токенов)

tg (decode):
  KV-cache для ~2500 токенов (prompt + часть output): ~0.16 GB
  tok/s = 256 / (4.9 + 0.16) = ~50 tok/s

Время decode:
  500 / 50 = ~10 с

TGT (полное):
  0.4 + 10 = ~10.4 с

Скорость "видимая" пользователем:
  500 / 10.4 = ~48 tok/s
```


### Профилирование в llama-server

llama-server выводит метрики в лог:

```
llama_print_timings:        load time =   1234.56 ms
llama_print_timings:      sample time =     12.34 ms / 100 runs
llama_print_timings: prompt eval time =    456.78 ms / 2048 tokens (  0.22 ms per token, 4481.28 tokens per second)
llama_print_timings:        eval time =   2345.67 ms / 100 runs   ( 23.46 ms per token,   42.63 tokens per second)
llama_print_timings:       total time =   2814.79 ms / 2148 tokens

Расшифровка:
  load time       -- время загрузки модели
  sample time     -- время сэмплирования (незначительное)
  prompt eval     -- prefill (pp = 4481 tok/s для 2048 токенов)
  eval            -- decode (tg = 42.6 tok/s)
  total time      -- полное время
```


## Связь с платформой

### Radeon 8060S: bandwidth = 256 GB/s

Bandwidth -- главный определяющий фактор скорости генерации.

```
256 GB/s -- это:
  - 256 миллиардов байт в секунду
  - За 1 секунду можно прочитать:
    - Llama 3.1 8B Q4 (4.9 GB): ~52 раз --> 52 tok/s (теория)
    - Llama 3.1 70B Q4 (40 GB): ~6 раз  --> 6 tok/s (теория)
```

### Сравнение с другими GPU

| GPU | VRAM | Bandwidth | 8B Q4 tok/s | 70B Q4 tok/s |
|-----|------|-----------|-------------|--------------|
| RTX 4090 | 24 GB | 1008 GB/s | ~160 | - (не хватит VRAM) |
| RTX 4080 | 16 GB | 717 GB/s | ~110 | - |
| Radeon 8060S | 96 GB | 256 GB/s | ~42 | ~5 |
| Mac M4 Max | 128 GB | 546 GB/s | ~90 | ~11 |
| Mac M4 Ultra | 192 GB | 819 GB/s | ~130 | ~16 |
| A100 80GB | 80 GB | 2039 GB/s | ~330 | ~40 |
| H100 | 80 GB | 3352 GB/s | ~550 | ~67 |

[!] Radeon 8060S уступает по bandwidth, но выигрывает по VRAM.
96 GiB позволяет запустить 70B модель, что невозможно на RTX 4090.


### Рекомендации для разных задач

| Задача | Модель | Контекст | tok/s | Комфортность |
|--------|--------|----------|-------|-------------|
| Автодополнение кода | 3B Q4 | 4K | ~100 | Мгновенно |
| Чат (бытовой) | 8B Q4 | 8K | ~45 | Комфортно |
| Чат (сложный) | 32B Q4 | 16K | ~10 | Приемлемо |
| Анализ документа | 70B Q4 | 32K | ~5 | Медленно, но работает |
| Код (большой проект) | 32B Q4 | 32K | ~9 | Приемлемо |
| Максимальное качество | 70B Q4 | 8K | ~6 | Терпимо |

### llama-server: ключевые параметры

```bash
llama-server \
    --model model.gguf \
    --gpu-layers 99 \       # максимум слоёв на GPU (все)
    --ctx-size 8192 \       # контекстное окно
    --batch-size 512 \      # размер батча для prefill
    --ubatch-size 512 \     # размер микро-батча
    --parallel 1 \          # количество параллельных запросов
    --n-predict 2048 \      # максимум токенов в ответе
    --flash-attn \          # Flash Attention (если поддерживается)
    --port 8080
```

## Ключевые формулы (справочник)

```
Скорость decode:
  tok/s = bandwidth / (model_size + kv_cache_size)

Время prefill:
  t_prefill ~ 2 * N * params / GPU_FLOPS

KV-cache размер:
  kv_bytes = n_layers * n_kv_heads * head_dim * 2 * context * 2

TTFT:
  TTFT = t_prefill

Полное время:
  TGT = TTFT + n_output_tokens / tg

Arithmetic Intensity:
  AI = 2 / bytes_per_param  (для decode)
```
