## Анатомия имени модели

Возьмём типичное имя GGUF-файла:

```
Qwen3.5-122B-A10B-Q4_K_M-GGUF
└─┬──┘ └─┬──┘└─┬─┘ └──┬───┘└─┬──┘
  │      │     │      │      └─ формат файла
  │      │     │      └──────── тип квантизации
  │      │     └─────────────── Active параметры (только для MoE)
  │      └───────────────────── Total параметры
  └──────────────────────────── семейство и поколение
```

### Поколение: `Qwen3.5`

- Major: `Qwen3` -- архитектурное поколение
- Minor: `.5` -- ревизия с улучшениями (более качественные данные, новые техники)
- В Qwen-серии: 1 → 1.5 → 2 → 2.5 → 3 → 3.5 → 3.6 (пока API-only)

### Total параметры: `122B`

**Общее количество весов в модели**. В `122B` -- 122 миллиарда параметров.

Это **верхний предел знаний** модели -- сколько фактов и паттернов она может выучить. Чем больше total -- тем умнее модель в среднем (при равном качестве данных).

Также total = размер файла на диске. 122B fp16 ≈ 244 GB, Q4_K_M ≈ 71 GB.

### Active параметры: `A10B`

**Только для MoE**. Сколько параметров активны на ОДИН токен генерации.

`A10B` -- 10 миллиардов активных. Из 122B всего активны 10B одновременно.

Это **скорость** -- модель работает примерно как dense 10B по скорости генерации, потому что на каждый токен нужно прочитать только 10B весов из VRAM (а не все 122B).

**Если в имени нет `Axx` -- модель dense**, активны все параметры.

### Тип квантизации: `Q4_K_M`

Сжатие весов для уменьшения размера. Подробнее -- в [quantization.md](quantization.md).

Краткая шпаргалка:
- `F16` / `BF16` -- оригинал, без сжатия
- `Q8_0` -- лёгкое сжатие, ~95% качества F16
- `Q5_K_M` -- ~92% качества
- `Q4_K_M` -- ~88% качества (рекомендуется как баланс)
- `Q3_K_M` -- ~80% качества (для слабых GPU)
- `Q2_K` -- ~65% качества (для CPU/edge)
- `IQ4_XS`, `IQ3_M` -- умные квантизации (importance-weighted)

Размер файла зависит от квантизации:
- F16: 100% от total params × 2 байта
- Q8_0: ~50%
- Q4_K_M: ~25%
- Q2_K: ~12%

### Формат: `GGUF`

Quantized format для llama.cpp. Альтернативы:
- `safetensors` -- оригинал PyTorch (для ComfyUI, vLLM, transformers)
- `AWQ`, `GPTQ` -- альтернативные квантизации (не для llama.cpp)

## Active vs Total параметры -- ключевой вопрос

Это самое важное для выбора между Dense и MoE.

### Аналогия

Представь библиотеку:
- **Total params = размер библиотеки** (сколько книг есть)
- **Active params = сколько книг можно открыть одновременно**

**Dense модель** = маленькая комната, где все книги в открытом виде на столе. Вся библиотека = всё что доступно.

**MoE модель** = огромная библиотека, где открыть можно только N полок одновременно. Если вопрос про физику -- открываются полки про физику. Про историю -- про историю. Большая библиотека (total), но за одно "обращение" читаются только N полок (active).

### Что влияет на что

| Параметр | Влияет на |
|----------|-----------|
| **Total** | Знания, "эрудиция", качество ответов на сложных задачах, размер на диске, размер в VRAM |
| **Active** | Скорость генерации (tok/s), вычислительная нагрузка, "интеллект на токен" |

### Скорость прямо пропорциональна Active

Memory bandwidth определяет скорость: чем меньше параметров читать на токен, тем быстрее генерация.

| Модель | Active | Bandwidth read | tg tok/s (платформа) |
|--------|--------|----------------|----------------------|
| Llama-3.3 70B (dense) | 70B | весь модель | ~5 |
| Qwen3.5-122B-A10B (MoE) | 10B | ~10B на токен | **22** |
| Qwen3-Coder Next 80B-A3B | 3B | ~3B на токен | **53** |
| Qwen3-Coder 30B-A3B | 3B | ~3B на токен | **86** |

При том что 122B-A10B (122B total) больше Llama 70B по знаниям, она генерирует **в 4 раза быстрее**.

### Качество прямо пропорционально Total (приблизительно)

Total params -- верхняя граница "интеллекта" модели. MoE 122B-A10B на сложных задачах часто умнее dense 70B, потому что в ней больше выученных паттернов.

Но **на простых задачах** разница между active 3B и active 10B не критична -- 3B хватает чтобы "переварить" вопрос.

## Разбор Qwen3.5-серии

Три варианта на нашей платформе. Все мультимодальные (vision из коробки).

### Qwen3.5-27B (dense)

**Тип**: dense, 27B параметров активны на каждый токен

```
Total = Active = 27B
VRAM Q4_K_M ≈ 17 GiB
Скорость на платформе: 12.6 tok/s
```

**Что происходит при генерации**: на каждый токен модель читает все 27B параметров из VRAM. Скорость = 256 GB/s ÷ 17 GB ≈ 15 tok/s теория, реально 12.6 (77% bandwidth).

**Когда брать**:
- Хочется **универсальную дешёвую** модель
- Помещается в любой запас VRAM
- Контекст 128K
- Не нужен максимум интеллекта, нужна стабильная "хорошая" модель

**Сценарий**: фоновая модель для постоянных задач (русский чат, суммаризация, перевод), где скорость не критична, но и качества "27B уровня" хватает.

### Qwen3.5-35B-A3B (MoE)

**Тип**: MoE, 35B total, **3B active**

```
Total = 35B (в VRAM)
Active = 3B (читается на токен)
VRAM Q4_K_M ≈ 22 GiB
Скорость: ~80-90 tok/s (теория)
```

**Что происходит при генерации**: модель занимает в VRAM 22 GiB (все 35B надо где-то держать). Но на каждый токен router выбирает только маленькую часть экспертов общим размером ~3B и читает её.

Скорость = 256 GB/s ÷ 3 GB ≈ 85 tok/s.

**Когда брать**:
- Нужна **высокая скорость отклика** (3B active = 80+ tok/s)
- VRAM позволяет (22 GB на платформе -- легко)
- Нужны знания **уровня выше** чем dense 7B (которые дают такую же скорость)
- Multimodal (vision встроен)

**Сценарий**: интерактивный чат для работы. 80+ tok/s -- ощущается как мгновенный отклик. Знания на уровне 30B-моделей.

### Qwen3.5-122B-A10B (MoE)

**Тип**: MoE, 122B total, **10B active**

```
Total = 122B (в VRAM)
Active = 10B (читается на токен)
VRAM Q4_K_M ≈ 71 GiB
Скорость: 22.2 tok/s (замер)
```

**Что происходит**: модель занимает огромные 71 GiB VRAM, но на токен читается только 10B. Скорость = 256 ÷ 10 = 25.6 tok/s теория, реально 22.2 (87% bandwidth -- хорошо).

**Когда брать**:
- Нужно **максимум знаний/качества** среди того что помещается на платформе
- Сложные задачи (анализ, reasoning, длинные тексты)
- VRAM есть (71 GB занято, 49 GB остаётся на контекст и параллельные модели)
- Готов терпеть "только" 22 tok/s

**Сценарий**: флагман для важных запросов. Когда хочется максимум качества и нет требования "стрелять очередью".

## Decision tree: какой вариант выбрать

```
Что важнее всего?
│
├── СКОРОСТЬ (>50 tok/s)
│   └── Qwen3.5-35B-A3B (или Qwen3-Coder 30B-A3B для кода)
│
├── КАЧЕСТВО (не торопимся)
│   └── Qwen3.5-122B-A10B
│
└── БАЛАНС / экономия VRAM (для параллельных моделей)
    └── Qwen3.5-27B
```

### Расширенный decision tree

```
Сколько VRAM могу выделить под LLM?
│
├── < 25 GiB  → Qwen3.5-27B (17 GB) или Qwen3.5-35B-A3B (22 GB)
│              если приоритет скорость → 35B-A3B
│              если экономия → 27B
│
├── 25-75 GiB → выбор по приоритету:
│              скорость + хорошее качество → Qwen3-Coder Next 80B-A3B (45 GB) -- если кодинг
│              качество > скорости → Qwen3.5-122B-A10B (71 GB)
│
└── > 75 GiB  → Qwen3.5-122B-A10B + Coder 1.5B FIM параллельно (~73 GB)
              или Llama-3.3-70B Q8 (74 GB) если важен английский
```

## VRAM-математика

### Куда уходит VRAM при загрузке модели

```
[Веса модели]      ← Q4_K_M размер total params (всегда в VRAM)
[KV-cache]         ← пропорционально контексту × кол-ву слоёв × n_kv_heads
[Activations]      ← временные буферы в каждом forward pass
[Overhead]         ← framework + CUDA/Vulkan runtime
─────────────────
Итого
```

### Расчёт для Qwen3.5-122B-A10B

| Компонент | Размер |
|-----------|--------|
| Веса Q4_K_M | 71 GiB |
| KV-cache (32K контекст) | ~6 GiB |
| Activations + overhead | ~3 GiB |
| **Итого с 32K контекстом** | **~80 GiB** |

Запас на платформе: 120 - 80 = **40 GiB** -- достаточно на параллельный FIM-сервер или больший контекст.

### Расчёт для Qwen3.5-27B (dense)

| Компонент | Размер |
|-----------|--------|
| Веса Q4_K_M | 17 GiB |
| KV-cache (32K контекст) | ~3 GiB |
| Activations + overhead | ~1 GiB |
| **Итого** | **~21 GiB** |

Запас: 120 - 21 = **99 GiB** -- можно держать ещё 2-3 модели одновременно.

### MoE парадокс: больше не значит хуже по скорости

| Модель | Размер веса | Active | tok/s |
|--------|-------------|--------|-------|
| Llama-3.3 70B Q4 (dense) | 42 GiB | 70B | ~5 |
| Qwen3.5-122B-A10B Q4 (MoE) | **71 GiB** | 10B | **22** |

Хотя 122B-A10B **больше по диску в 1.7 раза**, она генерирует **в 4 раза быстрее**, потому что на токен читается 10B вместо 70B.

Это и есть **главное преимущество MoE на платформе с большим VRAM**: можно держать "большую модель" и получать "скорость маленькой".

## Практические сценарии

### Сценарий 1: opencode + быстрый chat

**Хочу**: opencode для агентного кодинга + быстрый chat в Open WebUI.

**Решение**:
- Порт 8081: [Qwen3-Coder Next 80B-A3B](../models/families/qwen3-coder.md#next-80b-a3b) (45 GB) -- для opencode
- Порт 8080: Qwen3.5-35B-A3B (22 GB) -- для quick chat

Итого: **67 GB занято**, остаётся 53 GB на контексты и эксперименты.

### Сценарий 2: максимум знаний + русский

**Хочу**: лучшее качество русскоязычных ответов на сложные вопросы.

**Решение**:
- Один сервер: [Qwen3.5-122B-A10B](../models/families/qwen35.md#122b-a10b) (71 GB)
- Скорость 22 tok/s -- терпимо для "research-style" чата

### Сценарий 3: интерактивный assistant с картинками

**Хочу**: понимание скриншотов + быстрый отклик.

**Решение**: [Gemma 4 26B-A4B](../models/families/gemma4.md) (17 GB + 1.2 mmproj)
- 3.8B active = быстрый отклик
- Native vision через mmproj
- Function calling
- 256K контекст

### Сценарий 4: экономия VRAM (несколько моделей)

**Хочу**: 4 разные модели одновременно.

**Решение**: брать dense варианты поменьше:
- Qwen3.5-27B (17 GB) -- основной chat
- Qwen2.5-Coder 1.5B (2 GB) -- FIM
- ACE-Step (4 GB) -- music
- ComfyUI + FLUX schnell (7 GB) -- картинки

Итого ~30 GB, остаётся 90 GB на контексты и эксперименты.

## Что брать в первую очередь на платформу

Если бы пришлось выбрать **только три** Qwen3.5-варианта:

1. **[Qwen3.5-35B-A3B](../models/families/qwen35.md#35b-a3b)** -- основной workhorse, 80+ tok/s, multimodal
2. **[Qwen3.5-122B-A10B](../models/families/qwen35.md#122b-a10b)** -- флагман для важных запросов, 22 tok/s
3. **[Qwen3.5-27B](../models/families/qwen35.md#27b)** -- запасной dense для параллельной работы

На платформе сейчас скачаны **27B и 122B-A10B**. Третий приоритет -- скачать 35B-A3B для быстрого MoE-варианта в multimodal-сценариях.
