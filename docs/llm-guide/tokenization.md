# Токенизация

Токенизация -- процесс преобразования текста в последовательность
числовых идентификаторов (token IDs), с которыми работает нейросеть.
Модель не оперирует буквами или словами напрямую -- только числами.

## Зачем нужна токенизация

Нейронная сеть принимает на вход тензоры (многомерные массивы чисел).
Текст нужно преобразовать в числа, прежде чем модель сможет с ним
работать.

```
Текст                    Токены               Token IDs        Embedding
"Hello world"   -->   ["Hello", " world"]  -->  [15339, 1917]  -->  [[0.12, -0.34, ...],
                                                                     [0.56,  0.78, ...]]
                                                                     ^
                                                                     |
                                                                 вектора [d_model]
```

Процесс:
1. **Токенизация**: текст -> последовательность token IDs
2. **Embedding**: token ID -> вектор размерности d_model
3. Дальше вектора проходят через слои Transformer
   ([transformer.md](transformer.md))

Обратный процесс (detokenization): token IDs -> текст.


### Почему не посимвольно

Можно было бы кодировать каждый символ отдельно (a=1, b=2, ...),
но это неэффективно:

| Подход | Словарь | "Hello world" | Длина |
|--------|---------|---------------|-------|
| Посимвольно | ~256 | [H,e,l,l,o, ,w,o,r,l,d] | 11 |
| Пословно | ~100K+ | [Hello, world] | 2 |
| BPE (подслова) | ~32K-128K | [Hello, _world] | 2 |

Посимвольная токенизация:
- Очень длинные последовательности (attention O(n^2))
- Модели нужно "выучить" составление букв в слова

Пословная токенизация:
- Огромный словарь
- Неизвестные слова (OOV -- out of vocabulary)
- Не обрабатывает опечатки, новые слова, код

BPE (Byte Pair Encoding) -- компромисс: подслова (subwords).
Частые слова -- один токен, редкие -- разбиваются на части.


## Словарь (vocabulary)

Словарь (vocabulary) -- фиксированный набор всех токенов, известных
модели. Каждый токен имеет уникальный числовой ID.

### Размеры словарей

| Модель | Размер словаря | Примечание |
|--------|---------------|------------|
| GPT-2 | 50257 | BPE, английский фокус |
| Llama 2 | 32000 | SentencePiece |
| Llama 3 | 128256 | tiktoken, мультиязычный |
| Qwen2.5 | 151936 | BPE, расширенный CJK |
| DeepSeek-V3 | 129280 | BPE |
| Mistral | 32768 | SentencePiece |
| Gemma 2 | 256000 | SentencePiece, огромный словарь |

### Что определяет размер словаря

Больший словарь:
- Больше целых слов и подслов -> короче последовательности
- Лучше покрытие разных языков
- Больше embedding-таблица (vocab_size * d_model параметров)
- Llama 3: 128256 * 4096 = 525M параметров только на embedding

Меньший словарь:
- Длиннее последовательности -> медленнее
- Хуже покрытие редких языков
- Меньше параметров на embedding

### Структура словаря

Типичный словарь BPE содержит:

```
ID     Токен              Тип
---    -----              ---
0      <|begin_of_text|>  специальный
1      <|end_of_text|>    специальный
...
256    "a"                базовый (один символ)
257    "b"                базовый
...
512    "th"               мерж (два символа)
513    "he"               мерж
514    "the"              мерж (три символа)
...
1000   "Hello"            мерж (целое слово)
...
50000  " implementation"  мерж (длинное подслово)
...
128255 "<|reserved|>"     зарезервированный
```

Первые 256 ID -- отдельные байты (обеспечивают кодирование
любого текста, даже неизвестного). Далее -- всё более длинные
подслова, полученные алгоритмом BPE.


## Алгоритм BPE

BPE (Byte Pair Encoding) -- основной алгоритм построения словаря
для LLM. Обучается на большом корпусе текстов.

### Принцип работы

Начинаем с отдельных символов (байтов) и итеративно объединяем
самые частые пары в новые токены.

### Пошаговый пример

Корпус для обучения:

```
"low" (5 раз)
"lower" (2 раза)
"newest" (6 раз)
"widest" (3 раза)
```

**Шаг 0**: начальный словарь -- отдельные символы:

```
Словарь: {l, o, w, e, r, n, s, t, i, d}

Представление корпуса (с частотами):
l o w       (5)
l o w e r   (2)
n e w e s t (6)
w i d e s t (3)
```

**Шаг 1**: находим самую частую пару символов:

```
Пары и их частоты:
  "e s" -> 6+3 = 9   <-- самая частая
  "l o" -> 5+2 = 7
  "o w" -> 5+2 = 7
  "s t" -> 6+3 = 9   <-- тоже 9 (берём одну из них)
  "e w" -> 6
  ...
```

Объединяем "es" в один токен:

```
Словарь: {l, o, w, e, r, n, s, t, i, d, es}

l o w           (5)
l o w e r       (2)
n e w es t      (6)
w i d es t      (3)
```

**Шаг 2**: следующая самая частая пара:

```
"es t" -> 6+3 = 9   <-- самая частая
```

Объединяем "est" в один токен:

```
Словарь: {l, o, w, e, r, n, s, t, i, d, es, est}

l o w           (5)
l o w e r       (2)
n e w est       (6)
w i d est       (3)
```

**Шаг 3**: "lo" -> 5+2 = 7:

```
Словарь: {..., lo}

lo w           (5)
lo w e r       (2)
n e w est      (6)
w i d est      (3)
```

**Шаг 4**: "low" -> 5+2 = 7:

```
Словарь: {..., low}

low            (5)
low e r        (2)
n e w est      (6)
w i d est      (3)
```

Процесс продолжается до достижения желаемого размера словаря
(32K, 128K).

### Формальный алгоритм

```
ВХОД: корпус текстов, желаемый размер словаря V

1. Инициализировать словарь всеми отдельными байтами (256 токенов)
2. Подсчитать частоту каждой пары соседних токенов в корпусе
3. Найти самую частую пару
4. Добавить новый токен (объединение пары) в словарь
5. Заменить все вхождения пары в корпусе на новый токен
6. Повторять 2-5 пока |словарь| < V
```

### Свойства BPE

- **Частые слова** становятся одним токеном: "the", "and", "is"
- **Редкие слова** разбиваются на подслова: "непредусмотренных" ->
  ["не", "предус", "мотрен", "ных"]
- **Любой текст** кодируем: базовые байты как fallback
- **Детерминированный**: одинаковый вход -> одинаковый выход
- **Без обучения модели**: словарь строится статистически,
  отдельно от обучения нейросети


### Byte-level BPE

Современные токенизаторы (tiktoken для Llama 3, Qwen) работают
на уровне байтов, а не символов:

```
Классический BPE: работает с Unicode-символами
  "Привет" -> символы: [П, р, и, в, е, т]

Byte-level BPE: работает с UTF-8 байтами
  "Привет" -> байты: [0xD0, 0x9F, 0xD1, 0x80, 0xD0, 0xB8,
                       0xD0, 0xB2, 0xD0, 0xB5, 0xD1, 0x82]
  -> 12 байтов вместо 6 символов
```

Преимущества byte-level:
- Гарантированно кодирует любой текст (включая бинарные данные)
- Начальный словарь всегда 256 (все возможные байты)
- Нет проблемы с неизвестными символами


## SentencePiece и другие токенизаторы

### SentencePiece

Токенизатор от Google, используется в Llama 2, Mistral, Gemma.
Отличия от стандартного BPE:

```
Стандартный BPE:           SentencePiece:
- Сначала пресегментация  - Работает с "сырым" текстом
  (разбиение по пробелам) - Пробелы как часть токенов
- Пробелы не кодируются   - Символ _ заменяет пробел
- Зависит от языка        - Языконезависимый
```

Пример:
```
Текст: "I like cats"

BPE (с пресегментацией):
  ["I", "like", "cats"]  -- пробелы отдельно

SentencePiece:
  ["_I", "_like", "_cats"]  -- пробелы включены в токены
  (_ = начало слова)
```

### tiktoken

Токенизатор от OpenAI, используется в GPT-4 и Llama 3.
Byte-level BPE с эффективной реализацией на Rust.

| Свойство | SentencePiece | tiktoken |
|----------|--------------|----------|
| Алгоритм | Unigram / BPE | Byte-level BPE |
| Язык реализации | C++ | Rust + Python |
| Пробелы | Символ _ | Включены в токены |
| Скорость | Средняя | Высокая |
| Используется в | Llama 2, Mistral, Gemma | GPT-4, Llama 3 |

### Hugging Face Tokenizers

Библиотека, поддерживающая несколько алгоритмов:
- BPE
- WordPiece (BERT)
- Unigram (SentencePiece)

Реализована на Rust с Python-биндингами, высокая
производительность.

### Английский текст

```
Текст: "Hello, world! How are you?"

Llama 3 tokenizer (tiktoken, vocab=128256):
  Токены: ["Hello", ",", " world", "!", " How", " are", " you", "?"]
  IDs:    [9906,     11,   1917,    0,    1724,   527,   498,    30]
  Длина:  8 токенов

Llama 2 tokenizer (SentencePiece, vocab=32000):
  Токены: ["Hello", ",", " world", "!", " How", " are", " you", "?"]
  IDs:    [15043,   29892, 3186,  29991, 1128,  526,    366,   29973]
  Длина:  8 токенов
```

### Русский текст

```
Текст: "Привет, мир! Как дела?"

Llama 3 tokenizer:
  Токены: ["При", "вет", ",", " мир", "!", " Как", " дела", "?"]
  IDs:    [95, 12448, 11, 29830, 0, 21735, 25312, 30]
  Длина:  8 токенов

Llama 2 tokenizer:
  Токены: ["<0xD0>", "<0x9F>", "ри", "вет", ",", ...] (побайтовая разбивка)
  Длина:  ~18 токенов (кириллица расщепляется на байты)
```

[!] Llama 2 плохо токенизирует кириллицу -- словарь слишком маленький
(32K). Llama 3 с словарём 128K значительно лучше, но всё ещё хуже,
чем для английского.

### Код (Python)

```
Текст: "def fibonacci(n):\n    if n <= 1:\n        return n"

Llama 3 tokenizer:
  Токены: ["def", " fib", "onacci", "(", "n", "):", "\n",
           "   ", " if", " n", " <=", " ", "1", ":", "\n",
           "       ", " return", " n"]
  Длина:  18 токенов
```

### Числа

```
Текст: "1234567890"

Llama 3: ["123", "456", "789", "0"]  -- 4 токена
  Числа разбиваются на группы по 3-4 цифры

Текст: "3.14159265"

Llama 3: ["3", ".", "141", "592", "65"]  -- 5 токенов
```

[!] LLM работают с числами как с текстом -- отсюда проблемы
с арифметикой. Модель не "видит" число 1234567890 как единое
значение, а обрабатывает 4 отдельных токена.

### JSON

```
Текст: '{"name": "test", "value": 42}'

Llama 3:
  Токены: ['{"', 'name', '":', ' "', 'test', '",', ' "', 'value', '":', ' ', '42', '}']
  Длина: 12 токенов
```

Структурированный текст (JSON, XML, YAML) требует много токенов
на разделители и форматирование.


## Разница языков

Токенизаторы обучены преимущественно на английском тексте.
Другие языки (особенно с не-латинскими алфавитами) кодируются
менее эффективно.

### Сравнение эффективности

Одинаковый по смыслу текст на разных языках:

```
Английский: "The context window determines the maximum number of tokens"
  Llama 3: 9 токенов
  ~0.78 слов/токен

Русский: "Контекстное окно определяет максимальное количество токенов"
  Llama 3: 13 токенов
  ~0.46 слов/токен

Китайский: "上下文窗口决定了令牌的最大数量"
  Llama 3: 14 токенов
  ~0.43 иероглифов/токен (но каждый иероглиф = слово)

Японский: "コンテキストウィンドウはトークンの最大数を決定します"
  Llama 3: 18 токенов
```

### Таблица эффективности по языкам

| Язык | Слов/токен (Llama 3) | Множитель к англ. | Причина |
|------|---------------------|-------------------|---------|
| Английский | ~0.75 | 1.0x | Основной язык обучения |
| Французский | ~0.65 | 1.15x | Латиница, много в корпусе |
| Немецкий | ~0.60 | 1.25x | Латиница, длинные слова |
| Русский | ~0.50 | 1.5x | Кириллица, флексии |
| Арабский | ~0.45 | 1.7x | Другой алфавит |
| Китайский | ~0.40 | 1.9x | Иероглифы, мало в ранних корпусах |
| Тайский | ~0.35 | 2.1x | Сложная письменность |
| Корейский | ~0.40 | 1.9x | Хангыль |

### Практические последствия

1. **Контекст заполняется быстрее**: русский текст в ~1.5 раза
   "тяжелее" английского по токенам. Контекст 32K для русского --
   это примерно 16K слов, а для английского -- 24K слов.

2. **Дороже по API**: провайдеры тарифицируют по токенам.
   Русский текст стоит в ~1.5 раза дороже английского.

3. **Медленнее генерация**: больше токенов для того же количества
   текста -> дольше генерация.

4. **Выбор модели**: модели с большим словарём (Qwen: 151K,
   Gemma: 256K) эффективнее для не-английских языков.

```
Пример: статья на 5000 слов

Английский:
  ~6700 токенов (Llama 3)
  Укладывается в контекст 8K

Русский:
  ~10000 токенов (Llama 3)
  Нужен контекст минимум 16K (с запасом для ответа)

Русский + Qwen2.5:
  ~8500 токенов (лучшая токенизация кириллицы)
  Почти укладывается в 8K
```


### Qwen vs Llama: токенизация русского

Qwen2.5 имеет словарь 151936 токенов с лучшим покрытием кириллицы:

```
Текст: "Модель генерирует текст токен за токеном"

Llama 3:
  ["Мод", "ель", " генер", "ирует", " текст", " ток", "ен",
   " за", " ток", "ен"]
  Длина: 10 токенов

Qwen2.5:
  ["Модель", " генерирует", " текст", " токен", " за", " токен"]
  Длина: 6 токенов
```

[!] Для работы с русскоязычными текстами Qwen2.5 и DeepSeek
предпочтительнее Llama по эффективности токенизации.


## Специальные токены

Помимо обычных текстовых токенов, словарь содержит специальные
токены -- служебные маркеры, управляющие поведением модели.

### Основные специальные токены

| Токен | Назначение | Пример |
|-------|-----------|--------|
| BOS (Begin of Sequence) | Начало последовательности | `<s>`, `<\|begin_of_text\|>` |
| EOS (End of Sequence) | Конец последовательности | `</s>`, `<\|end_of_text\|>` |
| PAD | Дополнение до одинаковой длины | `<pad>` |
| UNK | Неизвестный токен | `<unk>` |

### Chat-специфичные токены

Для моделей-ассистентов добавляются токены разделения ролей:

```
Llama 3 chat tokens:
  <|begin_of_text|>     -- начало диалога
  <|start_header_id|>   -- начало заголовка роли
  <|end_header_id|>     -- конец заголовка роли
  <|eot_id|>            -- конец хода (end of turn)
  <|end_of_text|>       -- конец всего текста

Qwen2.5 chat tokens:
  <|im_start|>          -- начало сообщения
  <|im_end|>            -- конец сообщения
```

### Как специальные токены используются

```
Обычный текст:
  "Hello world"

С обрамлением:
  <|begin_of_text|> Hello world <|end_of_text|>

В контексте чата (Llama 3):
  <|begin_of_text|>
  <|start_header_id|>system<|end_header_id|>
  You are a helpful assistant.
  <|eot_id|>
  <|start_header_id|>user<|end_header_id|>
  Hello!
  <|eot_id|>
  <|start_header_id|>assistant<|end_header_id|>
```

[!] Специальные токены не должны появляться в пользовательском тексте.
Если пользователь введёт `<|end_of_text|>` -- токенизатор должен
экранировать это как обычный текст, а не как спецтокен.


### FIM-токены (Fill-in-the-Middle)

Для автодополнения кода используются специальные токены:

```
<|fim_prefix|>   -- начало префикса (код до курсора)
<|fim_suffix|>   -- начало суффикса (код после курсора)
<|fim_middle|>   -- начало заполнения (то, что модель генерирует)

Пример:
  Код: def hello(name):
           print(f"Hello, {<CURSOR>}!")

  FIM-формат:
  <|fim_prefix|>def hello(name):
      print(f"Hello, {<|fim_suffix|>}!")
  <|fim_middle|>
  --> модель генерирует: name
```


## Chat Templates

Chat template -- формат, в котором диалог передаётся модели.
Каждое семейство моделей использует свой формат.

### Llama 3 Chat Template

```
<|begin_of_text|><|start_header_id|>system<|end_header_id|>

{system_prompt}<|eot_id|><|start_header_id|>user<|end_header_id|>

{user_message}<|eot_id|><|start_header_id|>assistant<|end_header_id|>

{assistant_response}<|eot_id|>
```

Конкретный пример:

```
<|begin_of_text|><|start_header_id|>system<|end_header_id|>

Ты -- полезный ассистент. Отвечай кратко.<|eot_id|><|start_header_id|>user<|end_header_id|>

Что такое токенизация?<|eot_id|><|start_header_id|>assistant<|end_header_id|>

```

### Qwen2.5 / ChatML Template

```
<|im_start|>system
{system_prompt}<|im_end|>
<|im_start|>user
{user_message}<|im_end|>
<|im_start|>assistant
{assistant_response}<|im_end|>
```

### Mistral Chat Template

```
[INST] {user_message} [/INST] {assistant_response}</s>
[INST] {user_message_2} [/INST]
```

### Зачем нужны разные шаблоны

- Модель обучена на конкретном шаблоне
- Неправильный шаблон -> деградация качества
- llama-server автоматически определяет шаблон из метаданных GGUF

[!] Не нужно вручную формировать шаблон при использовании
llama-server API (endpoint /v1/chat/completions). Сервер
применяет шаблон автоматически.


## Как проверить токенизацию

### Python: transformers

```python
# установка: pip install transformers
from transformers import AutoTokenizer

# загрузка токенизатора (без загрузки модели)
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-3.1-8B-Instruct")

# токенизация
text = "Привет, мир! Как дела?"
tokens = tokenizer.encode(text)
print(f"Token IDs: {tokens}")
print(f"Количество токенов: {len(tokens)}")

# декодирование обратно
decoded = tokenizer.decode(tokens)
print(f"Декодированный текст: {decoded}")

# просмотр отдельных токенов
for tid in tokens:
    print(f"  {tid:6d} -> '{tokenizer.decode([tid])}'")
```

Вывод:

```
Token IDs: [128000, 95, 12448, 11, 29830, 0, 21735, 25312, 30]
Количество токенов: 9
Декодированный текст: <|begin_of_text|>Привет, мир! Как дела?
  128000 -> '<|begin_of_text|>'
      95 -> 'При'
   12448 -> 'вет'
      11 -> ','
   29830 -> ' мир'
       0 -> '!'
   21735 -> ' Как'
   25312 -> ' дела'
      30 -> '?'
```

### Python: tiktoken

```python
# установка: pip install tiktoken
import tiktoken

# токенизатор GPT-4 / Llama 3
enc = tiktoken.get_encoding("cl100k_base")

text = "Hello, world!"
tokens = enc.encode(text)
print(f"Tokens: {tokens}")
print(f"Count: {len(tokens)}")

# декодирование
for t in tokens:
    print(f"  {t} -> '{enc.decode([t])}'")
```

### Командная строка: llama-tokenize

В составе llama.cpp есть утилита для токенизации:

```bash
# токенизация файла
echo "Hello, world!" | llama-tokenize -m model.gguf

# подсчёт токенов
echo "Длинный текст..." | llama-tokenize -m model.gguf | wc -l
```

## Практическое влияние

### На контекстное окно

Контекстное окно измеряется в токенах. Разный текст занимает
разное количество токенов:

```
Контекст 8192 токена -- сколько это текста:

Английская проза:     ~6000 слов  (~24 страницы A4)
Русская проза:        ~4000 слов  (~16 страниц A4)
Python-код:           ~2500 строк (зависит от стиля)
JSON:                 ~150 KB     (много разделителей)
Markdown:             ~5000 слов  (проза + разметка)
```

Подробнее: [context-window.md](context-window.md).


### На скорость inference

Больше токенов = дольше обработка:

| Фаза | Зависимость от количества токенов |
|------|----------------------------------|
| Prefill | Линейная (2x токенов -> 2x времени) |
| Decode | Без изменений (1 токен за шаг) |
| KV-cache | Линейный рост по VRAM |

Русский текст на ~50% длиннее в токенах -> prefill на ~50% дольше.

### На стоимость API

API-провайдеры тарифицируют по токенам:

```
GPT-4o (пример цен):
  Input:  $5 / 1M tokens
  Output: $15 / 1M tokens

Запрос на 1000 слов:
  Английский: ~1300 токенов -> $0.0065
  Русский:    ~2000 токенов -> $0.0100  (+54%)
```

### На выбор модели

Для русскоязычных задач предпочтительны модели с:
- Большим словарём (>100K)
- Хорошим покрытием кириллицы в данных обучения
- Qwen2.5 и DeepSeek превосходят Llama 3 по эффективности
  токенизации кириллицы


## Связь с платформой

### llama-server и токенизация

llama-server автоматически загружает токенизатор из GGUF-файла.
Специальная настройка не требуется.

```bash
# эндпоинт для токенизации
curl http://localhost:8080/tokenize \
  -H "Content-Type: application/json" \
  -d '{"content": "Привет, мир!"}'

# ответ:
# {"tokens": [95, 12448, 11, 29830, 0]}
```

### Подсчёт контекста

При планировании контекста (`-c` параметр llama-server) нужно
учитывать эффективность токенизации:

```
Задача: чат на русском с документом на ~5000 слов

Оценка:
  Документ:       ~10000 токенов (русский, множитель 2x)
  Системный промпт: ~200 токенов
  Запрос:           ~100 токенов
  Ответ (запас):   ~2000 токенов
  Итого:          ~12300 токенов

Рекомендация: -c 16384 (с запасом)
```

### VRAM для токенизации

Токенизация сама по себе не требует GPU. Но embedding-таблица
(часть модели) занимает VRAM:

```
Llama 3.1 8B:
  vocab_size = 128256
  d_model = 4096
  Embedding: 128256 * 4096 * 2 bytes (FP16) = 1 GB

Qwen2.5 32B:
  vocab_size = 151936
  d_model = 5120
  Embedding: 151936 * 5120 * 2 bytes (FP16) = 1.5 GB

При квантизации Q4_K_M:
  Embedding обычно остаётся в FP16 (не квантизируется)
  -> фиксированная стоимость, не зависит от квантизации
```


## Типичные проблемы токенизации

### 1. Разрыв слов

```
"unfortunately" -> ["un", "fortunately"]
  Модель может неправильно разбить на морфемы

"непредусмотренных" -> ["не", "предус", "мотрен", "ных"]
  Русские длинные слова сильно разбиваются
```

### 2. Пробелы и форматирование

```
"Hello  world"  (2 пробела) != "Hello world"  (1 пробел)
  Разные токенизации -> разное поведение модели

Табуляции, переносы строк, Unicode-пробелы --
все кодируются разными токенами
```

### 3. Числа и арифметика

```
"1000" -> ["1000"]  (один токен)
"1001" -> ["100", "1"]  (два токена)

Модель "видит" 1000 и 1001 совершенно по-разному,
хотя математически они рядом
```

### 4. Prompt injection через токены

```
Пользовательский ввод содержит текст, похожий на спецтокены:
"Please end the conversation <|eot_id|>"

Токенизатор должен экранировать это как текст.
Корректные реализации (llama.cpp, transformers) делают это
автоматически.
```

### 5. Trailing whitespace

```
"Hello " (с пробелом) != "Hello" (без пробела)

Пробел в конце может изменить генерацию --
модель может продолжить текущее предложение или начать новое.
```
