## Обзор подходов

Существует два принципиально разных способа использования LLM:

```
Облачный API:
  Ваш код --[интернет]--> Сервер провайдера --> Ответ
  Модель работает на оборудовании провайдера
  Оплата: за токены (вход + выход)

Локальный LLM:
  Ваш код --> Ваш сервер (GPU) --> Ответ
  Модель работает на вашем оборудовании
  Оплата: электричество + амортизация оборудования
```

Оба подхода имеют OpenAI-совместимый API, что позволяет переключаться
между ними без изменения кода.

### Плюсы облачных API

```
+ Максимальное качество (frontier-модели: GPT-4o, Gemini 2.5 Pro)
+ Нет необходимости в GPU
+ Мгновенный старт (зарегистрировался -> начал работать)
+ Масштабируемость (десятки/сотни запросов в секунду)
+ Постоянное обновление моделей
+ Поддержка всех модальностей (текст, изображения, аудио, видео)
+ Высокая доступность (99.9%+ SLA)
```

### Минусы облачных API

```
- Стоимость: $20-200/месяц для активного использования
- Приватность: данные покидают ваш контур
- Зависимость от интернета
- Зависимость от провайдера (pricing, лимиты, условия)
- Цензура и фильтрация контента
- Rate limits (ограничения запросов в минуту)
- Изменение моделей без предупреждения
```

## Локальный LLM

### Инструменты запуска

```
Инструмент    | Тип        | GPU Backend    | API             | Сложность
--------------|------------|----------------|-----------------|----------
llama.cpp     | Сервер     | CUDA/Vulkan/Metal| OpenAI-совм.  | Средняя
llama-server  | Сервер     | CUDA/Vulkan/Metal| OpenAI-совм.  | Средняя
Ollama        | Менеджер   | CUDA/Vulkan/Metal| Свой + OpenAI | Низкая
LM Studio     | GUI        | CUDA/Vulkan/Metal| OpenAI-совм.  | Низкая
vLLM          | Сервер     | CUDA/ROCm      | OpenAI-совм.    | Высокая
TGI           | Сервер     | CUDA           | OpenAI-совм.    | Высокая
```

Для данной платформы (Radeon 8060S, Vulkan) -- **llama-server** из llama.cpp
является основным инструментом.

### Плюсы локального LLM

```
+ Приватность: данные не покидают ваш сервер
+ Стоимость: $0 за inference (только электричество)
+ Нет зависимости от интернета
+ Полный контроль: выбор модели, параметров, квантизации
+ Нет цензуры (ваша модель -- ваши правила)
+ Нет rate limits
+ Предсказуемая задержка (нет сетевых задержек)
+ Кастомизация: fine-tuning, LoRA, промпт-инженерия без ограничений
```

### Минусы локального LLM

```
- Качество ниже frontier-моделей
- Необходимость GPU (инвестиции в оборудование)
- Настройка и обслуживание
- Ограниченный context window (зависит от VRAM)
- Обновление моделей -- ваша забота
- Один пользователь за раз (без доп. настроек)
```

## Когда локальный

### 1. Чувствительные данные

```
Сценарии:
  - Медицинские данные (HIPAA)
  - Финансовые документы
  - Персональные данные (GDPR)
  - Корпоративные секреты
  - Код с NDA

Риск API: данные проходят через серверы провайдера.
Даже с "data processing agreement" -- это дополнительный вектор утечки.

Локальный LLM: данные не покидают ваш контур. Нулевой риск утечки
через провайдера.
```

### 2. Отсутствие интернета

```
Сценарии:
  - Закрытые контуры (гос. организации, банки)
  - Полевая работа (без связи)
  - Воздушный зазор (air-gapped сети)
  - Нестабильный интернет

Локальный LLM работает полностью автономно.
```

### 3. Предсказуемая стоимость

```
Сценарий: ежедневная обработка 1000 документов

API (GPT-4o-mini):
  1000 документов * 2000 токенов input * 500 токенов output = 2.5M токенов/день
  Input: 2M * $0.15/1M = $0.30
  Output: 0.5M * $0.60/1M = $0.30
  Итого: $0.60/день = ~$18/месяц

API (GPT-4o):
  Input: 2M * $2.50/1M = $5.00
  Output: 0.5M * $10.00/1M = $5.00
  Итого: $10.00/день = ~$300/месяц

Локальный (Radeon 8060S):
  Электричество: ~200W * 8 часов/день * 30 дней * $0.07/кВтч = ~$3.4/месяц
  (Амортизация оборудования -- отдельный расчет)
```

### 4. Кастомизация и эксперименты

```
Сценарии:
  - Fine-tuning под конкретную задачу
  - Эксперименты с промптами без ограничений
  - Тестирование разных моделей и квантизаций
  - Специфические grammar/schema для structured output
  - Нестандартные chat template
```

### 5. Высокая частота запросов

```
Сценарий: 10 000+ запросов в день

API: стоимость растет линейно с количеством запросов.
  10K * 3K токенов = 30M токенов/день
  GPT-4o-mini: ~$6.75/день = ~$200/месяц
  GPT-4o: ~$112/день = ~$3400/месяц

Локальный: стоимость фиксирована (электричество).
  ~$3.4/месяц при любом количестве запросов.
```


## Когда API

### 1. Максимальное качество

```
Frontier-задачи, где локальные модели не справляются:
  - Сложный reasoning (математика, логика, планирование)
  - Длинный контекст (>128K токенов)
  - Мультимодальность (видео, сложные изображения)
  - Code generation для редких языков/фреймворков
  - Задачи, требующие "мирового знания"
```

### 2. Нет GPU

```
Ситуации:
  - Ноутбук без дискретной GPU
  - Сервер с CPU-only
  - Мобильная разработка
  - Облачная инфраструктура без GPU

Минимальные требования для локального LLM:
  - 8 GiB VRAM -- маленькие модели (3B-7B)
  - 16 GiB VRAM -- средние модели (7B-14B)
  - 24 GiB VRAM -- 32B в Q4
  - 48+ GiB VRAM -- 70B модели
```

### 3. Быстрый старт

```
Время от идеи до работающего прототипа:

API:
  1. Зарегистрироваться (5 минут)
  2. Получить API key (1 минута)
  3. Написать код (10 минут)
  Итого: 15-20 минут

Локальный:
  1. Установить llama.cpp (30 минут)
  2. Скачать модель (30-120 минут, зависит от размера)
  3. Настроить и запустить (20 минут)
  4. Написать код (10 минут)
  Итого: 1.5-3 часа (первый раз)
```

### 4. Масштабируемость

```
Сценарий: SaaS с 100+ одновременными пользователями

Локальный: один GPU обслуживает 1-5 параллельных запросов.
  Для 100 пользователей нужно 20-100 GPU.

API: масштабируется автоматически.
  Провайдер управляет инфраструктурой.
```


## Гибридный подход

Оптимальная стратегия -- комбинация локального и облачного:

### Архитектура

```
+------------------+     +------------------+
| Локальный LLM    |     | Облачный API     |
| (повседневные)   |     | (сложные задачи)  |
+------------------+     +------------------+
         |                        |
         +------+---------+------+
                |         |
                v         v
         +------------------+
         | Маршрутизатор    |
         | (по сложности    |
         | задачи)          |
         +------------------+
                |
                v
         +------------------+
         | Приложение       |
         +------------------+
```

### Реализация маршрутизатора

```python
import requests
from typing import Optional


class HybridLLM:
    """Гибридный LLM: локальный для простых задач, API для сложных."""

    def __init__(
        self,
        local_url: str = "http://localhost:8080",
        api_url: str = "https://api.openai.com/v1",
        api_key: str = "",
        api_model: str = "gpt-4o-mini",
        complexity_threshold: int = 500
    ):
        self.local_url = local_url
        self.api_url = api_url
        self.api_key = api_key
        self.api_model = api_model
        self.complexity_threshold = complexity_threshold

    def _estimate_complexity(self, messages: list[dict]) -> int:
        """
        Оценка сложности запроса (эвристика).

        Возвращает значение 0-1000. Выше = сложнее.
        """
        total_text = " ".join(m["content"] for m in messages if isinstance(m["content"], str))
        score = 0

        # Длинный контекст = сложнее
        score += min(len(total_text) // 100, 200)

        # Ключевые слова сложности
        complex_keywords = [
            "проанализируй", "сравни", "объясни почему",
            "напиши алгоритм", "оптимизируй", "доказательство",
            "архитектура", "рефакторинг", "математика"
        ]
        for kw in complex_keywords:
            if kw in total_text.lower():
                score += 100

        # Простые задачи
        simple_keywords = [
            "переведи", "исправь", "форматируй",
            "кратко", "список", "определение"
        ]
        for kw in simple_keywords:
            if kw in total_text.lower():
                score -= 50

        return max(0, min(1000, score))

    def _call_local(self, messages: list[dict], **kwargs) -> dict:
        """Вызов локального LLM."""
        response = requests.post(
            f"{self.local_url}/v1/chat/completions",
            json={"messages": messages, **kwargs}
        )
        return response.json()

    def _call_api(self, messages: list[dict], **kwargs) -> dict:
        """Вызов облачного API."""
        response = requests.post(
            f"{self.api_url}/chat/completions",
            headers={"Authorization": f"Bearer {self.api_key}"},
            json={"model": self.api_model, "messages": messages, **kwargs}
        )
        return response.json()

    def chat(
        self,
        messages: list[dict],
        force_local: bool = False,
        force_api: bool = False,
        **kwargs
    ) -> dict:
        """
        Отправить запрос, автоматически выбирая локальный или API.

        force_local -- всегда использовать локальный
        force_api -- всегда использовать API
        """
        if force_local:
            return self._call_local(messages, **kwargs)
        if force_api:
            return self._call_api(messages, **kwargs)

        complexity = self._estimate_complexity(messages)

        if complexity > self.complexity_threshold:
            # Сложная задача -> API
            try:
                return self._call_api(messages, **kwargs)
            except Exception:
                # Fallback на локальный
                return self._call_local(messages, **kwargs)
        else:
            # Простая задача -> локальный
            try:
                return self._call_local(messages, **kwargs)
            except Exception:
                # Fallback на API
                return self._call_api(messages, **kwargs)


# Пример использования
llm = HybridLLM(
    local_url="http://localhost:8080",
    api_key="sk-...",
    api_model="gpt-4o-mini",
    complexity_threshold=300
)

# Простой запрос -> локальный
result = llm.chat([
    {"role": "user", "content": "Переведи 'hello' на русский"}
])

# Сложный запрос -> API
result = llm.chat([
    {"role": "user", "content": "Проанализируй архитектуру микросервисов и предложи оптимизацию..."}
])

# Принудительно локальный (приватные данные)
result = llm.chat(
    [{"role": "user", "content": "Проанализируй договор: ..."}],
    force_local=True
)
```

### Стратегии маршрутизации

```
Стратегия          | Локальный                | API
-------------------|--------------------------|----------------------------
По сложности       | Простые задачи           | Сложные задачи
По приватности     | Чувствительные данные    | Открытые данные
По модальности     | Текст                    | Изображения, видео, аудио
По качеству        | Черновики, прототипы     | Финальные результаты
По скорости        | Интерактивный чат        | Пакетная обработка (API дешевле)
По доступности     | Основной                 | Fallback при недоступности
```

## Качество: сравнение моделей

### Приблизительное соответствие

```
Локальная модель        | Сопоставимый API уровень | Задачи
------------------------|-------------------------|------------------
7B-8B Q4_K_M            | GPT-3.5 (базовый)       | Простые задачи
14B Q4_K_M              | GPT-3.5 (хороший)       | Код, перевод
32B Q4_K_M              | GPT-4-mini              | Большинство задач
70B Q4_K_M              | GPT-4-mini (хороший)    | Сложный код, анализ
70B Q8_0                | Между GPT-4-mini и 4o   | Качественный вывод
```

Сравнение приблизительное. На конкретных задачах результаты могут
различаться.

### Где локальные модели уступают

```
Задача                      | Локальный 70B | GPT-4o | Разрыв
----------------------------|---------------|--------|-------
Математика (сложная)        | Средне        | Хорошо | Большой
Reasoning (многошаговый)    | Средне        | Хорошо | Большой
Код (редкие языки)          | Плохо         | Хорошо | Большой
Длинный контекст (>32K)     | Деградация    | Хорошо | Средний
Vision (детальный)          | Средне        | Хорошо | Средний
Мультиязычность             | Хорошо        | Отлично| Малый
```

### Где локальные модели не уступают

```
Задача                      | Локальный 70B | GPT-4o | Разрыв
----------------------------|---------------|--------|-------
Перевод                     | Хорошо        | Хорошо | Малый
```

## Задержка (latency)

### Составляющие задержки

```
API:
  Сетевая задержка (RTT):       50-300 мс
  Ожидание в очереди:            0-2000 мс
  Prompt processing:             100-500 мс
  Token generation (первый):     10-50 мс
  Token generation (остальные):  10-30 мс/токен
  ------------------------------------------
  Time to first token (TTFT):    200-3000 мс

Локальный:
  Сетевая задержка (локальная):  <1 мс
  Ожидание в очереди:            0 мс (один пользователь)
  Prompt processing:             500-5000 мс (зависит от GPU и промпта)
  Token generation:              20-200 мс/токен (зависит от модели)
  ------------------------------------------
  TTFT:                          500-5000 мс
```

### Сравнение для типичных запросов

```
Запрос (1000 токенов input, 200 токенов output):

API (GPT-4o-mini через Groq):
  TTFT: ~200 мс
  Генерация: 200 * 10 мс = 2000 мс
  Итого: ~2.2 сек

API (GPT-4o):
  TTFT: ~500 мс
  Генерация: 200 * 20 мс = 4000 мс
  Итого: ~4.5 сек

Локальный (Qwen 2.5 32B Q4, Radeon 8060S):
  TTFT: ~1000 мс (prompt processing)
  Генерация: 200 * 80 мс = 16000 мс (~12 tok/s)
  Итого: ~17 сек

Локальный (Llama 3.1 70B Q4, Radeon 8060S):
  TTFT: ~2000 мс
  Генерация: 200 * 160 мс = 32000 мс (~6 tok/s)
  Итого: ~34 сек
```

Локальный LLM медленнее API, особенно для больших моделей. Но:
- Нет непредсказуемых задержек из-за перегрузки API
- Нет таймаутов из-за сетевых проблем
- Стабильная производительность


## Приватность и безопасность

### Модель угроз

```
Облачный API:
  Данные передаются:
    Ваш сервер -> [интернет] -> Провайдер -> [модель] -> [логи?]

  Риски:
    - Перехват данных (TLS снижает, но не исключает)
    - Логирование запросов провайдером
    - Утечка через сотрудников провайдера
    - Использование данных для обучения (зависит от ToS)
    - Запросы от гос. органов

Локальный:
  Данные не покидают сервер:
    Ваш сервер -> [модель на этом же сервере] -> ответ

  Риски:
    - Компрометация самого сервера (стандартная ИБ)
    - Нет специфичных рисков LLM
```

### Комплаенс

```
Требование        | Облачный API          | Локальный
-------------------|-----------------------|------------------
GDPR               | Нужен DPA с провайдером| Нет доп. требований
HIPAA              | Нужен BAA             | Нет доп. требований
PCI DSS            | Зависит от провайдера | Нет доп. требований
Гос. тайна         | Недопустимо           | Допустимо (air-gap)
Персональные данные| DPA + оценка рисков   | Стандартные меры ИБ
```


## Доступность и надежность

### Облачный API

```
Плюсы:
  - SLA 99.9%+ (гарантированная доступность)
  - Автоматическое масштабирование
  - Geo-distributed (несколько регионов)

Минусы:
  - Зависимость от интернета
  - Зависимость от провайдера (outage, deprecation)
  - Rate limits (RPM, TPM)
  - Непредсказуемые задержки при пиковой нагрузке
```

### Локальный LLM

```
Плюсы:
  - Нет зависимости от интернета
  - Нет rate limits
  - Предсказуемая производительность

Минусы:
  - Нет автоматического failover
  - Обслуживание -- ваша ответственность
  - Аппаратные сбои -> недоступность
  - Обновления требуют downtime
```


## Масштабирование

### Вертикальное (больше мощности)

```
Локальный:
  1 GPU -> 2 GPU -> 4 GPU
  Сложно: llama-server не поддерживает multi-GPU нативно.
  Нужны: vLLM, TensorRT-LLM, или tensor parallelism.

API:
  Автоматически. Просто платите больше.
```

### Горизонтальное (больше экземпляров)

```
Локальный:
  Запуск нескольких llama-server на разных GPU.
  Load balancer (nginx, HAProxy) распределяет запросы.
  Каждый GPU = один экземпляр модели.

API:
  Автоматически. Rate limits можно увеличить.
```

### Расчет для команды

```
Команда из 5 разработчиков, каждый -- 100 запросов/день

API (GPT-4o-mini):
  5 * 100 * 3K токенов = 1.5M/день = 45M/месяц
  Стоимость: ~$13.5/месяц

Локальный (1 x Radeon 8060S):
  Параллельность: 2-4 запроса (с --parallel 4)
  Достаточно для 500 запросов/день
  Стоимость: ~$43/месяц

Вывод: для команды из 5 человек API дешевле.
Но если данные чувствительные -- локальный оправдан.
```


## Миграция между подходами

### OpenAI-совместимый API

Все основные инструменты (llama-server, Ollama, vLLM) предоставляют
OpenAI-совместимый API. Миграция сводится к изменению URL и API key.

```python
# Код, работающий и с API, и с локальным LLM
import requests


class LLMClient:
    """Универсальный клиент для OpenAI-совместимого API."""

    def __init__(
        self,
        base_url: str = "http://localhost:8080/v1",
        api_key: str = "not-needed",
        model: str = "local-model"
    ):
        self.base_url = base_url
        self.api_key = api_key
        self.model = model

    def chat(self, messages: list[dict], **kwargs) -> str:
        """Универсальный метод чата."""
        response = requests.post(
            f"{self.base_url}/chat/completions",
            headers={"Authorization": f"Bearer {self.api_key}"},
            json={
                "model": self.model,
                "messages": messages,
                **kwargs
            }
        )
        return response.json()["choices"][0]["message"]["content"]


# Конфигурации
configs = {
    "local": {
        "base_url": "http://localhost:8080/v1",
        "api_key": "not-needed",
        "model": "qwen2.5-32b"
    },
    "openai": {
        "base_url": "https://api.openai.com/v1",
        "api_key": "sk-...",
        "model": "gpt-4o-mini"
    },
    "deepseek": {
        "base_url": "https://api.deepseek.com/v1",
        "api_key": "sk-...",
        "model": "deepseek-chat"
    },
    "groq": {
        "base_url": "https://api.groq.com/openai/v1",
        "api_key": "gsk_...",
        "model": "llama-3.1-70b-versatile"
    }
}

# Переключение -- одна строка
client = LLMClient(**configs["local"])
# client = LLMClient(**configs["openai"])

answer = client.chat([
    {"role": "user", "content": "Привет, как дела?"}
])
```


## Рекомендации по выбору

### Алгоритм принятия решения

```
Данные чувствительные?
  |
  +--> Да --> ЛОКАЛЬНЫЙ (приватность важнее)
  |
  +--> Нет --> Нужно максимальное качество?
                |
                +--> Да --> API (frontier-модели)
                |
                +--> Нет --> Есть GPU?
                              |
                              +--> Да --> Больше 100 запросов/день?
                              |            |
                              |            +--> Да --> ЛОКАЛЬНЫЙ (экономия)
                              |            |
                              |            +--> Нет --> API (дешевле)
                              |
                              +--> Нет --> API (нет альтернативы)
```

### Сводная таблица рекомендаций

```
Профиль                      | Рекомендация
-----------------------------|-----------------------------------
Индивидуальный разработчик   | API (GPT-4o-mini) + локальный для экспериментов
Стартап (5-10 человек)       | API + локальный для чувствительных данных
Корпорация (NDA, комплаенс)  | Локальный (основной) + API (по согласованию)
Исследователь AI/ML          | Локальный (полный контроль) + API для сравнения
Без GPU                      | API (единственный вариант)
Без интернета                | Локальный (единственный вариант)
Большой объем данных         | Локальный (предсказуемая стоимость)
```

### Для данной платформы (Radeon 8060S, 96 GiB)

```
Основной рабочий режим:
  Локальный Qwen 2.5 32B Q4_K_M (или 70B Q4_K_M для сложных задач)
  Покрывает 80-90% повседневных задач

Сложные задачи (при необходимости):
  API: GPT-4o-mini (бюджетный) или GPT-4o (максимальное качество)
  Или DeepSeek-V3 (баланс качества и стоимости)

Чувствительные данные:
  Всегда локальный, без исключений.
```
