# Контекстное окно

Контекстное окно -- максимальное количество токенов, которое модель
может обработать за один сеанс. Это общий лимит для входных токенов
(промпт) и выходных (сгенерированный ответ). Контекстное окно --
главный ресурс при работе с LLM: его объём определяет, сколько
информации модель может учитывать одновременно.

## Определение

```
+-----------------------------------------------------------+
|                 Контекстное окно (context window)          |
|                                                           |
|  [system prompt] [user message] ... [assistant response]  |
|  |<------------- input tokens ----------->|<-- output -->||
|  |<---------------- context_size ----------------------->||
+-----------------------------------------------------------+
```

Контекстное окно включает:
- Системный промпт (инструкции для модели)
- Историю диалога (предыдущие сообщения)
- Текущий запрос пользователя
- Контекст (документы, код, RAG-результаты)
- Сгенерированный ответ модели

Всё это должно уместиться в context_size токенов.


### Бюджет контекста

Типичное распределение для чат-сессии:

```
context_size = 8192 токенов

+----------------------------------------------+
| System prompt:        ~200 токенов           |
| История диалога:     ~3000 токенов           |
| Текущий запрос:       ~500 токенов           |
| Запас на ответ:      ~4492 токенов           |
+----------------------------------------------+
  Итого:                8192 токенов
```

Если контекст заполнен -- нужно либо обрезать историю,
либо начать новый диалог, либо увеличить context_size.

## KV-cache

### Что это

KV-cache (Key-Value cache) -- область памяти, где хранятся
промежуточные результаты вычислений attention для каждого
обработанного токена. Без KV-cache каждый decode-шаг требовал
бы пересчёта attention для всех предыдущих токенов.

Подробнее о механизме: [generation.md](generation.md).

### Структура KV-cache

Для каждого обработанного токена, в каждом слое Transformer,
для каждой KV-головы хранится:
- **Key** вектор размерности head_dim
- **Value** вектор размерности head_dim

```
KV-cache структура:

  Слой 1:
    KV-голова 1: [K_1, K_2, ..., K_ctx] [V_1, V_2, ..., V_ctx]
    KV-голова 2: [K_1, K_2, ..., K_ctx] [V_1, V_2, ..., V_ctx]
    ...
    KV-голова H: [K_1, K_2, ..., K_ctx] [V_1, V_2, ..., V_ctx]

  Слой 2:
    (аналогично)
    ...

  Слой L:
    (аналогично)
```


### Почему KV-cache важен

```
Без KV-cache:
  Шаг 100: обработать все 100 токенов -> O(100 * d_model^2 * n_layers)
  Шаг 101: обработать все 101 токенов -> O(101 * d_model^2 * n_layers)
  Итого для 1000 токенов: O(500000 * d_model^2 * n_layers)

С KV-cache:
  Шаг 100: обработать 1 токен + прочитать кэш
  Шаг 101: обработать 1 токен + прочитать кэш
  Итого для 1000 токенов: O(1000 * d_model^2 * n_layers)
                          + чтение кэша (O(n * head_dim))

Экономия: ~500x вычислений
Стоимость: VRAM для хранения кэша
```


## Формула KV-cache

### Полная формула

```
kv_cache_bytes = n_layers * n_kv_heads * head_dim * 2 * context_len * bytes_per_element
                                                    ^
                                                K и V (два тензора)
```

Где:
- `n_layers` -- количество слоёв Transformer
- `n_kv_heads` -- количество KV-голов (при GQA < n_heads)
- `head_dim` -- размерность одной головы (обычно 128)
- `2` -- два тензора: Key и Value
- `context_len` -- длина контекста (количество токенов)
- `bytes_per_element` -- размер одного числа (FP16 = 2, Q8 = 1)


### Параметры популярных моделей

| Модель | n_layers | n_kv_heads | head_dim | GQA-ratio |
|--------|----------|------------|----------|-----------|
| Llama 3.2 1B | 16 | 8 | 64 | 4:1 |
| Llama 3.2 3B | 28 | 8 | 128 | 3:1 |
| Llama 3.1 8B | 32 | 8 | 128 | 4:1 |
| Llama 3.1 70B | 80 | 8 | 128 | 8:1 |
| Llama 3.1 405B | 126 | 8 | 128 | 16:1 |
| Qwen2.5 7B | 28 | 4 | 128 | 7:1 |
| Qwen2.5 32B | 64 | 8 | 128 | 5:1 |
| Qwen2.5 72B | 80 | 8 | 128 | 8:1 |
| Mistral 7B | 32 | 8 | 128 | 4:1 |
| DeepSeek-V3 | 61 | 128 (MLA) | 128 | special |

[!] GQA-ratio = n_heads / n_kv_heads. Большее отношение --
компактнее KV-cache. Llama 3.1 70B: 64 Q-головы,
8 KV-голов -> GQA-ratio 8:1 -> KV-cache в 8 раз меньше
чем при MHA.


### Пример расчёта

Llama 3.1 8B, контекст 32K, FP16:

```
kv_cache = 32 * 8 * 128 * 2 * 32768 * 2

Пошагово:
  n_layers = 32
  n_kv_heads = 8
  head_dim = 128
  K_and_V = 2
  context = 32768
  bytes = 2 (FP16)

  = 32 * 8 * 128 * 2 * 32768 * 2
  = 32 * 8 * 128 * 2 * 32768 * 2
  = 4,294,967,296 bytes
  = 4.0 GB

Итого: 4.0 GB для KV-cache при контексте 32K
```

Llama 3.1 70B, контекст 32K, FP16:

```
kv_cache = 80 * 8 * 128 * 2 * 32768 * 2
         = 10,737,418,240 bytes
         = 10.0 GB
```

### Формула для быстрого расчёта KV-cache

Упрощённая формула (FP16, head_dim=128):

```
KV_GB = n_layers * n_kv_heads * context_K * 0.000488

Где context_K = context_size / 1024

Примеры:
  Llama 3.1 8B, 32K:  32 * 8 * 32 * 0.000488 = 4.0 GB  (верно)
  Llama 3.1 70B, 32K: 80 * 8 * 32 * 0.000488 = 10.0 GB (верно)
  Qwen2.5 7B, 32K:    28 * 4 * 32 * 0.000488 = 1.75 GB (верно)
```

[!] Для Q8 KV-cache -- делить результат на 2.


## Что происходит при превышении

### Truncation (обрезка)

Самый простой подход: если контекст заполнен, обрезать начало.

```
Контекст 4K:

  Шаг 1: [system] [user_1] [assistant_1]                    (3K)
  Шаг 2: [system] [user_1] [assistant_1] [user_2]           (3.5K)
  Шаг 3: [system] [user_1] [assistant_1] [user_2] [asst_2]  (4K)
  Шаг 4: [system] [user_2] [assistant_2] [user_3]           (3.5K)
                   ^^^^^^^^
                   user_1 и assistant_1 обрезаны!

  Модель "забыла" начало диалога.
```

### Sliding Window Attention

Некоторые модели (Mistral 7B) используют sliding window:
каждый токен "видит" только W последних токенов.

```
Sliding window W = 4096:

  Токены: [1] [2] [3] ... [4096] [4097] [4098] ...

  Токен 4098 видит: [3] [4] ... [4098]
  Токен 4098 НЕ видит: [1] [2]

  KV-cache фиксированного размера (не растёт с контекстом)
```

Преимущества:
- Фиксированный VRAM для KV-cache
- Фиксированное время decode-шага

Недостатки:
- Информация за пределами окна теряется
- Не подходит для задач, требующих доступа к началу текста

### Разница truncation и sliding window

```
Truncation:
  Модель обучена с контекстом 128K
  При превышении -- обрезка старых токенов
  До обрезки все токены видят все предыдущие

Sliding window:
  Модель обучена с ограниченным attention
  Каждый токен видит только W последних
  Контекст может быть формально длинным, но "зрение" ограничено
```


## RoPE и позиционное кодирование

### Как модель понимает позицию

Transformer не имеет встроенного понятия порядка токенов.
RoPE (Rotary Positional Embedding) кодирует позицию каждого
токена через вращение вектора.

Подробнее о механизме: [transformer.md](transformer.md).

### Связь с контекстным окном

Модель обучена с RoPE на определённых позициях (0..max_pos).
При выходе за max_pos качество деградирует -- модель
не "понимает" позиции, которых не видела при обучении.

```
Llama 3.1: обучена с позициями 0..131071 (128K)

  Позиция 0-8K:       высокое качество (больше всего данных)
  Позиция 8K-32K:     хорошее качество
  Позиция 32K-128K:   приемлемое качество (меньше данных)
  Позиция >128K:      деградация (за пределами обучения)
```

### RoPE scaling для длинного контекста

Для расширения контекста за пределы обучения применяют
масштабирование RoPE:

```
Исходный RoPE:
  theta(i) = 10000^(-2i/d)
  Позиция m -> вращение на угол m * theta(i)

Linear scaling (factor S):
  theta_new(i) = theta(i) / S
  Позиция m -> вращение на угол m * theta(i) / S
  Эффект: позиция 8K "выглядит" как 4K при S=2

NTK-aware scaling:
  base_new = base * S^(d/(d-2))
  Масштабирует base (10000), а не theta напрямую
  Лучше сохраняет высокочастотные компоненты

YaRN (Yet Another RoPE eNtension):
  Комбинация:
  - NTK-aware scaling для низких частот
  - Без масштабирования для высоких частот
  - Temperature scaling для attention scores
  Наилучшее качество при экстраполяции
```

```bash
# llama-server: RoPE scaling
llama-server \
    --model model.gguf \
    --ctx-size 65536 \           # контекст больше обученного
    --rope-scaling linear \       # тип масштабирования
    --rope-freq-scale 0.5         # коэффициент (1/S)
```

[!] RoPE scaling -- компромисс. Качество на длинных контекстах
хуже, чем у модели, обученной с таким контекстом нативно.
Llama 3.1 обучена с 128K контекстом -- scaling не нужен.


### "Lost in the middle"

Исследования показывают, что модели хуже работают с информацией,
расположенной в середине длинного контекста:

```
Качество извлечения информации по позиции в контексте:

  Качество
    |  #           #
    |  #  #     #  #
    |  #  #  #  #  #
    |  #  #  #  #  #
    |  #  #  #  #  #  #
    |  #  #  #  #  #  #
    +--+--+--+--+--+--+--> Позиция
       |           |  |
       Начало  Середина  Конец

  Начало: хорошее (первые токены attention)
  Середина: худшее ("потеря" информации)
  Конец: хорошее (недавние токены в KV-cache)
```

Практический совет: важную информацию размещать в начале
или в конце контекста.


## Практические ограничения

### VRAM: главный ограничитель

```
Доступная VRAM = Total VRAM - Модель - Overhead

Для KV-cache:
  Доступная VRAM для KV = Total - Model_weights - ~2 GB

Radeon 8060S (96 GiB):

  Llama 3.1 8B Q4:  96 - 5 - 2 = 89 GB для KV
    Макс контекст FP16: 89 / (32*8*128*2*2 / 1024^3) = ~700K
    Макс контекст Q8:   ~1.4M
    --> Ограничение модели (128K), а не VRAM

  Llama 3.1 70B Q4: 96 - 41 - 2 = 53 GB для KV
    Макс контекст FP16: 53 / (80*8*128*2*2 / 1024^3) = ~340K
    Макс контекст Q8:   ~680K
    --> Можно использовать 128K с запасом

  Qwen2.5 32B Q4:   96 - 20 - 2 = 74 GB для KV
    Макс контекст FP16: 74 / (64*8*128*2*2 / 1024^3) = ~600K
    Макс контекст Q8:   ~1.2M
    --> Ограничение модели, а не VRAM
```

### Prefill time: длинный контекст = долгий старт

Время обработки промпта растёт с длиной контекста:

| Контекст | Llama 3.1 8B | Qwen2.5 32B | Llama 3.1 70B |
|----------|-------------|-------------|---------------|
| 4K | ~0.1 с | ~0.4 с | ~0.5 с |
| 8K | ~0.2 с | ~0.8 с | ~1 с |
| 32K | ~1 с | ~4 с | ~7 с |
| 128K | ~8 с | ~30 с | ~50 с |

[!] На длинных контекстах пользователь ожидает первый токен
ответа десятки секунд. Это может быть неприемлемо для
интерактивных приложений.

### Decode speed: деградация на длинных контекстах

Скорость генерации снижается, потому что KV-cache увеличивает
объём чтения из памяти:

```
Llama 3.1 8B Q4_K_M, Radeon 8060S (256 GB/s):

Контекст | Model | KV     | Total | tok/s
---------|-------|--------|-------|------
    4K   | 4.9   | 0.25   | 5.15  | ~50
    8K   | 4.9   | 0.50   | 5.40  | ~47
   16K   | 4.9   | 1.00   | 5.90  | ~43
   32K   | 4.9   | 2.00   | 6.90  | ~37
   64K   | 4.9   | 4.00   | 8.90  | ~29
  128K   | 4.9   | 8.00   | 12.90 | ~20
```

```
Llama 3.1 70B Q4_K_M, Radeon 8060S (256 GB/s):

Контекст | Model | KV     | Total | tok/s
---------|-------|--------|-------|------
    4K   | 40.8  | 0.63   | 41.4  | ~6.2
    8K   | 40.8  | 1.25   | 42.1  | ~6.1
   32K   | 40.8  | 5.00   | 45.8  | ~5.6
  128K   | 40.8  | 20.00  | 60.8  | ~4.2
```


### Качество: больше контекст != лучше качество

Длинный контекст не всегда улучшает результат:

```
Плюсы длинного контекста:
  + Больше информации доступно
  + Длинная история диалога
  + Большие документы целиком

Минусы:
  - "Lost in the middle" (информация в середине теряется)
  - Больше шума (нерелевантная информация)
  - Медленнее prefill
  - Медленнее decode
  - Больше VRAM
```


## Когда нужен длинный контекст

### RAG (Retrieval-Augmented Generation)

```
Запрос: "Какие изменения внесены в API в версии 3.2?"

Retrievаl:
  [doc_1: changelog v3.2 -- 2000 токенов]
  [doc_2: migration guide -- 1500 токенов]
  [doc_3: API reference -- 3000 токенов]

Промпт:
  system + documents + query = ~7000 токенов
  + ответ ~1000 токенов
  Итого: ~8000 токенов --> контекст 8K минимум, 16K рекомендуется
```

### Анализ документов

```
Задача: суммаризация отчёта на 50 страниц

Документ: ~30000 русских слов -> ~60000 токенов
  + system prompt + query + ответ
  Итого: ~65000 токенов --> контекст 64K минимум
```

### Кодирование (большие проекты)

```
Задача: рефакторинг модуля из 15 файлов

  Файлы: ~5000 строк кода -> ~15000 токенов
  + system prompt + instructions + generated code
  Итого: ~20000 токенов --> контекст 32K
```

### Длинные диалоги

```
Техподдержка: многоходовой диалог с контекстом

  20 обменов сообщениями:
  ~20 * (user: 200 + assistant: 500) = 14000 токенов
  + system prompt: 500
  Итого: ~14500 --> контекст 16K
```


## Когда хватает короткого

### Простой чат

```
Вопрос: "Что такое HTTP?"
Ответ: ~200 токенов

Итого: ~400 токенов --> контекст 1K достаточно
```

### Автодополнение кода (FIM)

```
Контекст: 50 строк вокруг курсора -> ~500 токенов
Дополнение: ~20 токенов

Итого: ~520 токенов --> контекст 1K-2K
```

### Классификация / extraction

```
Текст для классификации: ~500 токенов
Ответ: 1-10 токенов

Итого: ~600 токенов --> контекст 1K
```

### Перевод коротких текстов

```
Исходный текст: ~500 токенов
Перевод: ~700 токенов (русский длиннее)

Итого: ~1200 токенов --> контекст 2K
```


## Оптимизации контекста

### 1. KV-cache квантизация

Снижение precision KV-cache для экономии VRAM:

```bash
llama-server \
    --model model.gguf \
    --ctx-size 32768 \
    --cache-type-k q8_0 \
    --cache-type-v q8_0
```

| KV-cache формат | Множитель VRAM | Потеря качества |
|----------------|---------------|-----------------|
| FP16 | 1.0x | Нет |
| Q8_0 | 0.5x | Минимальная |
| Q5_1 | 0.3x | Малая |
| Q4_0 | 0.25x | Заметная |

### 2. Prompt caching

Переиспользование KV-cache для повторяющегося системного
промпта:

```
Запрос 1: [SYSTEM: 1000 токенов] [USER_1: 200 токенов]
  --> KV-cache для SYSTEM сохраняется

Запрос 2: [SYSTEM: 1000 токенов] [USER_2: 300 токенов]
  --> KV-cache для SYSTEM переиспользуется
  --> Prefill только для USER_2 (300 токенов вместо 1300)

Экономия prefill: ~77%
```

### 3. Context pruning

Удаление нерелевантных частей контекста перед отправкой:

```
Вместо:
  [system] [msg_1] [msg_2] ... [msg_20] [query]
  (все 20 сообщений)

Оптимизированно:
  [system] [msg_18] [msg_19] [msg_20] [query]
  (только последние 3 релевантных)

Или с суммаризацией:
  [system] [summary_of_1-17] [msg_18] [msg_19] [msg_20] [query]
```

### 4. Эффективное использование контекста

```
Рекомендации:

1. Системный промпт -- компактный
   Плохо:  "Ты -- полезный ассистент. Ты всегда отвечаешь подробно
            и точно. Ты помогаешь пользователю..." (200 токенов)
   Лучше:  "Ассистент. Ответы: краткие, точные." (10 токенов)

2. RAG -- релевантные фрагменты
   Плохо:  весь документ (60K токенов)
   Лучше:  3-5 релевантных фрагментов (3-5K токенов)

3. Код -- только нужные файлы
   Плохо:  весь проект
   Лучше:  файл с багом + интерфейсы зависимостей

4. Важную информацию -- в начало или конец
   (из-за "lost in the middle")
```

```
Бюджет: 96 GiB = ~96 GB
```

## Ключевые формулы (справочник)

```
KV-cache (FP16):
  bytes = n_layers * n_kv_heads * head_dim * 2 * context * 2

KV-cache (упрощённая):
  GB = n_layers * n_kv_heads * context_K * 0.000488
  (context_K = context / 1024, head_dim = 128, FP16)

Полный VRAM:
  VRAM = model_weights + KV_cache + overhead (~2 GB)

Максимальный контекст:
  max_ctx = (available_VRAM - model_weights - 2 GB) / kv_per_token
  kv_per_token = n_layers * n_kv_heads * head_dim * 2 * 2 bytes

Скорость с учётом KV-cache:
  tok/s = bandwidth / (model_size + kv_cache_size)
```
