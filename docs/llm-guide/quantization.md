## Зачем нужна квантизация

Большая языковая модель -- это набор числовых параметров (весов). Каждый вес
хранится в определенном формате с плавающей точкой. Размер модели в памяти
напрямую зависит от количества параметров и точности их хранения.

Пример для модели с 70 миллиардами параметров:

```
Формат    | Бит на вес | Размер модели | Помещается в 96 GiB?
----------|------------|---------------|---------------------
FP32      | 32         | 280 GiB       | Нет
FP16      | 16         | 140 GiB       | Нет
INT8 (Q8) | 8          | 70 GiB        | Да (с запасом)
INT4 (Q4) | 4          | ~40 GiB       | Да (свободно)
INT2 (Q2) | 2          | ~20 GiB       | Да
```

Формула расчета размера:

```
Размер (GiB) = (параметры * бит_на_вес) / 8 / 1024^3
```

Для 70B модели в FP16:

```
(70 * 10^9 * 16) / 8 / (1024^3) = ~130 GiB (+ overhead ~10 GiB)
```

Квантизация до Q4_K_M дает экономию примерно в 3.5 раза:

```
140 GiB (FP16) -> 40 GiB (Q4_K_M)
```

Это позволяет запустить модели, которые иначе не помещаются в VRAM, и
увеличить скорость генерации за счет уменьшения объема данных, читаемых
из памяти.


## Как работает квантизация

### Базовый принцип

Квантизация -- это отображение непрерывного диапазона значений на дискретный
набор уровней. Чем меньше уровней, тем меньше бит для хранения, но выше
потеря точности.

```
FP16 (65536 уровней)                    INT4 (16 уровней)
     |                                       |
     v                                       v
-1.0 -------- 0.0 -------- +1.0        -1.0 .. -0.7 .. 0.0 .. +0.7 .. +1.0
     тысячи промежуточных                    только 16 значений
     значений
```

### Пошаговый процесс

1. **Калибровка**: определение диапазона значений весов для каждого слоя
2. **Масштабирование**: вычисление scale и zero-point для отображения
3. **Округление**: каждый вес округляется до ближайшего квантованного значения
4. **Упаковка**: квантованные значения записываются в компактном формате

```
Исходные веса (FP16):  [0.123, -0.456, 0.789, -0.012, 0.345, ...]
                                    |
                                    v
                           Калибровка: min=-0.456, max=0.789
                           Scale = (max - min) / (2^4 - 1) = 0.083
                                    |
                                    v
Квантованные (INT4):   [6, 0, 15, 5, 9, ...]
                                    |
                                    v
                        Хранение: 4 бита на каждое значение
```

### Симметричная vs асимметричная квантизация

**Симметричная**: zero-point = 0, диапазон [-max, +max]

```
Дано: веса в диапазоне [-0.5, +0.8]
Расширяем до [-0.8, +0.8]
Scale = 0.8 / 7 = 0.114  (для INT4: [-7, +7])
```

**Асимметричная**: zero-point != 0, диапазон [min, max]

```
Дано: веса в диапазоне [-0.5, +0.8]
Scale = (0.8 - (-0.5)) / 15 = 0.087  (для INT4: [0, 15])
Zero-point = round(-(-0.5) / 0.087) = 6
```

Асимметричная квантизация эффективнее использует битовый бюджет,
но требует хранения дополнительного параметра (zero-point).


## Форматы данных: от FP32 до INT2

### Представления с плавающей точкой

```
FP32 (32 бита): 1 знак + 8 экспонента + 23 мантисса
    Диапазон: ~1.2 * 10^-38 .. 3.4 * 10^38
    Точность: ~7 десятичных цифр

FP16 (16 бит): 1 знак + 5 экспонента + 10 мантисса
    Диапазон: ~6.1 * 10^-5 .. 65504
    Точность: ~3.3 десятичных цифры

BF16 (16 бит): 1 знак + 8 экспонента + 7 мантисса
    Диапазон: как FP32 (~1.2 * 10^-38 .. 3.4 * 10^38)
    Точность: ~2.4 десятичных цифры
```

BF16 (bfloat16) -- компромисс: сохраняет диапазон FP32, жертвуя точностью.
Популярен в обучении, потому что не требует loss scaling.

### Целочисленные представления

```
INT8 (8 бит):  256 уровней   [-128, 127] или [0, 255]
INT4 (4 бита): 16 уровней    [-8, 7] или [0, 15]
INT3 (3 бита): 8 уровней     [-4, 3] или [0, 7]
INT2 (2 бита): 4 уровня      [-2, 1] или [0, 3]
```

Каждый шаг вниз уменьшает количество уровней вдвое и повышает ошибку
квантизации.


## GGUF-квантизации

GGUF (GGML Universal Format) -- формат хранения квантованных моделей,
используемый в llama.cpp. Содержит метаданные модели, токенизатор и
квантованные веса в одном файле.

```
BF16     | 16.0    | Brain float 16                    | *****    | **
F16      | 16.0    | Половинная точность               | *****    | **
Q8_0     | 8.5     | 8-бит, простая квантизация         | ****+    | ***
Q6_K     | 6.6     | 6-бит, K-квантизация              | ****     | ***+
Q5_K_M   | 5.7     | 5-бит, K-кв., medium              | ****     | ****
Q5_K_S   | 5.5     | 5-бит, K-кв., small               | ***+     | ****
Q4_K_M   | 4.8     | 4-бит, K-кв., medium              | ***+     | ****+
Q4_K_S   | 4.6     | 4-бит, K-кв., small               | ***      | ****+
Q3_K_M   | 3.9     | 3-бит, K-кв., medium              | ***      | *****

```

### Суффиксы в названиях

- **_S** (Small): минимальный размер в пределах данной битности. Attention и
  feed-forward слои квантуются одинаково
- **_M** (Medium): баланс размера и качества. Attention-слои в более высокой
  точности
- **_L** (Large): максимальное качество для данной битности. Больше слоев
  сохраняют высокую точность

### IQ-квантизации (importance-weighted)

IQ-форматы (IQ1, IQ2, IQ3, IQ4) используют importance-based квантизацию:
веса, вносящие больший вклад в итоговый результат, квантуются с меньшей
ошибкой. Это дает лучшее качество при том же битрейте по сравнению с
обычными Q-форматами.

```
Q3_K_M  (3.9 бит) vs IQ3_M (3.4 бита):
  - IQ3_M меньше по размеру
  - Качество сопоставимо или лучше
  - Но IQ-форматы медленнее на CPU
  - На GPU разница в скорости минимальна
```


## K-квантизация: mixed precision

K-квантизация -- ключевая технология, обеспечивающая высокое качество при
низком битрейте. Идея: не все слои модели одинаково важны для итогового
качества.

### Как работает

```
Transformer-блок:
  +----------------------------------+
  | Self-Attention                   |
  |   Q, K, V проекции   <-- важные |  -> более высокая точность
  |   Output проекция     <-- важные |  -> более высокая точность
  +----------------------------------+
  | Feed-Forward Network             |
  |   Gate проекция                  |  -> стандартная точность
  |   Up проекция                    |  -> стандартная точность
  |   Down проекция                  |  -> стандартная точность
  +----------------------------------+
  | LayerNorm/RMSNorm     <-- важные |  -> всегда FP32 или FP16
  +----------------------------------+
```

### Пример: Q4_K_M

В формате Q4_K_M слои квантуются по-разному:

```
Компонент               | Квантизация | Бит/вес
------------------------|-------------|--------
Embedding               | Q6_K        | 6.6
Attention Q, K проекции | Q4_K        | 4.5
Attention V, O проекции | Q6_K        | 6.6
FFN gate, up            | Q4_K        | 4.5
FFN down                | Q6_K        | 6.6
Output head             | Q6_K        | 6.6
RMSNorm                 | F32         | 32.0
```

Средний битрейт: ~4.8 бит/вес. Но критически важные слои хранятся в
более высокой точности, что минимизирует деградацию качества.

### Сравнение K-вариантов для 4-бит

```
Формат   | Средний бит/вес | Attention | FFN    | Размер 70B
---------|-----------------|-----------|--------|----------
Q4_K_S   | 4.6             | Q4_K      | Q4_K   | ~38 GiB
Q4_K_M   | 4.8             | Q4_K/Q6_K | Q4_K/Q6_K | ~40 GiB
Q4_K_L   | 5.0             | Q6_K      | Q4_K/Q6_K | ~42 GiB
```


## Perplexity: метрика качества

Perplexity (PPL) -- основная метрика для оценки качества языковой модели.
Определяется как экспонента средней кросс-энтропии:

```
PPL = exp(-1/N * SUM(log P(token_i | context_i)))
```

Интерпретация: perplexity = K означает, что модель "выбирает" из K
равновероятных вариантов на каждом шаге. Чем меньше, тем лучше.

### Таблица perplexity для Llama 3.1 70B (wiki-test-2)

```
Квантизация | PPL    | Деградация | Размер | Пригодность
------------|--------|------------|--------|-------------------
F16         | 3.32   | baseline   | 140 GiB| Эталон
Q8_0        | 3.33   | +0.3%      | 70 GiB | Максимальное качество
Q6_K        | 3.34   | +0.6%      | 55 GiB | Отличное качество
Q5_K_M      | 3.36   | +1.2%      | 48 GiB | Хорошее качество
Q4_K_M      | 3.39   | +2.1%      | 40 GiB | Рабочий стандарт
Q4_K_S      | 3.42   | +3.0%      | 38 GiB | Приемлемо
Q3_K_M      | 3.55   | +6.9%      | 33 GiB | Заметная деградация
Q3_K_S      | 3.68   | +10.8%     | 30 GiB | Ощутимая потеря
Q2_K        | 4.10   | +23.5%     | 25 GiB | Крайний случай
```

### Интерпретация деградации

```
Деградация < 2%   : Разница незаметна для большинства задач
Деградация 2-5%   : Незначительная потеря, приемлемо для работы
Деградация 5-10%  : Заметно на сложных задачах (math, code, structured output)
Деградация 10-20% : Существенная потеря, только для простых задач
Деградация > 20%  : Серьезная деградация, ограниченное применение
```

### Как измерить perplexity

```bash
# Измерение perplexity на тестовом наборе
# Требует скачанного wikitext-2-raw
./llama-perplexity \
    --model model-q4_k_m.gguf \
    --file wikitext-2-raw/wiki.test.raw \
    --ctx-size 2048 \
    --threads 8
```

Время измерения: от 10 минут (7B) до нескольких часов (70B).


## Скорость vs качество

### Почему меньше бит = быстрее

Генерация токенов в LLM ограничена пропускной способностью памяти
(memory-bound). На каждый токен модель читает все веса из VRAM:

```
Время генерации одного токена:
  t = размер_модели / пропускная_способность_памяти

Radeon 8060S: 256 GB/s пропускная способность

Модель 70B:
  Q8_0 (70 GiB):  70 / 256 = 0.273 сек = 3.7 tok/s
  Q4_K_M (40 GiB): 40 / 256 = 0.156 сек = 6.4 tok/s
  Q2_K (25 GiB):  25 / 256 = 0.098 сек = 10.2 tok/s
```

Это теоретический максимум. Реальная скорость ниже из-за накладных расходов
на деквантизацию, attention KV-cache и другие вычисления.

### Таблица скоростей (оценка для Radeon 8060S, 256 GB/s)

```
Модель/Квантизация  | Размер  | tg (tok/s) | pp (tok/s) | Примечание
--------------------|---------|------------|------------|----------
Llama 3.1 8B Q4_K_M | 5 GiB  | ~40-50     | ~300-400   | Быстрая работа
Llama 3.1 8B Q8_0   | 8.5 GiB| ~25-30     | ~200-300   | Хороший запас
Llama 3.1 70B Q4_K_M| 40 GiB | ~5-7       | ~50-80     | Основной рабочий вариант
Llama 3.1 70B Q8_0  | 70 GiB | ~3-4       | ~30-50     | Медленно, но качественно
Qwen 2.5 32B Q4_K_M | 19 GiB | ~10-13     | ~100-150   | Баланс скорости/качества
Qwen 2.5 72B Q4_K_M | 42 GiB | ~5-6       | ~50-70     | Аналогично 70B
```

Обозначения:
- **tg** (token generation) -- скорость генерации токенов (один за другим)
- **pp** (prompt processing) -- скорость обработки промпта (параллельно)

### Баланс: что выбрать

```
Приоритет -- скорость:
  Интерактивная работа, чат, быстрые ответы
  -> Меньшая модель (8B-32B) в Q4_K_M
  -> Или большая модель в Q3_K_M / Q2_K

Приоритет -- качество:
  Код, анализ, сложные задачи
  -> Большая модель (70B+) в Q4_K_M или Q5_K_M
  -> Q8_0 если скорость не критична

Баланс:
  Повседневная работа
  -> 32B-70B в Q4_K_M -- оптимальный компромисс
```


## Таблица моделей для 96 GiB VRAM

Radeon 8060S с 96 GiB unified VRAM позволяет запускать модели, недоступные
на большинстве потребительских GPU. Ниже -- карта того, что помещается.

### Важно: KV-cache

Помимо весов модели, VRAM занимает KV-cache (кэш ключей и значений для
attention). Размер KV-cache зависит от контекста:

```
KV-cache (GiB) = 2 * n_layers * n_kv_heads * head_dim * context_len * bytes_per_param / 1024^3

Пример для Llama 3.1 70B, контекст 8K, FP16:
  2 * 80 * 8 * 128 * 8192 * 2 / 1024^3 = ~2 GiB

Пример для Llama 3.1 70B, контекст 32K, FP16:
  2 * 80 * 8 * 128 * 32768 * 2 / 1024^3 = ~8 GiB

С квантизацией KV-cache (Q4):
  ~8 GiB -> ~2 GiB (контекст 32K)
```

## Когда какую квантизацию выбрать

### Q4_K_M -- рабочий стандарт

Потеря качества ~1-2% от FP16. Подходит для:
- Генерация текста
- Чат-ассистент
- Код (при достаточно большой модели)
- Суммаризация
- Перевод

### Q8_0 -- максимальное качество

Потеря качества <0.5% от FP16. Оправдана когда:
- Есть достаточно VRAM
- Скорость не критична
- Нужно максимальное качество для benchmarks
- Задачи, чувствительные к точности (математика, логика)
- Сравнение с FP16 для оценки деградации

### Q5_K_M / Q6_K -- промежуточный вариант

Потеря качества 0.5-1.5%. Выбирают когда:
- Q4 ощутимо теряет на конкретной задаче
- Q8 не помещается или слишком медленный
- Structured output (JSON) -- большая точность = меньше ошибок формата

### Q3_K_M и ниже -- компромиссные варианты

Потеря качества 5-10%+. Допустимы когда:
- Нужна максимальная скорость
- Простые задачи (классификация, извлечение сущностей)
- Модель значительно больше, чем нужно для задачи
- Прототипирование и эксперименты

### Q2_K -- крайний случай

Потеря качества 15-25%. Только когда:
- Модель иначе не помещается в VRAM
- Лучше большая модель в Q2, чем маленькая в Q4 (спорно, зависит от задачи)
- Быстрая проверка возможностей модели

## Практика: квантизация с llama.cpp

### Скачивание готовых квантизованных моделей

Большинство моделей доступны в квантизованном виде на HuggingFace. Искать
репозитории с суффиксом `-GGUF`:

```bash
# Пример: скачивание Qwen 2.5 32B в Q4_K_M
# Требуется huggingface-cli (pip install huggingface-hub)
huggingface-cli download \
    Qwen/Qwen2.5-32B-Instruct-GGUF \
    qwen2.5-32b-instruct-q4_k_m.gguf \
    --local-dir ./models/
```

### Самостоятельная квантизация

Если нужна конкретная квантизация, отсутствующая в готовом виде:

```bash
# 1. Скачать модель в FP16/BF16 (safetensors)
git clone https://huggingface.co/Qwen/Qwen2.5-32B-Instruct

# 2. Конвертировать в GGUF F16
python convert_hf_to_gguf.py \
    Qwen2.5-32B-Instruct/ \
    --outfile qwen2.5-32b-f16.gguf \
    --outtype f16

# 3. Квантизовать
./llama-quantize \
    qwen2.5-32b-f16.gguf \
    qwen2.5-32b-q4_k_m.gguf \
    Q4_K_M

# Доступные типы квантизации:
#   Q2_K, Q3_K_S, Q3_K_M, Q3_K_L,
#   Q4_0, Q4_1, Q4_K_S, Q4_K_M,
#   Q5_0, Q5_1, Q5_K_S, Q5_K_M,
#   Q6_K, Q8_0, F16, BF16, F32,
#   IQ1_M, IQ2_M, IQ2_S, IQ3_M, IQ3_S,
#   IQ3_XS, IQ4_NL, IQ4_XS
```

### Запуск квантизованной модели

```bash
# Запуск llama-server с квантизованной моделью
llama-server \
    --model ./models/qwen2.5-32b-q4_k_m.gguf \
    --host 0.0.0.0 \
    --port 8080 \
    --ctx-size 8192 \
    --n-gpu-layers 999 \
    --threads 8
```

Параметр `--n-gpu-layers 999` выгружает все слои на GPU. Если модель не
помещается целиком, llama-server автоматически оставит часть слоев на CPU
(значительно медленнее).

### Квантизация KV-cache

llama-server поддерживает квантизацию KV-cache для экономии VRAM:

```bash
llama-server \
    --model ./models/llama-70b-q4_k_m.gguf \
    --ctx-size 32768 \
    --cache-type-k q4_0 \
    --cache-type-v q4_0 \
    --n-gpu-layers 999
```

Это уменьшает KV-cache в ~4 раза с минимальной потерей качества.
Для контекста 32K на 70B модели: ~8 GiB -> ~2 GiB.
