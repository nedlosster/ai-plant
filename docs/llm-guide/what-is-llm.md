# Что такое LLM

LLM (Large Language Model) -- большая языковая модель. Это нейронная сеть
с миллиардами обучаемых параметров (весов), натренированная на массивах
текстовых данных. Основная способность -- генерация текста: модель
принимает последовательность токенов и предсказывает следующий.

## Определение

Большая языковая модель -- это:

1. **Нейронная сеть** на основе архитектуры Transformer (подробнее:
   [transformer.md](transformer.md))
2. **Миллиарды параметров** -- обучаемых числовых коэффициентов, которые
   определяют поведение модели
3. **Обучена на текстовых данных** -- книги, статьи, код, веб-страницы,
   переписки
4. **Задача обучения** -- предсказание следующего токена (next token
   prediction)

Ключевой принцип: модель не "понимает" текст в человеческом смысле.
Она выучила статистические закономерности языка и способна генерировать
текст, который выглядит осмысленным, потому что следует этим
закономерностям.

```
Входные токены:    [Столица] [Франции] [--] [это]
                              |
                              v
                    +-------------------+
                    |    LLM (7B)       |
                    |  нейронная сеть   |
                    |  7 млрд весов     |
                    +-------------------+
                              |
                              v
Предсказание:      [Париж]  (вероятность 0.92)
                    [город]  (вероятность 0.03)
                    [Лион]   (вероятность 0.01)
                    ...
```

Модель выбирает один из токенов словаря (32k-128k вариантов) на основе
вычисленных вероятностей. Процесс выбора называется сэмплированием
(подробнее: [sampling.md](sampling.md)).


### Параметры и веса

"Параметр" и "вес" -- синонимы в контексте LLM. Это числа (обычно
float16 или bfloat16), которые модель выучила во время обучения.
Каждый параметр -- это один элемент матрицы в одном из слоёв сети.

Пример масштаба:

| Модель | Параметры | Размер в FP16 | Аналогия |
|--------|-----------|---------------|----------|
| Llama 3.2 1B | 1 000 000 000 | ~2 GB | Карманный калькулятор |
| Llama 3.1 8B | 8 000 000 000 | ~16 GB | Настольный компьютер |
| Llama 3.1 70B | 70 000 000 000 | ~140 GB | Серверная стойка |
| DeepSeek-V3 | 671 000 000 000 | ~1.3 TB | Дата-центр |


### Обучение: три этапа

Создание LLM -- трёхэтапный процесс:

```
Этап 1: Pre-training (предобучение)
+-------------------------------------------------------+
|  Корпус текстов (триллионы токенов)                   |
|  - веб-страницы (Common Crawl)                        |
|  - книги, статьи, Википедия                           |
|  - код (GitHub)                                       |
|  - научные публикации                                 |
|                                                       |
|  Задача: предсказать следующий токен                  |
|  Результат: base model (базовая модель)               |
|  Стоимость: миллионы долларов, тысячи GPU-часов       |
+-------------------------------------------------------+
                        |
                        v
Этап 2: Fine-tuning (дообучение)
+-------------------------------------------------------+
|  Датасеты инструкций и диалогов                       |
|  - пары (вопрос, ответ)                               |
|  - многоходовые диалоги                               |
|  - задачи с chain-of-thought рассуждениями            |
|                                                       |
|  Методы: SFT (Supervised Fine-Tuning)                 |
|  Результат: instruct model (модель-ассистент)         |
|  Стоимость: тысячи долларов, десятки GPU-часов        |
+-------------------------------------------------------+
                        |
                        v
Этап 3: Alignment (выравнивание)
+-------------------------------------------------------+
|  RLHF / DPO / ORPO                                   |
|  - обучение на предпочтениях людей                    |
|  - какой ответ лучше из двух                          |
|  - фильтрация вредного контента                       |
|                                                       |
|  Результат: aligned model (выровненная модель)        |
|  Стоимость: тысячи долларов                           |
+-------------------------------------------------------+
```

Подробнее об обучении: [Training](../training/README.md).

Base model (без fine-tuning) не умеет вести диалог -- она просто
продолжает текст. Instruct-версии (с суффиксом `-Instruct`,
`-Chat`, `-it`) обучены следовать инструкциям.


## Задачи, решаемые LLM

LLM -- это универсальный текстовый процессор. Одна и та же модель
способна решать разные задачи без дополнительного обучения:

### Генерация текста

```
Вход:  "Напиши краткое описание протокола HTTP"
Выход: "HTTP (Hypertext Transfer Protocol) -- протокол прикладного
        уровня для передачи гипертекстовых документов..."
```

### Перевод

```
Вход:  "Translate to English: Контекстное окно определяет
        максимальное количество токенов"
Выход: "The context window determines the maximum number of tokens"
```

### Суммаризация

```
Вход:  "Summarize: [длинный текст на 5000 слов]"
Выход: "Статья описывает три ключевых аспекта: ..."
```

### Генерация кода

```
Вход:  "Напиши функцию на Python для чтения CSV и подсчёта
        среднего значения в столбце 'price'"
Выход: def avg_price(path: str) -> float:
           import csv
           with open(path) as f:
               reader = csv.DictReader(f)
               prices = [float(row['price']) for row in reader]
           return sum(prices) / len(prices) if prices else 0.0
```

### Рассуждения (reasoning)

```
Вход:  "У Маши 5 яблок. Она дала Пете половину и ещё одно.
        Сколько осталось?"
Выход: "Половина от 5 = 2.5, но яблоки целые, значит 2.
        2 + 1 = 3 отдала. 5 - 3 = 2 осталось."
```

[!] Качество решения зависит от размера модели, данных обучения
и формулировки запроса (prompt). Подробнее о влиянии промптов --
в статье про prompt engineering.

### Классификация и анализ

```
Вход:  "Определи тональность отзыва: 'Товар пришёл быстро,
        но упаковка была повреждена'"
Выход: "Смешанная тональность: положительная (доставка),
        отрицательная (упаковка)"
```

### Структурированный вывод

```
Вход:  "Извлеки из текста имя, возраст и город в формате JSON:
        'Иван Петров, 34 года, живёт в Новосибирске'"
Выход: {"name": "Иван Петров", "age": 34, "city": "Новосибирск"}
```

### Таблица задач и рекомендуемых размеров моделей

| Задача | Минимум | Рекомендуемый | Комментарий |
|--------|---------|---------------|-------------|
| Автодополнение кода | 1B | 3-7B | FIM-модели (fill-in-the-middle) |
| Простой чат | 3B | 7-8B | Бытовые вопросы, FAQ |
| Перевод | 7B | 13-32B | Чем больше модель, тем лучше нюансы |
| Суммаризация | 7B | 13-32B | Нужен длинный контекст |
| Код (сложные задачи) | 7B | 32-70B | Архитектура, рефакторинг |
| Reasoning | 13B | 32-70B | Математика, логика, анализ |
| RAG | 7B | 13-32B | Работа с контекстом из документов |


## Краткая история

Хронология ключевых событий, формирующих текущий ландшафт LLM:

### 2017 -- Transformer

Публикация статьи "Attention Is All You Need" (Vaswani et al.).
Архитектура Transformer заменила RNN/LSTM для задач NLP.
Подробнее: [transformer.md](transformer.md).

### 2018-2019 -- GPT и BERT

- **BERT** (2018) -- encoder-модель от Google, задала стандарт
  для задач NLU (понимание языка)
- **GPT-2** (2019) -- decoder-модель от OpenAI, 1.5B параметров.
  Первая демонстрация генерации связного текста

### 2020 -- GPT-3

175B параметров. Продемонстрировал few-shot learning: модель
решает задачи по нескольким примерам без дообучения. Доступен
только через API.

### 2022 -- ChatGPT

Применение RLHF к GPT-3.5. Первый массовый чат-интерфейс для LLM.
100 млн пользователей за 2 месяца. Начало "AI-бума".

### 2023 -- Open-source: Llama

- **Llama** (февраль 2023) -- Meta выпускает 7B-65B моделей.
  Веса утекли в открытый доступ
- **Llama 2** (июль 2023) -- официальный open-source релиз.
  7B, 13B, 70B с instruct-версиями
- **Mistral 7B** (сентябрь 2023) -- небольшая модель от Mistral AI,
  превзошла Llama 2 13B по качеству
- **Mixtral 8x7B** (декабрь 2023) -- первая популярная MoE-модель

### 2024 -- Расцвет open-source

- **Llama 3** (апрель 2024) -- 8B и 70B, значительный рост качества
- **Llama 3.1** (июль 2024) -- 405B, первая open-source модель
  уровня GPT-4
- **Qwen2** (июнь 2024) -- серия от Alibaba, сильные
  мультиязычные модели
- **Qwen2.5** (сентябрь 2024) -- 0.5B-72B, Coder и Math варианты
- **DeepSeek-V2** (май 2024) -- MoE 236B, эффективная архитектура

### 2025 -- Конкурентный ландшафт

- **DeepSeek-V3** (январь 2025) -- MoE 671B (37B active),
  конкурент GPT-4o по качеству, open-source
- **DeepSeek-R1** (январь 2025) -- reasoning-модель, цепочки
  рассуждений
- **Qwen3** (2025) -- развитие серии, модели для кода
- **Llama 4** (апрель 2025) -- MoE-архитектура, мультимодальность
- **Gemma 3** (2025) -- open-source от Google, компактные модели

```
Хронология (масштаб условный):

2017      2019     2020    2022      2023         2024          2025
  |         |        |       |         |            |             |
  v         v        v       v         v            v             v
Transformer GPT-2   GPT-3  ChatGPT  Llama 1/2   Llama 3      DeepSeek-V3
            1.5B    175B   RLHF     Mistral     Qwen2.5      Qwen3
                                    Mixtral     DeepSeek-V2  Llama 4
                                                             Gemma 3
```

## Размеры моделей

"B" в названии модели означает "billion" -- миллиард параметров.
Llama 3.1 70B содержит примерно 70 миллиардов параметров.

### Что определяет количество параметров

Количество параметров зависит от трёх основных гиперпараметров
архитектуры Transformer (подробнее: [transformer.md](transformer.md)):

| Гиперпараметр | Описание | Пример (8B) | Пример (70B) |
|---------------|----------|-------------|--------------|
| `n_layers` | Количество слоёв (блоков) | 32 | 80 |
| `d_model` | Размерность скрытого состояния | 4096 | 8192 |
| `n_heads` | Количество голов attention | 32 | 64 |

Приблизительная формула:

```
params ~ 12 * n_layers * d_model^2
```

Для Llama 3.1 8B: `12 * 32 * 4096^2 ~ 6.4B` (остаток --
embedding-слой и другие компоненты).


### Как размер влияет на качество

Общее правило: больше параметров = лучше качество. Но зависимость
нелинейная -- каждое удвоение параметров даёт всё меньший прирост.

```
Качество (условные единицы)

  100 |                                        * 405B
      |                              * 70B
   80 |                    * 32B
      |           * 13B
   60 |     * 8B
      | * 3B
   40 |
      |
   20 |
      +--+------+------+------+------+------+----> Размер (B)
         1      8     13     32     70    405
```

[!] Это упрощение. На практике качество зависит от:
- объёма и качества данных обучения
- архитектуры (MoE может быть качественнее Dense при равном compute)
- метода alignment (RLHF, DPO)
- конкретной задачи

Qwen2.5-72B может превосходить Llama 3.1 70B на одних задачах
и уступать на других, несмотря на похожий размер.


### Размер и потребление VRAM

Каждый параметр хранится в памяти. Формат хранения определяет
объём VRAM:

| Формат | Байт/параметр | 7B | 13B | 34B | 70B |
|--------|--------------|-----|------|------|------|
| FP32 | 4 | 28 GB | 52 GB | 136 GB | 280 GB |
| FP16 / BF16 | 2 | 14 GB | 26 GB | 68 GB | 140 GB |
| Q8_0 | ~1 | 7 GB | 13 GB | 34 GB | 70 GB |
| Q4_K_M | ~0.56 | 4 GB | 7.4 GB | 19 GB | 40 GB |
| Q3_K_M | ~0.44 | 3.1 GB | 5.7 GB | 15 GB | 31 GB |

[!] Это только веса модели. Дополнительно нужна VRAM для KV-cache
(зависит от длины контекста). Подробнее: [context-window.md](context-window.md).


### Scaling Laws

Kaplan et al. (2020) и Chinchilla (Hoffmann et al., 2022)
установили эмпирические законы масштабирования:

1. **Больше параметров + больше данных = лучше качество**
   (предсказуемо, по степенному закону)
2. **Оптимальное соотношение**: ~20 токенов на параметр
   (правило Chinchilla)
3. **Compute-optimal**: для фиксированного бюджета вычислений
   лучше обучить среднюю модель на большом объёме данных,
   чем огромную модель на малом

Пример: Llama 3.1 8B обучена на 15 триллионах токенов --
это значительно больше Chinchilla-оптимума (~160B токенов
для 8B модели), что даёт модели лучшее качество при малом
размере.

|--------|--------------|---------------------|------------------|--------------|
| 1-3B | 1-2 GB | 2-3 GB | 80-120 | Автодополнение, draft model |
| 7-8B | 4-5 GB | 6-8 GB | 45-60 | Универсальная, быстрая |
| 13B | 7-8 GB | 10-12 GB | 25-35 | Хороший баланс |
| 32-34B | 18-20 GB | 22-28 GB | 10-18 | Код, сложные задачи |
| 70B | 38-42 GB | 45-55 GB | 5-8 | Максимальное качество |

[!] Скорость генерации (tok/s) ограничена пропускной способностью
памяти: 256 GB/s. Подробнее: [generation.md](generation.md).


## Open-source vs проприетарные

### Проприетарные модели (API)

Доступны только через API, веса не опубликованы:

| Модель | Провайдер | Особенности |
|--------|-----------|-------------|
| GPT-4o | OpenAI | Мультимодальная, высокое качество |
| GPT-4o-mini | OpenAI | Быстрая и дешёвая |
| o1 / o3 | OpenAI | Reasoning-модели |
| Gemini Pro/Ultra | Google | Мультимодальная, длинный контекст |

Преимущества API:
- Не нужен мощный GPU
- Всегда актуальные модели
- Простая интеграция

Недостатки API:
- Стоимость (при большом объёме -- значительная)
- Зависимость от провайдера
- Данные уходят на внешние сервера
- Латентность сети
- Ограничения rate limits

### Open-source модели (локальный запуск)

Веса опубликованы, можно запускать на своём оборудовании:

Преимущества:
- Полный контроль над данными (приватность)
- Нет зависимости от интернета
- Нет recurring costs (только электричество и железо)
- Возможность fine-tuning
- Нет rate limits
- Нет цензуры провайдера

Недостатки:
- Нужен GPU с достаточным VRAM
- Качество ниже топовых проприетарных моделей
- Нужно самостоятельно обновлять модели
- Настройка inference требует знаний

### Сравнение стоимости

Пример: 1 миллион токенов в месяц (вход + выход).

| Вариант | Стоимость |
|---------|-----------|
| GPT-4o API | ~$25/месяц |
| GPT-4o-mini API | ~$1.5/месяц |
| Локальный 70B (Q4) | ~$5/месяц (электричество) |
| Локальный 8B (Q4) | ~$3/месяц (электричество) |

[!] Локальный запуск выгоднее при большом объёме, но требует
начальных вложений в оборудование.


### Гибридный подход

На практике часто используют комбинацию:

```
+---------------------------+     +---------------------------+
|   Локальный inference     |     |   API (проприетарные)     |
|                           |     |                           |
|   - Повседневные задачи   |     |   - Сложные задачи        |
|   - Чувствительные данные |     |   - Когда нужен лучший    |
|   - Автодополнение кода   |     |     результат             |
|   - Быстрые запросы       |     |   - Мультимодальность     |
|                           |     |   - Резервный вариант     |
+---------------------------+     +---------------------------+
```


## Связь с платформой

### Radeon 8060S -- что можно запускать

С 96 GiB unified VRAM и 256 GB/s bandwidth платформа позволяет:

1. **Любую модель до 70B** в квантизации Q4_K_M (~40 GB) с запасом
   VRAM для контекста 32K-128K
2. **Модели до 32B** в FP16 (~64 GB) без квантизации, с некоторым
   запасом для контекста
3. **MoE-модели**: DeepSeek-V3 671B не поместится (нужно ~350 GB
   в Q4), но дистиллированные версии (32B) -- без проблем
4. **Одновременный запуск** нескольких моделей: 8B для
   автодополнения + 32B для chat -- примерно 25 GB суммарно

### llama-server

Inference-сервер на базе llama.cpp, оптимизированный для
CPU и GPU (Vulkan, CUDA, ROCm, Metal).

```bash
# пример запуска 32B модели на Vulkan
llama-server \
    --model Qwen2.5-Coder-32B-Instruct-Q4_K_M.gguf \
    --gpu-layers 99 \       # все слои на GPU
    --ctx-size 32768 \      # контекст 32K токенов
    --port 8080 \
    --host 0.0.0.0
```

Подробнее о настройке: [Vulkan + llama.cpp](../inference/vulkan-llama-cpp.md).


## Ключевые термины

Краткий словарь терминов, используемых в этой и последующих статьях:

| Термин | Определение |
|--------|-------------|
| **Параметр (вес)** | Одно числовое значение в нейросети, выученное при обучении |
| **Токен** | Единица текста (слово, часть слова, символ), с которой работает модель |
| **Inference** | Процесс генерации текста обученной моделью |
| **Контекст** | Все токены (вход + выход), обрабатываемые за один сеанс |
| **Квантизация** | Уменьшение точности хранения весов (FP16 -> INT4) для экономии VRAM |
| **GGUF** | Формат файла модели для llama.cpp |
| **Base model** | Модель после предобучения (без instruct fine-tuning) |
| **Instruct model** | Модель, дообученная для следования инструкциям |
| **VRAM** | Видеопамять GPU, используемая для хранения модели и KV-cache |
| **RLHF** | Reinforcement Learning from Human Feedback -- метод alignment |
| **MoE** | Mixture of Experts -- архитектура с частичной активацией параметров |
| **FIM** | Fill-in-the-Middle -- режим заполнения пропусков в тексте |
| **Bandwidth** | Пропускная способность памяти GPU (GB/s), определяет tok/s |
| **KV-cache** | Кэш ключей и значений attention для уже обработанных токенов |
| **Prefill** | Фаза обработки промпта (параллельная, compute-bound) |
| **Decode** | Фаза генерации (последовательная, memory-bound) |
| **tok/s** | Токенов в секунду -- метрика скорости генерации |
| **TTFT** | Time to First Token -- задержка до первого токена ответа |


## Типичные заблуждения

### "LLM понимает текст"

LLM не "понимает" текст в человеческом смысле. Модель выучила
статистические закономерности: какие последовательности токенов
вероятны. Результат выглядит осмысленным, потому что обучающие
данные осмысленны. Но модель может генерировать правдоподобный,
но фактически неверный текст (hallucination).

### "Больше параметров = строго лучше"

Не всегда. Качество зависит от:
- Объёма и качества данных обучения (data > params при прочих равных)
- Метода обучения (SFT, RLHF, DPO)
- Архитектуры (MoE может быть лучше Dense при равном compute)
- Конкретной задачи (7B-модель для кода может быть лучше 70B общего назначения)

Qwen2.5-Coder-32B на задачах кодирования может превосходить
Llama 3.1 70B, несмотря на разницу в размере.

### "Open-source модели сильно хуже проприетарных"

Разрыв сокращается. DeepSeek-V3 (open-source, MIT) конкурирует
с GPT-4o на многих бенчмарках. Для локального inference
open-source модели -- единственный вариант.

### "Квантизация сильно портит качество"

Q4_K_M (4 бита на параметр) снижает качество на ~1-3% по бенчмаркам.
Для большинства практических задач разница незаметна. Q3 и ниже --
заметная деградация.

### "Длинный контекст -- всегда лучше"

Длинный контекст требует больше VRAM (KV-cache), замедляет
prefill и не гарантирует лучшего результата из-за "lost in the
middle" -- модели хуже работают с информацией в середине
длинного контекста.


## Бенчмарки и оценка моделей

### Популярные бенчмарки

| Бенчмарк | Задача | Что измеряет |
|----------|--------|-------------|
| MMLU | Выбор из вариантов (57 предметов) | Общие знания |
| HumanEval / MBPP | Генерация кода | Программирование |
| GSM8K | Математические задачи | Арифметика и логика |
| MATH | Сложные математические задачи | Продвинутая математика |
| HellaSwag | Продолжение предложений | Понимание контекста |
| ARC | Вопросы по естественным наукам | Рассуждения |
| TruthfulQA | Фактическая точность | Правдивость |
| MT-Bench | Многоходовой диалог | Качество chat |

### Ограничения бенчмарков

- Утечка данных: тестовые примеры могут попасть в обучающий набор
- Overfitting: модель может быть оптимизирована под бенчмарк
- Не отражают реальное использование: chat-качество, следование
  инструкциям, длинный контекст -- плохо измеряются стандартными
  бенчмарками
- Субъективность: для chat и creative задач нет объективных метрик

[!] Бенчмарки -- ориентир, не абсолютная истина. Лучший способ
оценки -- тестирование на своих задачах.
