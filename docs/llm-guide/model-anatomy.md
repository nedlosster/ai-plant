# Анатомия LLM: что внутри модели

## Что такое "параметр"

Параметр -- одно число (вес) в нейросети. Модель "7B" содержит ~7 миллиардов таких чисел. Каждое число хранится в формате FP16 или BF16 (2 байта).

Формула размера модели в памяти:

```
размер (байты) = количество_параметров * байт_на_параметр

7B * 2 байта (FP16) = 14 GiB
32B * 2 = 64 GiB
70B * 2 = 140 GiB
```

При квантизации байт на параметр уменьшается:

| Формат | Байт/параметр | 7B | 32B | 70B |
|--------|--------------|-----|------|------|
| FP16 / BF16 | 2.0 | 14 GiB | 64 GiB | 140 GiB |
| Q8_0 | ~1.1 | ~8 GiB | ~35 GiB | ~77 GiB |
| Q4_K_M | ~0.6 | ~4.5 GiB | ~20 GiB | ~42 GiB |

## Структура модели

LLM -- последовательность слоев (transformer blocks):

```
Вход (текст)
    |
[Tokenizer] -- текст -> токены -> числа (token IDs)
    |
[Embedding Layer] -- token IDs -> вектора (hidden states)
    |
[Transformer Block 1]
    |
[Transformer Block 2]
    |
    ...
    |
[Transformer Block N]
    |
[Output Head (LM Head)] -- вектора -> вероятности токенов
    |
Выход (следующий токен)
```

Основная масса параметров -- в transformer blocks. Их количество (N) -- ключевой фактор размера модели.

## Ключевые размерности

Каждая модель определяется набором гиперпараметров:

| Параметр | Обозначение | Описание |
|----------|-----------|----------|
| `n_layers` | L | Количество transformer blocks (глубина модели) |
| `hidden_dim` | d | Размерность скрытого состояния (ширина модели) |
| `n_heads` | h | Количество attention heads |
| `head_dim` | d_h | Размерность одной head (обычно d / h) |
| `n_kv_heads` | h_kv | Количество KV-голов (для GQA, <= n_heads) |
| `intermediate_size` | d_ff | Размерность FFN (обычно ~2.7 * d для SwiGLU) |
| `vocab_size` | V | Размер словаря токенов |

### Размерности реальных моделей

| Модель | n_layers | hidden_dim | n_heads | n_kv_heads | intermediate | vocab_size |
|--------|----------|-----------|---------|------------|-------------|-----------|
| Llama-3.1-8B | 32 | 4096 | 32 | 8 | 14336 | 128256 |
| Qwen2.5-7B | 28 | 3584 | 28 | 4 | 18944 | 151936 |
| Qwen2.5-14B | 48 | 5120 | 40 | 8 | 13824 | 151936 |
| Qwen2.5-32B | 64 | 5120 | 40 | 8 | 27648 | 151936 |
| Llama-3.1-70B | 80 | 8192 | 64 | 8 | 28672 | 128256 |
| Qwen2.5-72B | 80 | 8192 | 64 | 8 | 29568 | 151936 |

## Слои модели в деталях

### Embedding Layer

Преобразует token ID в вектор фиксированной длины (hidden_dim).

```
Параметры = vocab_size * hidden_dim

Llama-8B: 128256 * 4096 = 525M параметров (~1 GiB в FP16)
Qwen2.5-72B: 151936 * 8192 = 1.24B параметров (~2.5 GiB)
```

Каждому токену в словаре соответствует вектор из hidden_dim чисел. При инференсе: token ID -> lookup в таблице -> вектор.

### Transformer Block

Каждый block содержит два основных компонента:

```
Input
  |
[Layer Norm]
  |
[Self-Attention] -- "на что смотреть"
  |
[+ Residual]     -- добавляем вход обратно
  |
[Layer Norm]
  |
[MLP / FFN]      -- "что с этим делать"
  |
[+ Residual]
  |
Output
```

### Self-Attention (детально)

Четыре линейных проекции: Q (query), K (key), V (value), O (output).

```
Параметры Attention (один block) =
  Q: hidden_dim * hidden_dim     = d * d
  K: hidden_dim * head_dim * n_kv_heads  = d * d_h * h_kv
  V: hidden_dim * head_dim * n_kv_heads  = d * d_h * h_kv
  O: hidden_dim * hidden_dim     = d * d

Без GQA (n_kv_heads = n_heads):
  4 * d * d = 4 * d^2

С GQA (n_kv_heads < n_heads):
  Q: d^2, K: d * d_h * h_kv, V: d * d_h * h_kv, O: d^2
  = 2 * d^2 + 2 * d * d_h * h_kv
```

**Пример Llama-3.1-8B** (d=4096, h=32, h_kv=8, d_h=128):
```
Q: 4096 * 4096 = 16.8M
K: 4096 * 128 * 8 = 4.2M
V: 4096 * 128 * 8 = 4.2M
O: 4096 * 4096 = 16.8M
Итого: 42M на block
```

### GQA (Grouped Query Attention)

В стандартном Multi-Head Attention: Q, K, V имеют одинаковое число голов. KV-cache растет пропорционально n_heads.

GQA: Q имеет n_heads голов, но K и V делят головы между группами. Экономия KV-cache и параметров.

```
Стандартный MHA (n_heads=32, n_kv_heads=32):
  Каждая Q-голова имеет свою K и V голову
  KV-cache: 32 * 2 * head_dim * context_len

GQA (n_heads=32, n_kv_heads=8):
  4 Q-головы делят одну пару K/V
  KV-cache: 8 * 2 * head_dim * context_len
  Экономия: 4x по KV-cache
```

Все современные модели (Llama 3, Qwen2.5, Gemma 3) используют GQA.

### MLP / FFN (Feed-Forward Network)

Основная масса параметров модели. Современные LLM используют SwiGLU-вариант:

```
Три матрицы: gate_proj, up_proj, down_proj

gate_proj: hidden_dim * intermediate_size   = d * d_ff
up_proj:   hidden_dim * intermediate_size   = d * d_ff
down_proj: intermediate_size * hidden_dim   = d_ff * d

Итого: 3 * d * d_ff на block
```

**Пример Llama-3.1-8B** (d=4096, d_ff=14336):
```
3 * 4096 * 14336 = 176M параметров на block
```

Для сравнения: Attention ~42M, MLP ~176M. **MLP содержит ~80% параметров** каждого block.

### Output Head (LM Head)

Преобразует hidden state обратно в вероятности токенов:

```
Параметры = hidden_dim * vocab_size (часто shared с embedding)

Llama-8B: 4096 * 128256 = 525M (shared с embedding -- не считается дважды)
```

## Подсчет параметров модели

Общая формула (приблизительная, SwiGLU + GQA):

```
Total ~ vocab_size * hidden_dim                        # embedding
      + n_layers * (
          2 * d^2 + 2 * d * d_h * n_kv_heads          # attention (Q,K,V,O)
        + 3 * d * d_ff                                  # MLP (gate, up, down)
        + 2 * d                                         # layer norms
      )
      + d                                               # final norm
```

**Проверка для Llama-3.1-8B** (L=32, d=4096, h=32, h_kv=8, d_h=128, d_ff=14336, V=128256):
```
Embedding: 128256 * 4096 = 525M
Per block:
  Attention: 2 * 4096^2 + 2 * 4096 * 128 * 8 = 33.6M + 8.4M = 42M
  MLP: 3 * 4096 * 14336 = 176M
  Norms: 2 * 4096 = 8K (пренебрежимо)
  Block total: ~218M
32 blocks: 32 * 218M = 6,976M = ~7B
+ Embedding: 525M
Total: ~7.5B
```

Близко к заявленным 8B (разница -- дополнительные компоненты и rounding).

## Что определяет качество

### Глубина (n_layers)

Больше слоев = больше последовательных преобразований = более сложные рассуждения.

| n_layers | Модели | Характер |
|----------|--------|----------|
| 28-32 | 7-8B | Простые задачи, chat, быстрый отклик |
| 48-64 | 14-32B | Сложные задачи, кодинг, reasoning |
| 80 | 70-72B | Максимальная глубина рассуждений |

### Ширина (hidden_dim)

Больше hidden_dim = более богатое представление каждого токена.

| hidden_dim | Модели | Характер |
|-----------|--------|----------|
| 3584-4096 | 7-8B | Базовое представление |
| 5120 | 14-32B | Среднее |
| 8192 | 70-72B | Максимальное |

### Данные обучения

При одинаковой архитектуре качество определяется объемом и качеством обучающих данных:
- Llama 3.1: 15T токенов
- Qwen2.5: 18T токенов
- DeepSeek-V3: 14.8T токенов

Больше данных -- лучше, но с diminishing returns после определенного объема.

## Scaling Laws

Эмпирическое наблюдение (Chinchilla scaling): качество модели предсказуемо растет с увеличением:
- Количества параметров
- Объема обучающих данных
- Compute (GPU-часов)

Примерные соотношения:
- 2x параметров при том же объеме данных -> ~+5% MMLU
- 2x данных при тех же параметрах -> ~+3% MMLU
- Реальный прирост нелинейный и зависит от задачи

## Практическое значение

### Как выбирать размер модели

| Задача | Минимум | Рекомендация |
|--------|---------|-------------|
| Простой chat | 7B | 7-14B |
| Кодинг | 14B | 32B |
| Reasoning, математика | 14B | 32B (R1-Distill, QwQ) |
| Сложный анализ | 32B | 70B |
| Перевод, суммаризация | 7B | 14-32B |
| RAG | 7B | 14B |

### Компромисс: размер vs скорость

На данной платформе (256 GB/s bandwidth):

| Размер (Q4) | VRAM | tok/s (tg) | Характер |
|-------------|------|------------|----------|
| 7B (~5 GiB) | 5 GiB | ~50 | Мгновенный отклик |
| 14B (~9 GiB) | 9 GiB | ~28 | Быстрый |
| 32B (~20 GiB) | 20 GiB | ~13 | Комфортный |
| 70B (~42 GiB) | 42 GiB | ~6 | Заметная задержка |

Для интерактивной работы (chat, IDE) комфортно при >10 tok/s. Для batch-задач скорость менее критична.
