# Сколько знаний помещается в модель

Парадокс: файл размером 45 GiB "знает" больше, чем 28 TB обучающих данных. Как это возможно, что теряется при сжатии, и почему квантизация Q4 сохраняет 96% качества.

## Парадокс сжатия

Представь библиотеку. В ней 10 миллионов книг -- это примерно 60 TB текста. Ты читаешь их все. Потом тебе дают 45 GiB флешку и просят записать на неё **всё что ты узнал**. Невозможно? Библиотека в 1300 раз больше флешки.

Но человеческий мозг делает это постоянно. За жизнь мозг обрабатывает петабайты сенсорных данных, но вся долговременная память занимает по разным оценкам 1-10 ТБ. Степень сжатия: 100-1000x. Мозг не помнит каждую страницу каждой книги -- он помнит **идеи, паттерны, связи**.

LLM работает так же. Qwen3-Coder Next -- файл 45 GiB -- обучена на ~7 триллионах токенов (~28 TB текста). Степень сжатия ~600x. Модель не хранит текст -- она хранит **статистические паттерны**, из которых текст можно регенерировать.

И вот парадокс: модель может **сгенерировать** правильный ответ на вопрос, ответа на который в обучающих данных **не было**. Потому что она выучила не факты, а **правила**. Формула `y = 2x + 3` занимает 10 байт, но описывает бесконечное множество пар (x, y). Модель -- это миллиарды таких "формул", каждая из которых кодирует фрагмент знания о мире.

## Что хранится в весах

### Нейрон как единица знания

Каждый параметр модели -- это число с плавающей запятой (FP16: 16 бит, BF16: 16 бит). 70 миллиардов таких чисел = 140 GiB в FP16. Но отдельный параметр **ничего не значит** сам по себе -- значение имеют **паттерны** связей между параметрами.

Аналогия: один нейрон мозга не "знает" ничего. Знание закодировано в **силе связей** (синапсов) между нейронами. В LLM параметр -- это сила связи между "виртуальными нейронами".

### Слои знания

Исследования (probing experiments) показывают, что разные слои transformer кодируют разные уровни абстракции:

| Слои | Что кодируют | Пример |
|------|-------------|--------|
| Первые (1-10) | Синтаксис, токенизация | "def" → начало функции Python |
| Средние (10-40) | Семантика, отношения | "sort" связан с "array", "compare", "O(n log n)" |
| Глубокие (40-60+) | Reasoning, логика, планирование | "если массив почти отсортирован → insertion sort лучше quicksort" |
| Attention heads | Контекстные связи | "this" → на что ссылается в данном предложении |

Модель с 60 слоями -- это 60 уровней абстракции, каждый из которых "видит" данные на разном масштабе. Первые слои видят символы, последние -- идеи.

### Что модель "помнит" хорошо

**Высокочастотные паттерны** -- то, что встречалось тысячи раз в training data:

- Грамматика и синтаксис всех языков программирования (Python, JS, Rust, Go...)
- Стандартные алгоритмы и структуры данных
- API популярных библиотек (React, Django, pandas, torch)
- Архитектурные паттерны (MVC, REST, microservices)
- Идиомы и best practices ("используй context manager для файлов")
- Стиль и тон (технический, разговорный, юридический)
- Связи между концепциями ("Docker" → "контейнер" → "image" → "Dockerfile")

**Среднечастотные** -- встречалось сотни раз:

- Специфичные библиотеки (FastAPI middleware, SQLAlchemy events)
- Редкие языки программирования (Elixir, Nim, Zig)
- Domain-specific знания (финансовые формулы, медицинская терминология)

### Что модель "забывает"

**Низкочастотные факты** -- встречалось <10 раз в training data:

- Точные URL, email-адреса, телефоны
- Конкретные даты событий (если не общеизвестные)
- Содержание конкретного файла конкретного проекта
- Редкие имена, названия, идентификаторы
- Численные результаты конкретных экспериментов

**Обобщающее правило**: если факт можно **вывести из паттерна** -- модель его "знает". Если факт **уникален** и не укладывается в паттерн -- модель его "забыла" или галлюцинирует.

## Информационная плотность: биты на параметр

### Теоретическая оценка

Каждый параметр FP16 = 16 бит. Но сколько из них несут **полезную информацию**?

Эмпирическое наблюдение: квантизация показывает ответ:

| Квант | Бит/параметр | Качество (% от FP16) | Вывод |
|-------|-------------|----------------------|-------|
| FP16 | 16 | 100% (baseline) | 14 бит -- redundancy |
| Q8_0 | 8 | ~99.5% | 8 бит избыточны |
| Q6_K | 6 | ~99% | 10 бит избыточны |
| **Q4_K_M** | 4 | **~96%** | **12 бит избыточны** |
| Q3_K_M | 3 | ~92% | Начинается потеря |
| Q2_K | 2 | ~85% | Заметная деградация |
| IQ1_S | 1.5 | ~70% | Сильная деградация |

**Вывод**: полезная информация на параметр -- **~2-3 бита**. Всё выше 4 бит -- redundancy, которая помогает при обучении (gradient flow), но не нужна при inference.

### Формула полезной информации

```
Полезная информация ≈ N_параметров * 2.5 бит (среднее Q3-Q4 boundary)

Примеры (наши модели):

Qwen3-Coder Next 80B:
  80 * 10^9 * 2.5 бит = 200 Гбит = 25 GiB полезной информации
  Файл Q4_K_M: 45 GiB (overhead на метаданные, квантизационные таблицы, KV-heads)

Qwen3-Coder 30B-A3B:
  30 * 10^9 * 2.5 бит = 75 Гбит = ~9.4 GiB полезной информации
  Файл Q4_K_M: 17 GiB

Qwen2.5-Coder 1.5B:
  1.5 * 10^9 * 2.5 бит = 3.75 Гбит = ~0.47 GiB полезной информации
  Файл Q8_0: 1.5 GiB
```

Файл всегда больше "полезной информации" из-за квантизационных таблиц, special tokens, embedding matrices (которые менее compressible).

### Закон масштабирования (scaling law)

Chinchilla scaling law (2022, DeepMind) показывает оптимальное соотношение:

```
Оптимальные tokens для обучения ≈ 20 * N_параметров

70B модель: 70B * 20 = 1.4T tokens (оптимум)
Фактически: 15T tokens (10x overtraining -- улучшает inference quality)
```

**Overtraining** (больше данных, чем оптимум Chinchilla) -- стандартная практика 2025-2026. Qwen3-Coder Next обучена на 7T tokens при оптимуме ~1.6T -- 4x overtraining. Это "впрессовывает" больше знаний в те же параметры за счёт более длительного обучения.

## MoE: та же информация, быстрее доступ

Mixture of Experts (MoE) не увеличивает **объём знаний** линейно с total параметрами. 80B MoE с 3B active -- это не "80B знаний", а "80B знаний распределённых по 384 экспертам, из которых 8+1 активны на каждый токен".

```
Dense 30B:  30B параметров, все читаются на каждый токен
MoE 80B/3B: 80B параметров, но ~3B читаются на каждый токен

Знаний в MoE 80B > Dense 30B (больше параметров = больше "формул")
Но bandwidth-cost MoE 80B ≈ Dense 3B (читается только 3B)
```

Это объясняет, почему MoE 80B-A3B даёт 53 tok/s на нашей платформе (bandwidth 256 GB/s), а Dense 30B -- только 12.6 tok/s: MoE читает в 10x меньше данных из памяти при каждой генерации.

**Специализация экспертов**: исследования показывают, что разные эксперты "специализируются" на разных доменах. Один эксперт может быть "экспертом по Python", другой -- "по математике", третий -- "по естественному языку". Router (маршрутизатор) выбирает нужных экспертов для каждого токена.

## Что это значит для практики

### 1. Больше модель ≠ линейно больше знаний

Переход 7B → 70B (10x параметров) не даёт 10x знаний. Scaling law показывает **логарифмическую** зависимость качества от размера. Каждое удвоение параметров даёт фиксированное (не удвоенное) улучшение.

На практике: Qwen3-Coder 30B-A3B (30B) решает ~62% SWE-bench, а Qwen3-Coder Next 80B-A3B (80B, 2.7x больше) решает 70.6% -- прирост 8.6 пунктов, не 2.7x. Diminishing returns.

### 2. Квантизация -- бесплатное сжатие (до предела)

Q4_K_M убирает 75% данных (16 → 4 бита) при потере 4% качества. Это работает, потому что 75% данных были **redundancy**, не информацией. Но Q2_K убирает 87.5% -- и потеря уже 15%: начали резать полезную информацию.

**Правило**: Q4_K_M -- sweet spot для inference. Q6_K_XL -- если VRAM позволяет и хочется на 1-2% лучше. Ниже Q3 -- только если модель иначе не влезает.

### 3. Контекст важнее размера модели для специфичных знаний

Модель 80B "знает" общие паттерны программирования, но **не знает** ваш конкретный проект. Для конкретики нужен **контекст** (256K tokens у Qwen3-Coder Next).

```
Знание = веса (общие паттерны) + контекст (конкретные факты)

"Напиши функцию сортировки" → достаточно весов
"Исправь баг в моём проекте" → нужен контекст (файлы проекта)
```

Это объясняет, почему [opencode](../ai-agents/agents/opencode.md) и [Cline](../ai-agents/agents/cline.md) грузят файлы в контекст -- модель применяет **знания из весов** к **фактам из контекста**.

### 4. Fine-tune -- это "дозапись" знаний

Fine-tune (LoRA, QLoRA) модифицирует часть весов, "записывая" новые паттерны. Пример: [TeichAI/Qwen3-14B-Claude-4.5-Opus-Reasoning-Distill](../models/families/qwen35.md) -- 250 samples из Claude Opus 4.5, $52 на обучение. Результат: 14B модель с улучшенным reasoning.

Но 250 samples = ~1M tokens ≈ несколько килобайт полезной информации. Это не "знания Claude Opus" -- это **стиль мышления** (structured `<think>` blocks, step-by-step planning). Fine-tune меняет **как** модель думает, не **что** она знает.

### 5. Почему 1.5B модель полезна для FIM

Qwen2.5-Coder 1.5B (1.5 GiB Q8) содержит ~0.5 GiB полезной информации. Немного? Для FIM (Fill-in-the-Middle) достаточно: модель должна знать **синтаксис** и **типичные паттерны завершения**, а не всю Computer Science. Это как автодополнение в IDE -- не нужен PhD, нужно знать API.

## Пределы: сколько знаний вообще можно впихнуть

### Теоретический предел

Информационная ёмкость модели ограничена числом параметров и точностью:

```
Верхний предел = N_параметров * log2(уровней_точности) бит

FP16: 70B * 16 = 1120 Гбит = 140 GiB (но реально ~17.5 GiB полезных)
INT4:  70B * 4  = 280 Гбит  = 35 GiB (но реально ~17.5 GiB полезных)

Вывод: точность хранения НЕ лимитирует полезную информацию.
Лимитирует АРХИТЕКТУРА и ОБУЧЕНИЕ.
```

### Что лимитирует на самом деле

1. **Архитектура** -- количество и размер attention heads, глубина, width. Transformer определённого размера может "выучить" ограниченное число паттернов.

2. **Обучение** -- даже с бесконечными данными, SGD/Adam не могут "впрессовать" бесконечно много информации. Loss plateau -- это предел текущей архитектуры.

3. **Интерференция** -- новые знания могут "стирать" старые (catastrophic forgetting). Fine-tune на одном домене ухудшает другой.

4. **Линейная суперпозиция** -- гипотеза Anthropic (Superposition hypothesis): нейроны кодируют **больше features**, чем у них dimensions, через суперпозицию. Это увеличивает ёмкость, но создаёт "полтергейст"-эффекты (hallucinations, когда суперпозиция распадается неправильно).

### Текущий frontier

По оценкам исследователей (~2025-2026), frontier-модели работают на 10-30% от теоретической информационной ёмкости архитектуры. Остальное -- dead neurons, redundant representations, training artifacts. Области улучшения:
- Pruning (удаление "мёртвых" параметров)
- Distillation (сжатие знаний из большой модели в маленькую)
- MoE (специализация параметров)
- Sparse attention (более эффективное использование attention heads)

## Аналогия с человеческой памятью

| Аспект | Человек | LLM |
|--------|---------|-----|
| "Долговременная память" | Синапсы мозга (~1-10 TB) | Веса модели (17-38 GiB полезных) |
| "Оперативная память" | Рабочая память (~7 элементов) | Контекст (256K tokens ≈ 500 страниц) |
| Обучение | Годы опыта, чтение | Training (недели GPU-кластера) |
| "Дозапись" | Новый опыт, изучение | Fine-tune (LoRA, QLoRA) |
| Забывание | Естественное угасание | Catastrophic forgetting при fine-tune |
| Галлюцинации | Ложные воспоминания, confabulation | Генерация несуществующих фактов |
| Reasoning | Сознательное мышление | Chain-of-thought, `<think>` блоки |
| Интуиция | Быстрые ассоциации | Прямая генерация без CoT |

Главное различие: человек **понимает** то, что знает. LLM **воспроизводит паттерны** без понимания. Это дискуссионный вопрос в философии AI (Chinese Room, Understanding vs Simulation), но для практических целей различие несущественно -- модель, которая "симулирует" понимание кода на 70.6% SWE-bench, решает реальные задачи.

## Практические выводы

1. **Q4_K_M -- оптимальный квант** для inference: убирает redundancy, сохраняет информацию. Не стоит платить VRAM за FP16, если не fine-tune.

2. **Контекст (256K) важнее размера модели** для задач, требующих конкретных знаний о вашем проекте. 30B с полным контекстом проекта > 122B без контекста.

3. **MoE -- архитектурный хак** для нашей платформы: 80B знаний при 3B bandwidth-cost. На 256 GB/s -- это решающее преимущество.

4. **Fine-tune меняет "как", не "что"**: 250 reasoning-трассов не добавят модели знаний, но изменят стиль мышления.

5. **Маленькие модели (1.5B) полезны** для узких задач (FIM): им не нужны все знания, только специализированные паттерны.
