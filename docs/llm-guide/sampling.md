# Сэмплирование

Сэмплирование (sampling) -- процесс выбора следующего токена
из распределения вероятностей, вычисленного моделью. На каждом
шаге генерации модель выдаёт вектор logits (по одному числу на
каждый токен словаря), из которого нужно выбрать один токен.
Параметры сэмплирования определяют баланс между
детерминированностью и разнообразием текста.

## Logits и Softmax

### Logits

На последнем шаге forward pass модель вычисляет вектор logits
размерности vocab_size (32K-152K). Каждый элемент -- "сырая"
оценка для соответствующего токена.

```
LM Head: [d_model] * W_lm --> logits [vocab_size]

Пример (vocab_size = 5 для простоты):

  Токен       | ID | Logit
  ------------|----|---------
  "Париж"     |  0 |  4.5
  "город"     |  1 |  2.1
  "Лион"      |  2 |  1.3
  "столица"   |  3 |  0.8
  "кот"       |  4 | -3.2
```

Logits -- не вероятности. Они могут быть отрицательными и не
суммируются в 1.

### Softmax

Softmax преобразует logits в вероятности:

```
                   exp(logit_i)
P(token_i) = -------------------------
             sum(exp(logit_j)) для всех j
```

Свойства:
- Все значения в диапазоне (0, 1)
- Сумма = 1
- Сохраняет порядок (больший logit -> большая вероятность)
- Экспоненциальная функция усиливает разницу

```
Logits -->  Softmax --> Вероятности

  4.5        exp(4.5) = 90.0       90.0/102.1 = 0.881
  2.1        exp(2.1) =  8.2        8.2/102.1 = 0.080
  1.3        exp(1.3) =  3.7        3.7/102.1 = 0.036
  0.8        exp(0.8) =  2.2        2.2/102.1 = 0.022
 -3.2        exp(-3.2)=  0.04       0.04/102.1= 0.0004
                        ------
                        102.14                   1.000
```

Результат: "Париж" с вероятностью 88.1%, "город" -- 8%,
остальные -- менее 4%.


### Визуализация распределения

```
Вероятность
  |
  |  #
  |  #
  |  #
  |  #
  |  #
  |  #
  |  #
  |  #  #
  |  #  #  #
  |  #  #  #  #
  +--+--+--+--+----> Токены (отсортированы по вероятности)
     1  2  3  4

  1: "Париж"   (0.881)
  2: "город"   (0.080)
  3: "Лион"    (0.036)
  4: "столица" (0.022)
```


## Greedy Decoding

Простейшая стратегия: всегда выбирать токен с наибольшей
вероятностью.

```
token = argmax(logits)
```

### Свойства

| Свойство | Значение |
|----------|----------|
| Детерминированность | Полная (одинаковый вход -> одинаковый выход) |
| Качество | Обычно хорошее для фактических задач |
| Разнообразие | Нулевое |
| Скорость | Максимальная (нет дополнительных вычислений) |

### Когда использовать

- Задачи с единственным правильным ответом
- Код (синтаксис детерминирован)
- Извлечение фактов
- Структурированный вывод (JSON)
- Классификация

### Проблемы Greedy

1. **Повторения**: модель часто зацикливается на одних и тех же фразах

```
Greedy output:
"Это очень важно. Это очень важно. Это очень важно.
 Это очень важно. Это очень важно."
```

2. **Неоптимальность**: лучшая последовательность может начинаться
   не с самого вероятного первого токена

```
Вероятности:
  P("Париж -- столица Франции") = 0.0030
  P("Столица Франции -- Париж") = 0.0045  <-- лучше

Greedy выберет "Париж" (0.88 > 0.08), хотя
"Столица" в итоге даёт более вероятную последовательность
```

3. **Скучность**: всегда предсказуемый, "безопасный" текст

В llama-server greedy decoding -- это `--temp 0`.


## Temperature

Temperature (T) -- масштабирование logits перед softmax.
Управляет "уверенностью" распределения.

### Формула

```
                      exp(logit_i / T)
P(token_i) = ------------------------------------
             sum(exp(logit_j / T)) для всех j
```

### Влияние температуры

```
T < 1:  logits масштабируются ВВЕРХ -> распределение ОСТРЕЕ
        (уверенные предсказания)

T = 1:  без изменений (стандартное поведение)

T > 1:  logits масштабируются ВНИЗ -> распределение РОВНЕЕ
        (менее уверенные предсказания)

T -> 0: greedy decoding (всегда top-1 токен)
T -> inf: равномерное распределение (случайный токен)
```

### Пример с числами

Исходные logits: [4.5, 2.1, 1.3, 0.8, -3.2]

```
Temperature | logits / T           | Вероятности
------------|----------------------|---------------------------
T = 0.3     | [15.0, 7.0, 4.3,    | [0.997, 0.003, 0.000,
            |  2.7, -10.7]        |  0.000, 0.000]
            |                      | --> почти greedy

T = 0.7     | [6.4, 3.0, 1.9,     | [0.953, 0.031, 0.010,
            |  1.1, -4.6]         |  0.005, 0.000]
            |                      | --> уверенно, но не greedy

T = 1.0     | [4.5, 2.1, 1.3,     | [0.881, 0.080, 0.036,
            |  0.8, -3.2]         |  0.022, 0.000]
            |                      | --> стандартное

T = 1.5     | [3.0, 1.4, 0.87,    | [0.688, 0.139, 0.082,
            |  0.53, -2.1]        |  0.058, 0.004]
            |                      | --> разнообразнее

T = 2.0     | [2.25, 1.05, 0.65,  | [0.530, 0.160, 0.107,
            |  0.4, -1.6]         |  0.083, 0.011]
            |                      | --> ещё ровнее
```

### Визуализация

```
T=0.3 (почти greedy):        T=1.0 (стандарт):

  P |  #                       P |  #
    |  #                         |  #
    |  #                         |  #
    |  #                         |  #
    |  #                         |  #  #
    |  #                         |  #  #  #
    +--+--+--+--+-->             +--+--+--+--+-->


T=0.7 (баланс):              T=2.0 (ровное):

  P |  #                       P |  #
    |  #                         |  #  #
    |  #                         |  #  #  #
    |  #                         |  #  #  #  #
    |  #  #                      |  #  #  #  #
    |  #  #  #                   |  #  #  #  #  #
    +--+--+--+--+-->             +--+--+--+--+-->
```


### Практический эффект

```
T=0 (greedy):
  "Контекстное окно определяет максимальное количество токенов,
   которые модель может обработать за один проход."
  --> Сухой, предсказуемый, фактически верный

T=0.7:
  "Контекстное окно -- это, по сути, рабочая память модели.
   Оно определяет, сколько токенов модель способна учитывать
   при генерации ответа."
  --> Естественный, разнообразный, информативный

T=1.2:
  "Контекстное окно можно представить как окно наблюдения,
   через которое модель смотрит на мир токенов. Чем шире
   окно, тем больший фрагмент текста попадает в поле зрения."
  --> Образный, менее предсказуемый, может быть менее точным

T=2.0:
  "Контекстное окно -- своеобразный калейдоскоп токенных
   последовательностей, причудливым образом сочетающий
   ограничения памяти с вычислительными возможностями."
  --> Хаотичный, неточный, иногда бессмысленный
```


## Top-K

Top-K фильтрует распределение: оставляет только K токенов
с наибольшей вероятностью, остальные получают вероятность 0.

### Алгоритм

```
1. Отсортировать токены по logit (от большего к меньшему)
2. Оставить первые K
3. Занулить вероятности остальных
4. Перенормировать (чтобы сумма = 1)
5. Сэмплировать из усечённого распределения
```

### Пример

```
Top-K = 3:

  Токен    | Logit | P (до)  | P (после Top-K)
  ---------|-------|---------|----------------
  "Париж"  | 4.5   | 0.881   | 0.881/0.997 = 0.884
  "город"  | 2.1   | 0.080   | 0.080/0.997 = 0.080
  "Лион"   | 1.3   | 0.036   | 0.036/0.997 = 0.036
  "столица"| 0.8   | 0.022   | 0.000  <-- отсечён
  "кот"    | -3.2  | 0.000   | 0.000  <-- отсечён
```

### Типичные значения

| K | Эффект | Применение |
|---|--------|-----------|
| 1 | Greedy decoding | Код, факты |
| 10 | Консервативно | Техническое письмо |
| 40-50 | Стандарт | Chat, общие задачи |
| 100+ | Широкий выбор | Creative writing |

### Проблема Top-K

Top-K -- фиксированный порог. Он не учитывает форму распределения:

```
Ситуация 1: Уверенная модель
  Токен А: 0.90
  Токен Б: 0.05
  Токен В: 0.03
  ... (остальные < 0.01)

  Top-K=40: оставляет 40 токенов, включая шум
  Лучше было бы оставить 3

Ситуация 2: Неуверенная модель
  Токен А: 0.05
  Токен Б: 0.04
  Токен В: 0.04
  Токен Г: 0.03
  ... (много токенов с ~0.01-0.03)

  Top-K=10: обрезает потенциально хорошие варианты
  Лучше было бы оставить 50
```

Top-P решает эту проблему динамическим порогом.


## Top-P (Nucleus Sampling)

Top-P (nucleus sampling) -- динамическая фильтрация: оставляет
минимальный набор токенов, чья суммарная вероятность >= P.

### Алгоритм

```
1. Отсортировать токены по вероятности (от большей к меньшей)
2. Накапливать вероятности: cum_prob += P(token_i)
3. Остановиться, когда cum_prob >= P
4. Оставить только включённые токены
5. Перенормировать и сэмплировать
```

### Пример

```
Top-P = 0.9:

  Токен    | P      | Cum P  | Включён?
  ---------|--------|--------|----------
  "Париж"  | 0.881  | 0.881  | Да
  "город"  | 0.080  | 0.961  | Да  (0.961 >= 0.9 -- порог достигнут)
  "Лион"   | 0.036  | 0.997  | Нет
  "столица"| 0.022  | 1.000  | Нет
  "кот"    | 0.000  | 1.000  | Нет

Результат: 2 токена ("Париж" и "город")
```

### Адаптивность Top-P

```
Уверенная модель (Top-P = 0.9):
  "Париж"  0.90 | cum = 0.90 >= 0.9 --> 1 токен
  Эффективный K = 1 (почти greedy)

Неуверенная модель (Top-P = 0.9):
  "А" 0.05 | cum = 0.05
  "Б" 0.04 | cum = 0.09
  "В" 0.04 | cum = 0.13
  ...
  "Щ" 0.02 | cum = 0.91 >= 0.9 --> ~30 токенов
  Эффективный K = 30 (широкий выбор)
```

[!] Top-P автоматически адаптируется к уверенности модели.
Это основное преимущество перед фиксированным Top-K.


### Типичные значения

| P | Эффект | Применение |
|---|--------|-----------|
| 0.1 | Очень консервативно | Факты, extraction |
| 0.5 | Консервативно | Техническое письмо |
| 0.9 | Стандарт | Chat, общие задачи |
| 0.95 | Широкий | Creative writing |
| 1.0 | Без фильтрации | Полное распределение |


## Min-P

Min-P -- относительный порог: отсекает токены, чья вероятность
меньше `min_p * P(top_token)`.

### Алгоритм

```
1. Найти top_token -- токен с максимальной вероятностью
2. threshold = min_p * P(top_token)
3. Отсечь все токены с P < threshold
4. Перенормировать и сэмплировать
```

### Пример

```
Min-P = 0.05:

  top_token = "Париж" (P = 0.881)
  threshold = 0.05 * 0.881 = 0.044

  Токен    | P      | >= 0.044? | Включён?
  ---------|--------|-----------|----------
  "Париж"  | 0.881  | Да        | Да
  "город"  | 0.080  | Да        | Да
  "Лион"   | 0.036  | Нет       | Нет
  "столица"| 0.022  | Нет       | Нет
```

### Сравнение с Top-K и Top-P

```
Уверенная модель:
  Top token: P = 0.95

  Top-K=40:  оставляет 40 токенов (много шума)
  Top-P=0.9: оставляет ~1 токен
  Min-P=0.1: оставляет ~1 токен (threshold = 0.095)

Неуверенная модель:
  Top token: P = 0.03

  Top-K=40:  оставляет 40 токенов (может быть мало)
  Top-P=0.9: оставляет ~40 токенов
  Min-P=0.1: threshold = 0.003, оставляет ~100 токенов
```

### Типичные значения

| min_p | Эффект |
|-------|--------|
| 0.01 | Мягкая фильтрация |
| 0.05 | Стандарт |
| 0.1 | Умеренная фильтрация |
| 0.2 | Жёсткая фильтрация |

[!] Min-P -- относительно новый метод. Считается более
стабильным, чем Top-P, особенно при высокой temperature.
Поддерживается в llama.cpp и llama-server.


## Repetition Penalty

Repetition penalty штрафует токены, которые уже встречались
в контексте, снижая вероятность повторений.

### Алгоритм

```
Для каждого токена, уже встречавшегося в контексте:
  if logit > 0:
      logit = logit / penalty
  else:
      logit = logit * penalty

penalty > 1: снижает вероятность повторения
penalty = 1: без изменений
penalty < 1: увеличивает вероятность повторения (необычно)
```

### Пример

```
Repetition penalty = 1.2

Контекст уже содержит: "Париж", "город"

Исходные logits:
  "Париж"   4.5  --> 4.5 / 1.2 = 3.75  (штраф)
  "город"   2.1  --> 2.1 / 1.2 = 1.75  (штраф)
  "Лион"    1.3  --> 1.3                 (без изменений)
  "столица" 0.8  --> 0.8                 (без изменений)
  "кот"    -3.2  --> -3.2 * 1.2 = -3.84 (ещё сильнее штраф)
```

### Типичные значения

| Penalty | Эффект | Применение |
|---------|--------|-----------|
| 1.0 | Без штрафа | По умолчанию |
| 1.05 | Минимальный | Код (повторения допустимы) |
| 1.1 | Лёгкий | Chat (стандарт) |
| 1.2 | Умеренный | Длинные тексты |
| 1.3 | Заметный | Борьба с зацикливанием |
| 1.5+ | Сильный | Побочные эффекты (нарушение связности) |

### Проблемы высокого penalty

```
Penalty = 1.5:

"Солнце светило ярко. Небо было голубым. Птицы пели.
 Деревья стояли зелёные. Ветер дул. Облака плыли."

--> Модель избегает повторения ЛЮБЫХ слов, включая артикли
    и служебные. Результат: неестественный текст, потеря
    когерентности.
```

### Repetition penalty range

Некоторые реализации позволяют указать "окно" -- сколько
последних токенов учитывать для penalty:

```bash
# llama-server
--repeat-last-n 256    # штрафовать за повторения в последних 256 токенах
--repeat-penalty 1.1   # коэффициент штрафа
```

`--repeat-last-n 0` -- штраф по всему контексту.
`--repeat-last-n -1` -- то же, что 0 (весь контекст).
`--repeat-last-n 64` -- только последние 64 токена.


## Frequency и Presence Penalty

Альтернатива repetition penalty, используемая в OpenAI API
и поддерживаемая llama-server.

### Frequency Penalty

Штраф пропорционален количеству вхождений токена:

```
logit_new = logit - frequency_penalty * count(token)

count(token) = сколько раз токен встречался
```

Пример:
```
frequency_penalty = 0.5
Токен "и" встречался 3 раза: logit -= 0.5 * 3 = 1.5
Токен "также" встречался 1 раз: logit -= 0.5 * 1 = 0.5
```

### Presence Penalty

Штраф за само наличие токена (бинарный -- есть или нет):

```
logit_new = logit - presence_penalty * (1 if token in context else 0)
```

Пример:
```
presence_penalty = 0.6
Токен "и" есть в контексте:     logit -= 0.6
Токен "также" есть в контексте: logit -= 0.6  (одинаково)
Токен "однако" нет в контексте: logit += 0    (без штрафа)
```

### Сравнение

| Параметр | Формула | Эффект |
|----------|---------|--------|
| repetition_penalty | logit / penalty | Мультипликативный |
| frequency_penalty | logit - fp * count | Аддитивный, пропорционален частоте |
| presence_penalty | logit - pp * has | Аддитивный, бинарный |

[!] Не используйте repetition_penalty и frequency/presence penalty
одновременно -- они могут конфликтовать.


## Beam Search

Beam search -- алгоритм поиска наиболее вероятной
последовательности. В отличие от greedy (один путь),
beam search ведёт несколько параллельных "лучей" (beams).

### Алгоритм

```
beam_width = 3 (количество параллельных гипотез)

Шаг 1: Из промпта генерируем 3 самых вероятных первых токена
  Beam 1: "Париж"    (log_prob = -0.13)
  Beam 2: "Столица"  (log_prob = -2.53)
  Beam 3: "Город"    (log_prob = -3.32)

Шаг 2: Для каждого beam генерируем 3 продолжения
  Beam 1 + "—":       log_prob = -0.13 + (-0.52) = -0.65
  Beam 1 + "является": log_prob = -0.13 + (-1.20) = -1.33
  Beam 2 + "Франции":  log_prob = -2.53 + (-0.10) = -2.63
  Beam 1 + ",":        log_prob = -0.13 + (-1.80) = -1.93
  ... (всего 9 вариантов, оставляем 3 лучших)

  Beam 1: "Париж —"         (-0.65)
  Beam 2: "Париж является"  (-1.33)
  Beam 3: "Столица Франции" (-2.63)

Шаг 3: Продолжаем до EOS...
```

### Beam search vs Sampling

| Свойство | Beam search | Sampling (T + Top-P) |
|----------|-------------|---------------------|
| Цель | Найти наиболее вероятную последовательность | Сэмплировать разнообразные продолжения |
| Детерминированность | Детерминированный | Стохастический |
| Разнообразие | Низкое | Высокое |
| Качество для перевода | Высокое | Среднее |
| Качество для chat | Низкое (скучное) | Высокое |
| Вычисления | beam_width * decode | 1 * decode |
| VRAM | beam_width * KV-cache | 1 * KV-cache |

### Когда использовать

```
Beam search:
  - Машинный перевод
  - Суммаризация (когда нужна точность)
  - Задачи с единственным "правильным" ответом
  - НЕ для chat / creative writing

Sampling (temperature + top-p):
  - Chat
  - Creative writing
  - Код (с низкой temperature)
  - Все задачи, где важна естественность
```

[!] Большинство современных LLM-приложений используют sampling,
а не beam search. Beam search актуален в основном для
seq2seq задач (перевод, ASR).


## Комбинации параметров

### Стандартная комбинация: Temperature + Top-P

Наиболее распространённая комбинация для chat:

```
Порядок:
  1. Logits / Temperature    --> масштабирование
  2. Softmax                 --> вероятности
  3. Top-P фильтрация        --> отсечение хвоста
  4. Перенормировка           --> сумма = 1
  5. Сэмплирование           --> выбор токена
```

### Комбинация с Min-P

```
Порядок:
  1. Logits / Temperature
  2. Softmax
  3. Min-P фильтрация
  4. Перенормировка
  5. Сэмплирование
```

### Temperature + Top-K + Top-P

```
Порядок:
  1. Logits / Temperature
  2. Softmax
  3. Top-K фильтрация        --> отсечение по количеству
  4. Top-P фильтрация        --> отсечение по сумме
  5. Перенормировка
  6. Сэмплирование
```

### Типичные пресеты

```
"Детерминированный" (код, факты):
  temperature: 0
  (остальные параметры не важны при T=0)

"Стандартный" (chat):
  temperature: 0.7
  top_p: 0.9
  repetition_penalty: 1.1

"Креативный" (creative writing):
  temperature: 1.0
  top_p: 0.95
  repetition_penalty: 1.1

"Осторожный" (перевод, суммаризация):
  temperature: 0.3
  top_p: 0.9
  repetition_penalty: 1.0

"Min-P" (альтернативный стандарт):
  temperature: 0.7
  min_p: 0.05
  repetition_penalty: 1.1
```


## Рекомендации по задачам

### Таблица рекомендованных параметров

| Задача | T | Top-P | Top-K | Rep. Pen. | Min-P | Примечание |
|--------|---|-------|-------|-----------|-------|-----------|
| Код | 0 | - | - | 1.0 | - | Greedy, детерминированный |
| Код (рефакторинг) | 0.3 | 0.9 | - | 1.05 | - | Лёгкая вариативность |
| Chat (общий) | 0.7 | 0.9 | 40 | 1.1 | - | Стандартный пресет |
| Chat (техн.) | 0.5 | 0.9 | - | 1.05 | - | Точнее, суше |
| Перевод | 0.3 | 0.9 | - | 1.0 | - | Точность важнее |
| Суммаризация | 0.3 | 0.9 | - | 1.1 | - | Без повторений |
| Creative | 1.0 | 0.95 | - | 1.15 | - | Разнообразие |
| Brainstorm | 1.2 | 0.95 | - | 1.2 | - | Максимум идей |
| JSON output | 0 | - | - | 1.0 | - | Строгий формат |
| Classification | 0 | - | - | 1.0 | - | Один ответ |

### Как подбирать параметры

```
Шаг 1: Определить задачу
  - Нужна точность? --> T=0 (greedy)
  - Нужна естественность? --> T=0.5-0.8
  - Нужно творчество? --> T=0.8-1.2

Шаг 2: Установить фильтрацию
  - Top-P=0.9 -- хороший дефолт
  - Или Min-P=0.05 -- альтернатива

Шаг 3: Борьба с повторениями
  - Короткие ответы: penalty не нужен
  - Длинные ответы: repetition_penalty=1.1

Шаг 4: Тестирование и корректировка
  - Запустить 5-10 примеров
  - Слишком скучно? -> увеличить T
  - Слишком хаотично? -> уменьшить T
  - Повторения? -> увеличить penalty
  - Нет связности? -> уменьшить penalty
```


### Влияние temperature на разные модели

Одна и та же temperature может давать разный эффект
для разных моделей:

```
Маленькая модель (3B):
  T=0.7: уже довольно хаотичная
  T=1.0: часто бессмысленный текст
  Рекомендация: T=0.3-0.5

Средняя модель (8-13B):
  T=0.7: хороший баланс
  T=1.0: разнообразная, но связная
  Рекомендация: T=0.5-0.8

Большая модель (32-70B):
  T=0.7: стабильная, естественная
  T=1.0: разнообразная, качественная
  Рекомендация: T=0.7-1.0
```


## Порядок применения

llama.cpp (и llama-server) применяет параметры сэмплирования
в определённом порядке. Порядок влияет на результат.

### Стандартный порядок в llama.cpp

```
1. Repetition penalty     (модификация logits)
2. Temperature            (масштабирование logits)
3. Top-K                  (фильтрация по количеству)
4. Tail Free Sampling     (фильтрация, редко используется)
5. Typical P              (фильтрация, редко используется)
6. Top-P                  (фильтрация по кумулятивной вероятности)
7. Min-P                  (фильтрация по относительному порогу)
8. Softmax                (нормализация в вероятности)
9. Сэмплирование          (выбор токена по вероятностям)
```

### Настройка порядка в llama-server

```bash
# стандартный порядок (по умолчанию)
llama-server --samplers "top_k;tfs_z;typ_p;top_p;min_p;temperature"

# альтернативный порядок (temperature сначала)
llama-server --samplers "temperature;top_k;top_p;min_p"
```

[!] В большинстве случаев стандартный порядок оптимален.
Менять его имеет смысл только при экспериментах.


## Связь с llama-server

### Параметры командной строки

```bash
llama-server \
    --model model.gguf \
    --temp 0.7 \              # temperature
    --top-p 0.9 \             # top-p
```

## Распространённые ошибки

### 1. Temperature 0 + Top-P < 1

```
--temp 0 --top-p 0.9

T=0 уже означает greedy. Top-P при greedy не имеет эффекта
(один токен всегда покрывает 100% вероятности).
```

### 2. Высокий penalty + длинная генерация

```
--repeat-penalty 1.3 --repeat-last-n 0 --n-predict 4096

На длинных текстах все частотные слова получат огромный штраф.
Модель начнёт генерировать редкие слова, потеряет связность.
```

### 3. Top-K=1

```
--top-k 1

Это greedy decoding. Лучше использовать --temp 0 --
явнее намерение.
```

### 4. Слишком высокая temperature с маленькой моделью

```
--temp 1.2 (для модели 3B)

Маленькие модели менее "уверены" в предсказаниях.
Высокая T усиливает неуверенность, результат -- бессмыслица.
Для 3B лучше T=0.3-0.5.
```
