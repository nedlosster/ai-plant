## Что такое мультимодальность

Стандартная LLM работает только с текстом: текст на вход, текст на выход.
Мультимодальная модель обрабатывает несколько типов данных (модальностей):
текст, изображения, аудио, видео.

```
Стандартная LLM:
  Текст -> [Модель] -> Текст

Vision LLM:
  Текст + Изображение -> [Модель] -> Текст

Audio LLM:
  Текст + Аудио -> [Модель] -> Текст (или Аудио)

Полностью мультимодальная:
  Текст + Изображение + Аудио -> [Модель] -> Текст + Изображение + Аудио
```

### Типы мультимодальных моделей

```
Модальности           | Примеры моделей        | Задачи
----------------------|------------------------|---------------------------
Текст + Изображение   | LLaVA, Qwen-VL,       | OCR, описание, VQA,
  -> Текст            | InternVL, Pixtral      | анализ скриншотов
                      |                        |
Речь -> Текст (ASR)  | Whisper, Canary        | Транскрибация, субтитры
                      |                        |
Текст -> Речь (TTS)  | XTTS, Bark, Piper      | Озвучивание, голосовые
                      |                        | ассистенты
                      |                        |
Текст -> Изображение  | Stable Diffusion,      | Генерация иллюстраций,
                      | FLUX, DALL-E           | дизайн, арт
                      |                        |
Текст -> Видео       | Mochi, CogVideoX       | Генерация видео
```

В этом документе основной фокус -- на vision LLM (текст + изображение -> текст),
как наиболее практичной и зрелой категории.


## Vision LLM: текст + изображение

Vision LLM принимает на вход текст и одно или несколько изображений,
генерирует текстовый ответ.

### Возможности

```
Ввод: фотография + вопрос "Что изображено?"
Вывод: "На фотографии -- рыжий кот, сидящий на подоконнике."

Ввод: скриншот кода + "Найди баг"
Вывод: "В строке 15 отсутствует закрывающая скобка."

Ввод: документ (скан) + "Извлеки данные"
Вывод: "Номер договора: 123-456, Дата: 01.01.2025, Сумма: 50 000 руб."

Ввод: график + "Опиши тренд"
Вывод: "Выручка росла на 15% ежемесячно с января по июнь, затем стабилизировалась."
```

### Основные модели

```
Модель            | Параметры | Vision encoder | Разрешение | Контекст
------------------|-----------|----------------|------------|--------
Qwen-VL 2.5       | 3B-72B   | ViT (SigLIP)   | Динамическое| 32K
InternVL 2.5      | 2B-78B   | InternViT-6B   | До 4K px   | 32K
LLaVA 1.6         | 7B-34B   | CLIP ViT-L     | 336-672 px | 4K

Phi-3.5 Vision    | 4.2B     | CLIP            | 1344 px    | 128K
```


## Как работает vision в LLM

### Архитектура

```
Изображение                    Текст (вопрос)
     |                              |
     v                              v
+----------------+           +----------------+
| Vision Encoder |           | Tokenizer      |
| (CLIP/SigLIP)  |           | (текст -> токены)|
+----------------+           +----------------+
     |                              |
     v                              v
[img_emb_1, img_emb_2, ...]  [tok_1, tok_2, ...]
     |                              |
     +--------- Конкатенация --------+
                    |
                    v
          [img_1, img_2, ..., tok_1, tok_2, ...]
                    |
                    v
          +------------------+
          | Transformer LLM  |
          | (Qwen, Llama...) |
          +------------------+
                    |
                    v
          Текстовый ответ
```

### Компоненты

**1. Vision Encoder (CLIP / SigLIP / ViT)**

Преобразует изображение в набор embedding-ов. Одно изображение становится
набором из 64-2048 "визуальных токенов".

```
Изображение 336x336 -> CLIP ViT-L -> 576 визуальных токенов (каждый 1024-мерный)
Изображение 672x672 -> CLIP ViT-L -> 2304 визуальных токена

Каждый визуальный токен соответствует патчу (участку) изображения:
  336 / 14 = 24 патча по горизонтали
  24 * 24 = 576 патчей = 576 визуальных токенов
```

**2. Multimodal Projector (mmproj)**

Проецирует embedding-ы vision encoder в пространство LLM. Это небольшая
нейросеть (MLP или cross-attention), которая "переводит" визуальные
признаки на "язык" текстовых токенов.

```
CLIP embedding (1024-мерный) -> mmproj -> LLM embedding (4096-мерный)
```

**3. LLM backbone**

Стандартный transformer, обрабатывающий визуальные и текстовые токены
одинаково. Модель не "знает", что часть токенов -- из изображения.

### Схема обработки

```
Исходное изображение (1920x1080)
       |
       v
  Ресайз и нарезка на тайлы (336x336 каждый)
       |
       v
  +-------+-------+-------+
  | Тайл 1| Тайл 2| Тайл 3|  (+ уменьшенное общее изображение)
  +-------+-------+-------+
  | Тайл 4| Тайл 5| Тайл 6|
  +-------+-------+-------+
       |
       v
  Vision Encoder: каждый тайл -> 576 токенов
  6 тайлов + thumbnail = 7 * 576 = 4032 визуальных токена
       |
       v
  mmproj: 4032 * 1024 -> 4032 * 4096
       |
       v
  Вставка перед текстовыми токенами
  [img_tokens (4032)] + [text_tokens (N)]
       |
       v
  LLM обрабатывает суммарно 4032 + N токенов
```

### Влияние на контекст

Каждое изображение занимает значительную часть контекста:

```
Разрешение  | Тайлов | Визуальных токенов | Контекст 8K: остаток
------------|--------|--------------------|-----------
336x336     | 1      | 576                | 7424
672x672     | 4+1    | 2880               | 5120
1344x1344   | 16+1   | 9792               | Не влезает (>8K)
```

Для работы с большими изображениями нужен длинный контекст (32K+).


## Задачи vision LLM

### OCR (распознавание текста)

```
Ввод: скан документа / фото текста
Вывод: распознанный текст

Преимущество перед классическим OCR (Tesseract):
  - Понимает контекст (таблицы, формулы)
  - Работает с рукописным текстом
  - Структурирует данные (JSON, таблицы)
```

### Visual Question Answering (VQA)

```
Ввод: изображение + вопрос
Вывод: ответ на вопрос об изображении

Примеры:
  "Сколько людей на фото?" -> "На фото 5 человек"
  "Какого цвета машина?" -> "Машина красная"
  "Что написано на вывеске?" -> "Аптека 24 часа"
```

### Описание изображений (captioning)

```
Ввод: изображение
Вывод: текстовое описание

Краткое: "Кот на подоконнике"
Подробное: "Рыжий кот сидит на подоконнике, за окном зимний пейзаж..."
```

### Анализ скриншотов и UI

```
Ввод: скриншот приложения
Вывод: описание интерфейса, ошибки, рекомендации

Применение:
  - Тестирование UI: "Есть ли визуальные баги на скриншоте?"
  - Документация: "Опиши, что видно на экране"
  - Accessibility: "Читаем ли текст на этом фоне?"
```

### Анализ графиков и диаграмм

```
Ввод: график / диаграмма + вопрос
Вывод: интерпретация данных

"Какой максимум на графике?" -> "Максимум ~850 в июле 2024"
"Какой тренд?" -> "Рост с января по июль, затем снижение"
```

### Анализ кода (скриншоты)

```
Ввод: скриншот IDE с кодом
Вывод: распознанный код + анализ

Применение:
  - Распознавание кода с фото (лекция, книга)
  - Анализ ошибок, видимых на скриншоте
  - Воспроизведение кода с видеоуроков
```

### ASR: речь -> текст

```
Whisper -- стандарт для локального ASR:

Модель            | Параметры | Размер  | Качество (RU) | Скорость
------------------|-----------|---------|---------------|--------
Whisper tiny      | 39M       | 75 MB   | Плохое        | Очень быстро
Whisper base      | 74M       | 142 MB  | Среднее       | Быстро
Whisper small     | 244M      | 466 MB  | Хорошее       | Быстро
Whisper medium    | 769M      | 1.5 GB  | Хорошее       | Средне
Whisper large-v3  | 1.5B      | 3 GB    | Отличное      | Средне
```

### TTS: текст -> речь

```
Модель    | Тип          | Клонирование голоса | Языки | Качество
----------|--------------|---------------------|-------|--------
Piper     | Быстрый TTS  | Нет                 | 30+   | Хорошее
XTTS v2   | Нейросетевой | Да (10 сек образец) | 17    | Отличное
Bark      | Генеративный | Нет                 | 13    | Хорошее
```

### Ограничения audio на AMD

```
Whisper:
  - whisper.cpp поддерживает Vulkan (работает на Radeon)
  - Или через Python + PyTorch + ROCm (требует настройки)
  - Рекомендуется whisper.cpp с Vulkan

XTTS / Bark:
  - Только PyTorch + ROCm
  - Требует установки ROCm stack
  - Не все версии ROCm стабильно работают на consumer GPU
  - Piper работает на CPU (достаточно быстро для TTS)
```

### Рекомендации

```
Быстрая работа (скриншоты, OCR):
  -> Qwen-VL 2.5 7B Q4_K_M (~5 GiB)
  -> Быстрый, достаточно точный для OCR

Баланс качество/скорость:
  -> Qwen-VL 2.5 32B Q4_K_M (~19 GiB)
  -> Хорошее понимание сложных изображений

Максимальное качество:
  -> Qwen-VL 2.5 72B Q4_K_M (~42 GiB)
  -> Или InternVL 2.5 78B Q4_K_M (~46 GiB)
  -> Детальный анализ, сложные вопросы

Экономия VRAM (рядом с другой моделью):
  -> Phi-3.5 Vision 4.2B (~3 GiB)
  -> Или Qwen-VL 2.5 3B (~2 GiB)
```

### Одновременная работа текстовой и vision-модели

96 GiB позволяют запустить текстовую и vision-модели одновременно:

```
Вариант 1 (экономный):
  Qwen 2.5 32B Q4_K_M (19 GiB) + Qwen-VL 2.5 7B Q4_K_M (5 GiB) = 24 GiB
  Остаток: 72 GiB

Вариант 2 (сбалансированный):
  Qwen 2.5 32B Q4_K_M (19 GiB) + Qwen-VL 2.5 32B Q4_K_M (19 GiB) = 38 GiB
  Остаток: 58 GiB

Вариант 3 (максимальный):
  Llama 3.1 70B Q4_K_M (40 GiB) + Qwen-VL 2.5 32B Q4_K_M (19 GiB) = 59 GiB
  Остаток: 37 GiB
```


## Запуск vision-моделей на Radeon 8060S

### llama-server с mmproj

Vision-модели в GGUF состоят из двух файлов:
1. Основная модель (LLM backbone) -- `.gguf`
2. Multimodal projector (mmproj) -- `mmproj-*.gguf`

```bash
# Скачивание модели и mmproj
huggingface-cli download \
    Qwen/Qwen2.5-VL-7B-Instruct-GGUF \
    qwen2.5-vl-7b-instruct-q4_k_m.gguf \
    mmproj-qwen2.5-vl-7b-instruct-f16.gguf \
    --local-dir ./models/

# Запуск llama-server с vision
llama-server \
    --model ./models/qwen2.5-vl-7b-instruct-q4_k_m.gguf \
    --mmproj ./models/mmproj-qwen2.5-vl-7b-instruct-f16.gguf \
    --host 0.0.0.0 \
    --port 8080 \
    --ctx-size 8192 \
    --n-gpu-layers 999
```

### Отправка изображения через API

```python
import base64
import requests
from pathlib import Path


def encode_image(image_path: str) -> str:
    """Кодирование изображения в base64."""
    with open(image_path, "rb") as f:
        return base64.b64encode(f.read()).decode("utf-8")


def ask_about_image(
    image_path: str,
    question: str,
    base_url: str = "http://localhost:8080"
) -> str:
    """Вопрос к vision-модели об изображении."""
    image_b64 = encode_image(image_path)

    # Определяем MIME-тип
    suffix = Path(image_path).suffix.lower()
    mime_types = {
        ".jpg": "image/jpeg",
        ".jpeg": "image/jpeg",
        ".png": "image/png",
        ".gif": "image/gif",
        ".webp": "image/webp"
    }
    mime = mime_types.get(suffix, "image/jpeg")

    response = requests.post(
        f"{base_url}/v1/chat/completions",
        json={
            "messages": [
                {
                    "role": "user",
                    "content": [
                        {
                            "type": "image_url",
                            "image_url": {
                                "url": f"data:{mime};base64,{image_b64}"
                            }
                        },
                        {
                            "type": "text",
                            "text": question
                        }
                    ]
                }
            ],
            "temperature": 0.3,
            "max_tokens": 1024
        }
    )

    return response.json()["choices"][0]["message"]["content"]


# Пример
result = ask_about_image(
    "./screenshot.png",
    "Опиши, что изображено на скриншоте. Если есть текст -- распознай его."
)
print(result)
```

## Архитектура vision-моделей

### Детальная схема Qwen-VL

```
+--------------------------------------------------+
|                  Qwen-VL 2.5                     |
|                                                  |
|  Изображение (1920x1080)                         |
|       |                                          |
|       v                                          |
|  Dynamic Resolution:                             |
|    Разбиение на тайлы 448x448                    |
|    + Thumbnail (448x448)                         |
|       |                                          |
|       v                                          |
|  +-----------------+                             |
|  | SigLIP ViT      |  (400M параметров)         |
|  | 448x448 -> 1024 |  Каждый тайл -> 1024 токенов|
|  +-----------------+                             |
|       |                                          |
|       v                                          |
|  Визуальные токены: N * 1024                     |
|       |                                          |
|       v                                          |
|  +-----------------+                             |
|  | MLP Projector   |  (Проекция в пространство LLM)|
|  | 1152 -> 3584    |                             |
|  +-----------------+                             |
|       |                                          |
|       v                                          |
|  [vis_tok_1, ..., vis_tok_N, text_tok_1, ...]    |
|       |                                          |
|       v                                          |
|  +-----------------+                             |
|  | Qwen2 LLM      |  (7B / 32B / 72B)           |
|  | Transformer     |                             |
|  +-----------------+                             |
|       |                                          |
|       v                                          |
|  Текстовый ответ                                 |
+--------------------------------------------------+
```

### Динамическое разрешение

Современные vision-модели (Qwen-VL 2.5, InternVL 2.5) используют
динамическое разрешение: изображение разбивается на столько тайлов,
сколько нужно для сохранения деталей.

```
Маленькое изображение (448x448):
  1 тайл + 1 thumbnail = 2 * 1024 = 2048 визуальных токенов

Среднее (896x896):
  4 тайла + 1 thumbnail = 5 * 1024 = 5120 визуальных токенов

Большое (1792x1792):
  16 тайлов + 1 thumbnail = 17 * 1024 = 17408 визуальных токенов
  (Требует контекст > 17K только для изображения!)
```

### mmproj: что внутри

mmproj (multimodal projector) -- относительно небольшая модель:

```
Архитектура: 2-слойный MLP (Multi-Layer Perceptron)
  Вход: 1152 (размерность SigLIP)
  Скрытый слой: 3584 (с GELU активацией)
  Выход: 3584 (размерность Qwen2 LLM)

Размер: 20-50 MB (зависит от модели)
Всегда хранится в FP16 (не квантуется -- слишком маленький)
```


## Ограничения

### Скорость

Vision-модели медленнее текстовых из-за обработки изображений:

```
Этап                          | Время (оценка для 7B Q4)
------------------------------|------------------------
Vision encoder (1 изображение)| 0.5-2 сек
mmproj                        | <0.1 сек
Prompt processing (4K токенов)| 0.5-1 сек
Token generation              | как обычная LLM

Итого: первый ответ на ~1-3 сек дольше, чем чисто текстовый запрос.
```

### Точность OCR

```
Тип текста              | Точность | Рекомендация
------------------------|----------|----------------------------
Печатный текст (скан)   | 90-98%   | Подходит для большинства задач
Рукописный текст        | 60-80%   | Только с крупными моделями (32B+)
Таблицы                 | 70-90%   | Структура может нарушаться
Формулы                 | 50-80%   | Зависит от модели и сложности
Мелкий текст (<10px)    | 30-60%   | Увеличить разрешение перед отправкой
```

### Галлюцинации

Vision-модели могут "видеть" то, чего нет на изображении:

```
Типичные ошибки:
  - Неверное количество объектов
  - Выдуманные надписи
  - Неправильные цвета (при низком разрешении)
  - Неверная интерпретация графиков

Снижение галлюцинаций:
  - Большая модель (72B лучше 7B)
  - Высокое разрешение изображения
  - Конкретные вопросы вместо "опиши все"
  - Температура 0.1-0.3
```

### Размер изображений

```
Проблема: большие изображения = много визуальных токенов = медленно + дорого

Рекомендации:
  - Ресайз до 1024px по длинной стороне (для общих задач)
  - Ресайз до 2048px (для OCR мелкого текста)
  - Кроп нужной области вместо целого скриншота
  - JPEG quality 85% (уменьшает base64 без потери для модели)
```

## Практические рецепты

### Утилита для пакетной обработки изображений

```python
"""
Пакетная обработка изображений через vision LLM.

Применение:
  - Массовый OCR сканов
  - Описание фотографий для каталога
  - Проверка скриншотов UI
"""
import base64
import json
import requests
from pathlib import Path
from concurrent.futures import ThreadPoolExecutor


def process_image(
    image_path: str,
    prompt: str,
    base_url: str = "http://localhost:8080"
) -> dict:
    """Обработка одного изображения."""
    with open(image_path, "rb") as f:
        image_b64 = base64.b64encode(f.read()).decode()

    suffix = Path(image_path).suffix.lower()
    mime = {"jpg": "image/jpeg", "jpeg": "image/jpeg",
            "png": "image/png"}.get(suffix.lstrip("."), "image/jpeg")

    try:
        response = requests.post(
            f"{base_url}/v1/chat/completions",
            json={
                "messages": [{
                    "role": "user",
                    "content": [
                        {"type": "image_url",
                         "image_url": {"url": f"data:{mime};base64,{image_b64}"}},
                        {"type": "text", "text": prompt}
                    ]
                }],
                "temperature": 0.1,
                "max_tokens": 1024
            },
            timeout=60
        )
        answer = response.json()["choices"][0]["message"]["content"]
        return {"file": image_path, "result": answer, "error": None}
    except Exception as e:
        return {"file": image_path, "result": None, "error": str(e)}


def batch_process(
    image_dir: str,
    prompt: str,
    output_file: str = "results.json",
    extensions: tuple = (".jpg", ".jpeg", ".png"),
    max_workers: int = 1  # Последовательно (GPU не параллелит)
) -> list[dict]:
    """Пакетная обработка всех изображений в директории."""
    images = [
        str(p) for p in Path(image_dir).iterdir()
        if p.suffix.lower() in extensions
    ]

    results = []
    for i, image_path in enumerate(images):
        print(f"[{i + 1}/{len(images)}] {Path(image_path).name}")
        result = process_image(image_path, prompt)
        results.append(result)

        if result["error"]:
            print(f"  ОШИБКА: {result['error']}")
        else:
            # Первые 100 символов результата
            print(f"  OK: {result['result'][:100]}...")

    # Сохранение результатов
    with open(output_file, "w", encoding="utf-8") as f:
        json.dump(results, f, ensure_ascii=False, indent=2)

    print(f"\nОбработано: {len(results)}, ошибок: {sum(1 for r in results if r['error'])}")
    return results


# Пример: OCR всех сканов
if __name__ == "__main__":
    batch_process(
        image_dir="./scans/",
        prompt="Распознай текст на изображении. Верни только распознанный текст.",
        output_file="ocr_results.json"
    )
```

### Предварительная обработка изображений

```python
from PIL import Image


def preprocess_for_vision(
    image_path: str,
    max_size: int = 1024,
    quality: int = 85
) -> str:
    """
    Подготовка изображения для vision LLM.

    Ресайз + JPEG-сжатие для уменьшения размера base64.
    Возвращает путь к обработанному файлу.
    """
    img = Image.open(image_path)

    # Ресайз с сохранением пропорций
    w, h = img.size
    if max(w, h) > max_size:
        ratio = max_size / max(w, h)
        new_size = (int(w * ratio), int(h * ratio))
        img = img.resize(new_size, Image.LANCZOS)

    # Сохранение в JPEG
    output_path = image_path.rsplit(".", 1)[0] + "_processed.jpg"
    img = img.convert("RGB")  # Убираем alpha-канал
    img.save(output_path, "JPEG", quality=quality)

    original_size = Path(image_path).stat().st_size
    processed_size = Path(output_path).stat().st_size
    print(f"Размер: {original_size / 1024:.0f} KB -> {processed_size / 1024:.0f} KB")

    return output_path
```
