# Промпт-инжиниринг для AI-кодинга

Промпты и системные инструкции для локальных моделей (llama.cpp) и агентов (OpenCode, Aider).

## Принципы

1. **Конкретность** -- "напиши функцию сортировки массива по полю name" лучше, чем "сортируй данные"
2. **Формат вывода** -- указать язык, фреймворк, стиль: "Python 3.12, pytest, Google docstrings"
3. **Контекст** -- передать минимально необходимый код, не весь проект
4. **Инструкции в начале или конце** -- не в середине промпта
5. **Короткие промпты для локальных моделей** -- менее мощное instruction-following, чем у frontier-моделей
6. **Русский язык в промптах** -- Qwen3-Coder и Qwen3.5 понимают русский, промпты на русском работают

## Системные промпты

### Ассистент-разработчик (универсальный)

```
Ты -- опытный разработчик. Пиши чистый, поддерживаемый код.
Следуй стилю и паттернам проекта.
Обрабатывай граничные случаи.
Отвечай кодом, минимум объяснений.
Язык по умолчанию: Python 3.12.
Комментарии и документация на русском.
```

### Code review

```
Ты -- ревьюер кода. Анализируй:
1. Логические ошибки и граничные случаи
2. Уязвимости (инъекции, XSS, SSRF, утечки секретов)
3. Проблемы производительности (N+1, лишние аллокации)
4. Утечки ресурсов (незакрытые файлы, соединения, курсоры)
5. Потокобезопасность и race conditions
Указывай конкретные строки и предлагай исправления.
Не комментируй стиль и форматирование.
```

### Генерация тестов

```
Генерируй тесты на pytest.
Для каждой функции покрой:
- Позитивный сценарий (нормальная работа)
- Граничные случаи (пустой вход, границы, None)
- Ошибки (неверные типы, отсутствующие данные, исключения)
Имена тестов: test_<функция>_<сценарий>_<ожидание>.
Используй фикстуры для общей настройки.
Моки -- только при необходимости.
Комментарии на русском.
```

### Документирование

```
Добавь docstring в стиле Google ко всем публичным функциям и классам.
Включи:
- Однострочное описание
- Args с типами и описаниями
- Returns с типом и описанием
- Raises для ожидаемых исключений
- Пример использования (если неочевидный)
Не меняй логику кода.
Описания на русском, технические термины не переводить.
```

### Debugging

```
Проанализируй ошибку и дай:
1. Root cause (одно предложение)
2. Минимальное исправление (код)
3. Как предотвратить подобные ошибки
Не предлагай улучшения, не относящиеся к ошибке.
```

### Рефакторинг

```
Выполни рефакторинг кода. Сохрани поведение.
Фокус:
- Уменьшение сложности (выделение функций, упрощение условий)
- Устранение дублирования (DRY)
- Улучшение именования (описательные, единообразные имена)
Не меняй публичный API.
Покажи полный результат, не diff.
```

## Шаблоны промптов

### Генерация функции

```
Напиши функцию `{name}` на Python:
- Вход: {описание входных данных}
- Выход: {описание результата}
- Ограничения: {граничные случаи}
- Ошибки: {что делать при ошибках}

Пример:
  Вход: {пример входа}
  Выход: {пример выхода}
```

### Генерация API-эндпоинта

```
Напиши FastAPI-эндпоинт:
- Метод: {GET/POST/PUT/DELETE}
- Путь: {/api/v1/...}
- Вход: {тело запроса / query-параметры}
- Выход: {схема ответа}
- Валидация: {ограничения}
- Ошибки: {коды и сообщения}
- Авторизация: {если нужно}
```

### Миграция кода

```
Перенеси код с {источник} на {цель}.
Сохрани поведение. Учти различия:
- {API}
- {Именование}
- {Обработка ошибок}

Исходный код:
```{code}```
```

### Объяснение кода

```
Объясни этот код:
1. Что делает (один абзац)
2. Как работает (по шагам)
3. Что может пойти не так (граничные случаи, сбои)
4. Как улучшить (если есть смысл)

Код:
```{code}```
```

### Bash-скрипт

```
Напиши bash-скрипт:
- Задача: {описание}
- Вход: {аргументы, stdin}
- Выход: {stdout, файлы}
- Обработка ошибок: set -euo pipefail
- Комментарии на русском
```

## Best practices для актуальных моделей

### Qwen3-Coder-30B-A3B (MoE) -- ежедневный инструмент

- 85 tok/s на платформе -- комфортная скорость
- Хорошо справляется с одношаговыми и средними задачами
- Tool calling стабильнее, чем у dense-моделей аналогичного размера
- Контекст 256K -- навигация по большим проектам
- Нет FIM -- только chat/agent
- Температура 0.0-0.3 для детерминированного кода

### Qwen3-Coder-Next 80B MoE -- сложные задачи

- 53 tok/s на платформе -- приемлемо для сложных задач
- SWE-bench Pro 44.3% -- лидер среди open-source
- MoE 3B active -- быстрый отклик при 80B знаний
- Контекст 256K
- Для архитектурных решений, рефакторинга 10+ файлов, отладки незнакомого кода

### Qwen3.5-122B-A10B (MoE) -- reasoning

- ~15-20 tok/s (прогноз) -- медленнее Coder'ов
- 10B active -- мощнее для сложного reasoning
- Не специализирован на коде, но сильнее в общих задачах
- Для архитектурного анализа, планирования, документации

### Qwen2.5-Coder-1.5B (FIM) -- автодополнение

- 120 tok/s -- мгновенный отклик
- Только автодополнение, не chat
- Контекст 2-4K токенов
- Предсказывает 1-10 строк вперед
- Работает параллельно с chat-моделью на порту 8081

## Работа с контекстом

### Что включать

- Файл, над которым работаем
- Связанные файлы (импорты, интерфейсы, типы)
- Тесты (если просим рефакторинг)
- Трейсбек (если debug)

### Что не включать

- Весь проект целиком
- Файлы конфигурации (если не относятся к задаче)
- Зависимости (node_modules, venv)
- Большие файлы данных

### Формат передачи

```
Файл: src/parser.py
```python
{содержимое файла}
```

Файл: src/models.py
```python
{содержимое файла}
```

Задача: Добавь валидацию типов в parser.parse_config()
```

## Итерационный процесс

1. Первый запрос -- задача
2. Если результат неточный -- уточнить требования
3. Если структура неверная -- дать пример ожидаемого вывода
4. Если стиль не тот -- указать конвенции проекта
5. Финальная проверка -- запустить тесты / линтер

Для локальных моделей итерации дешёвые (бесплатно, быстро). Лучше 3 коротких запроса, чем 1 длинный.
