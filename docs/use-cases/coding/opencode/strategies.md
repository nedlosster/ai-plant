# OpenCode: стратегии и приёмы работы

Практические приёмы эффективной работы с OpenCode и локальными моделями.

## Ключевое отличие от Claude Code

Claude Code работает с моделями 200B+ (Opus, Sonnet) с контекстом до 1M токенов. OpenCode с локальными моделями -- 7B-70B с контекстом 8K-128K. Это требует других стратегий:

| Аспект | Claude Code | OpenCode (локальные модели) |
|--------|-------------|----------------------------|
| Контекст | до 1M токенов | 8K-128K (зависит от модели) |
| Качество reasoning | высокое | среднее (зависит от модели) |
| Tool calling | надёжное | нестабильное на open-source моделях |
| Скорость | зависит от API | зависит от GPU и модели |

## Стратегия 1: декомпозиция задач

Локальные модели хуже справляются со сложными многошаговыми задачами. Разбивать на атомарные шаги:

**Плохо:** "Добавь авторизацию в API, миграцию БД, тесты и документацию"

**Хорошо:**
1. "Создай миграцию для таблицы users (id, email, password_hash)"
2. "Создай модель User и репозиторий"
3. "Добавь эндпоинт POST /auth/login"
4. "Напиши тест для /auth/login"
5. "Обнови документацию API"

Каждый шаг -- отдельный запрос. Контекст не переполняется, модель фокусируется на одной задаче.

## Стратегия 2: plan -> build

Использовать два агента последовательно:

1. **Plan** (read-only): "Проанализируй структуру проекта и предложи план рефакторинга модуля X"
2. Прочитать план, скорректировать
3. **Build** (full access): "Реализуй пункт 1 из плана: ..."

Plan-агент не может сломать код. Build-агент получает конкретную задачу.

Переключение: клавиша **Tab** в TUI.

## Стратегия 3: контекст -- ценный ресурс

При 32K контекста каждый файл в контексте -- дорого. Приёмы экономии:

### Указывать конкретные файлы

**Плохо:** "Найди баг в проекте"

**Хорошо:** "В файле src/auth/handler.go функция Login возвращает 500 вместо 401 при неверном пароле. Исправь."

### Не загружать лишнее

Если модель читает файлы через `read` tool -- она тратит контекст. Указывать точно, что читать:

"Прочитай src/auth/handler.go строки 45-80 и найди ошибку в обработке пароля"

### Начинать новую сессию

При длинном диалоге контекст заполняется историей. Если задача сменилась -- перезапустить OpenCode (Ctrl+C, `opencode`).

## Стратегия 4: выбор модели под задачу

Разные модели для разных задач:

| Задача | Рекомендуемая модель | Размер | Почему |
|--------|---------------------|--------|--------|
| Рефакторинг, новые фичи | Qwen3-Coder-30B (MoE) | 17 GiB | Хороший tool calling, знание кода |
| Быстрые правки | Qwen2.5-Coder-7B | 4 GiB | Быстрый, достаточный для простых задач |
| Сложная архитектура | Qwen3.5-27B | 16 GiB | Сильный reasoning |
| Документация | любая 7B+ | -- | Не требует tool calling |

Смена модели -- перезапуск inference-сервера:
```bash
./scripts/inference/stop-servers.sh
./scripts/inference/start-server.sh Qwen3-Coder-30B-A3B-Instruct-Q4_K_M.gguf --daemon
```

## Стратегия 5: инструкции вместо промптов

Повторяющиеся указания вынести в `instructions`:

```json
{
  "instructions": "{file:.opencode/instructions.md}"
}
```

Вместо каждый раз писать "пиши на русском, используй pytest, не трогай .env" -- один раз записать в инструкции.

## Стратегия 6: кастомные агенты для типовых задач

Создать агентов для повторяющихся задач:

```
.opencode/agents/
  reviewer.md     # @reviewer проверь src/auth/
  tester.md       # @tester напиши тесты для handler.go
  documenter.md   # @documenter обнови docs/api.md
```

Каждый агент имеет свой системный промпт с инструкциями. Это как skills в Claude Code, но через файлы.

## Стратегия 7: работа с tool calling

Open-source модели нестабильны в tool calling. Приёмы улучшения:

### Увеличить контекст

llama-server по умолчанию 32768 токенов. Для сложных задач с tool calling -- достаточно. Для простых моделей (7B) можно уменьшить до 8192:

```bash
./scripts/inference/start-server.sh model.gguf 8080 8192 --daemon
```

### Использовать модели с хорошим tool calling

Рейтинг по качеству tool calling (на 2026):
1. Qwen3-Coder-30B (MoE) -- лучший для кодинга
2. Qwen3.5-27B -- хороший general purpose
3. Llama-3.3-70B -- хороший, но требует много VRAM
4. Qwen2.5-Coder-32B -- хороший для кода

### Fallback: прямые команды

Если tool calling не работает -- давать команды явно:

"Выполни команду: `cat src/main.go | head -50` и покажи результат"

Вместо ожидания, что модель сама вызовет `read`.

## Стратегия 8: git-интеграция

OpenCode не делает автоматических коммитов (в отличие от Aider). Workflow:

1. Запустить OpenCode в ветке или чистом рабочем дереве
2. Дать задачу
3. Проверить изменения: `git diff`
4. Закоммитить вручную (или попросить OpenCode)

Для безопасности -- работать в feature-ветке:
```bash
git checkout -b feature/new-auth
opencode
# ... работа ...
git add -A && git commit -m "feat: auth module"
```

## Антипаттерны

### Не делать

- **Давать огромный контекст** -- "прочитай все файлы в src/ и найди баг". Контекст переполнится.
- **Цепочки зависимых задач в одном запросе** -- "сделай A, потом B на основе A, потом C". Разбивать на отдельные запросы.
- **Ожидать уровень Claude Opus** -- локальные модели 30B дают ~70% качества Opus. Для оставшихся 30% -- декомпозиция и ручная проверка.
- **Игнорировать ошибки tool calling** -- если модель вызывает инструмент неправильно, переформулировать задачу проще.
